# Layered validation surfaces in Test-as-operator (AJV + govaluate)

> **Status:** accepted (2026-05-29)
> Builds on [ADR 0038](./0038-data-element-governance-three-independent-layers.md) (the three governance layers this ADR surfaces), [ADR 0039](./0039-data-element-registration-admin-authored-single-page-flow.md) (the registration flow whose Review-tab Test-as-operator modal this ADR re-specifies), [ADR 0040](./0040-smart-start-assist-grounded-cross-tab-suggestion-engine.md) (Smart Start authoring loop whose `rules[]` output feeds L2), [ADR 0042](./0042-element-spec-sheet-onramp-deterministic-parser-audit-log-provenance.md) (strip-at-publish doctrine that keeps `elementSchema` interop-clean for AJV), [ADR 0043](./0043-element-persistence-schema-driven-composer-workspace-v6.md) (the workspace persistence that holds the bundle artefacts L1/L2/L3 read from), [ADR 0046](./0046-data-element-bundle-export-to-admin-corev2-registry.md) (the **`x-*` zero-residue cutover** that splits publication into `elementSchema` + `uiSchema` + `uiRules` + `rules[]` + `authoringMetadata`, defining the inputs each layer in this ADR consumes). References angle-gauge-ui's [help-system-framework-design.md](../../../angle-gauge-ui/docs/plans/2026-03-17-help-system-framework-design.md) (the layered-help discipline this ADR adopts) and dex-monorepo's [`FloatingPanel`](../../../dex-monorepo/ui/libs/src/components/FloatingPanel/FloatingPanel.tsx) primitive (the production surface this ADR designs against). Closes the open question implicit since ADR 0038 named the three governance layers: *how does the registration prototype's Test-as-operator modal surface them — to the platform admin authoring the element AND to the platform admin testing what the operator will experience — without conflating engines, audiences, or runtime moments?*

## Context

ADR 0038 names three independent governance layers a Data Element carries: **Schema validation** (AJV against JSON Schema), **Validation rules** (govaluate cross-field expressions), and **Routing rules** (orchestrator-time, out of registration scope). ADR 0039 establishes the four-tab registration canvas (Schema · Compose complexity · Rules · Review) and the **Test-as-operator** modal Sarah opens from Review — a full-screen Composer simulation that renders the current authored state as the operator will see it post-publish. ADR 0046's RJSF cutover splits the publication artefact into five co-versioned pieces (`elementSchema` + `uiSchema` + `uiRules` + `rules[]` + `authoringMetadata`), each owned by a different layer and consumed by a different runtime.

The Test-as-operator modal as it stands (post-cutover, pre-this-ADR) demonstrates only one of the three layers: govaluate-style `rules[]` evaluate live in a right-side panel against the synthesised sample payload, with PASS / FAIL / ERROR badges per rule. AJV is invisible — there is no surface where Sarah experiences what shape-layer rejection looks like for an operator typing into the form. `uiRules.visibility` (the runtime UI-visibility rules per ADR 0046's bundle anatomy) is invisible too — the modal renders all fields regardless of their visibility expressions. The result: Sarah authoring a schema with required fields, format constraints, and conditional visibility ships an element whose *only* tested behaviour is cross-field validation. Production AJV catches the shape errors at the operator's first keystroke; production RJSF re-renders the form when discriminator values change; production govaluate blocks submit on cross-field violations. The prototype tests one of three.

Two structural realities shape this ADR's design choices:

(1) **dex-monorepo already has a [[FloatingPanel]] primitive** ([`ui/libs/src/components/FloatingPanel/FloatingPanel.tsx`](../../../dex-monorepo/ui/libs/src/components/FloatingPanel/FloatingPanel.tsx)) — a fixed-bottom collapsible accordion taking a `messages: FloatingPanelMessage[]` array (each `{ message, type?, onClick? }`). Its job in production will be to surface validation failures to operators at submit time. It exists in the component library but is not yet wired into [`Form.tsx`](../../../dex-monorepo/ui/libs/src/components/Forms/Form.tsx) (which has `showErrorList={false}` reserving the slot). Mechanically the primitive renders one flat list with no engine attribution, no count split, no per-section locality, no live-update during typing, and conflates clickable / non-clickable messages by `onClick` presence. The prototype's Test-as-operator can either *match* this primitive (production-faithful but inherits its limits) or *design against* it (propose a more layered alternative the dex team can adopt when DSForm mounts in [`ui/apps/core/web/pitstop`](../../../dex-monorepo/ui/apps/core/web/pitstop)).

(2) **angle-gauge-ui has solved an analogous problem with a layered-help framework** ([`wiki/medical-assessments/contextual-help.md`](../../../angle-gauge-ui/wiki/medical-assessments/contextual-help.md) + [`docs/plans/2026-03-17-help-system-framework-design.md`](../../../angle-gauge-ui/docs/plans/2026-03-17-help-system-framework-design.md)). Four named layers (Workflow Rail / Field Guidance / Protocol Notes / Reference), each with one question, one primary surface, one codebase home, one authoring contract; a decision tree routes new content to its layer; boundary tests catch drift; a single explicit escape valve (dynamic Alerts) names what's *not* a help layer; plain-language policy is enforced by `*.test.ts` suites. The discipline is the inverse of the FloatingPanel problem: rather than one undifferentiated surface, multiple named surfaces with strict separation. This ADR adopts that discipline as the organising principle for validation rendering — the same anti-drift moves that worked for clinician help fit "Sarah authoring a Data Element" because the failure mode is identical: surfaces drift, audiences conflate, vocabulary leaks across boundaries.

A grilling session walked the decision tree from "which engines to surface" through "spatial mapping" / "Sarah-teaching vs operator-pure" / "show-at-open vs blur-trigger" / "Submit semantics" / "cascade behaviour" / "summary device placement" / "FloatingPanel anatomy" / "polymorphism handling" / "AJV package + load" / "boundary test enforcement", with the angle-gauge-ui framework providing the meta-discipline. The decisions below close that tree.

## Decision

The Test-as-operator modal becomes a **layered validation rendering surface** governed by an explicit four-layer taxonomy borrowed from angle-gauge-ui's help-system framework. Each layer answers one Sarah-or-operator question, has one primary surface, one authoring contract, one runtime moment, and one codebase home. A single out-of-band escape valve names what is *not* a validation layer. Twenty sub-decisions:

**1. Adopt angle-gauge-ui's layered-framework discipline as the organising principle.** Borrow ten principles wholesale: one question per layer, one primary surface per layer, one codebase home per layer, authoring contract per layer, agent decision tree, boundary tests as concrete IN/OUT examples, escape valves named explicitly, plain-language policy with vocabulary fences, tests enforce (not docs), hybrids documented with rationale. The framework's value is not just the taxonomy — it is the *anti-drift discipline*: surfaces stay separated because authoring/runtime contracts are explicit and enforced. The prototype gains the same loss-of-fidelity protection clinical help gained from the same discipline. Rejected: ad-hoc surface design (the historical pre-framework state — surfaces accrete, vocabulary leaks; FloatingPanel's flat-list-of-everything is what this looks like at maturity) and a portal-invented bespoke framework (angle-gauge-ui's framework is field-tested across 97 live `data-help-id` targets and 151 generated fallback IDs in a clinically-load-bearing UI; reinventing it for portal would gain nothing).

**2. Four validation layers + one out-of-band.** The taxonomy:

| Layer | Sarah/operator question | Engine | Artefact consumed | Primary surface | Effect on (faux-)submit |
|---|---|---|---|---|---|
| **L1 — Field shape** | *"Is this field's shape right?"* | AJV | `elementSchema` (pure JSON Schema) | Inline error under each input + attribution chip + hover popover | Blocks |
| **L2 — Cross-field rules** | *"Are the cross-field invariants satisfied?"* | govaluate (`rules[]`) | `rules[]` | Right-side panel (Sarah-HUD, live, exposes expressions) **AND** Submit-triggered FloatingPanel (operator-faithful, exposes `on_failure` only) | Blocks |
| **L3 — Conditional visibility** | *"Should this field be visible to the operator right now?"* | govaluate (visibility expressions in `uiRules.visibility`) | `uiRules.visibility` | The renderer itself — hidden fields don't render, L1 doesn't fire on them, L2 evaluates `undefined` for their values | Changes *which fields exist* in the form; does not directly block |
| **L4 — Aggregations** | *"How many problems remain, and where do I start?"* | Derived from L1 + L2 evaluator output | (none — projections of L1/L2 state) | Modal-header Schema chip (form-level) + conditional `<legend>` chips on each fieldset (section-level) | Click-to-scroll navigation only |
| **— out-of-band** | *"Does this field need a pre-publish review look?"* (Sarah-only) | (none — flag-shaped) | `authoringMetadata.reviewRequired` | Pre-flight publish gate on the Review tab — **not surfaced in Test-as-operator at all** | n/a — not a Test-as-operator concern |

L1, L2, L3, L4 each have one Sarah question and one operator question (collapsed in the table for brevity). The out-of-band entry names the artefact that *looks* like a validation layer but isn't — preventing future drift where someone adds review-required as an inline-error fifth concern.

**3. L2 is an acknowledged audience-split hybrid — two surfaces, one layer.** Following angle-gauge-ui's *"Layer 2 + 3 hybrids"* documentation pattern: L2 surfaces `rules[]` in two places that show *different projections of the same authored artefact*. The right-side panel exposes Sarah's authoring view (`name`, `expression`, status badge, click-to-field link); the Submit-triggered FloatingPanel exposes the operator's runtime view (`on_failure` message, click-to-field, no expression, no status badge). They share an authoring contract (`rules[i]`) but project different fields to different audiences. Rejected: splitting into L2a (Sarah HUD) and L2b (operator simulation) as separate numbered layers (artificial — the underlying engine, artefact, and run-time moment are identical; only the audience differs, which is a projection concern, not a layer concern) and merging into one surface that shows both projections (defeats the "operator-faithful" intent of the FloatingPanel; Sarah would see the operator simulation polluted with authoring artefacts no operator would ever see).

**4. Each layer carries an authoring contract — required + optional fields per record.** Borrowed directly from angle-gauge-ui's `helpRegistry` card contract pattern, adapted for validation output records:

| Layer | Contract |
|---|---|
| **L1** | `{ fieldName, primaryMessage, attribution: { engine: 'Schema', keyword, schemaPath, rawAjvMessage } }`. Required: `fieldName`, `primaryMessage`, `attribution.engine`, `attribution.keyword`. Optional: `attribution.schemaPath`, `attribution.rawAjvMessage` (popover only). |
| **L2 right panel** | `{ name, expression, status: 'PASSES' \| 'FAILS' \| 'ERROR', fieldRefs: string[], onFailure? }`. Required: `name`, `expression`, `status`. Optional: `fieldRefs` (derived per sub-decision 5; never authored), `onFailure` (only emitted when `status === 'FAILS'`). |
| **L2 FloatingPanel** | `{ fieldName, onFailureMessage, onClick? }`. Required: `fieldName`, `onFailureMessage`. Optional: `onClick` (omitted when no field anchor — see sub-decision 14). |
| **L4** | `{ scope: 'form' \| 'section', engine: 'Schema' \| 'Rules', count: number, scrollTo: () => void }`. Required: all four. Aggregations only — never carry error text. |

Records that violate the contract (missing required, populated-where-prohibited) are rejected by the surface-render function. The contracts make surfaces composable, testable, and resistant to drift.

**5. `fieldRefs[]` (L2) is derived, never authored.** The list of fields a `rules[i].expression` references is computed at evaluation time by regex-extracting identifiers from the expression text, stripping string literals first, then intersecting with `regDraft.fields[].name`. The function lives co-located with the rule evaluator. Rejected: authoring `fieldRefs` as an explicit array on `rules[i]` (maintenance hazard — Sarah edits the expression, forgets to update `fieldRefs`, click-to-field rot is silent; the derived value is self-consistent by construction) and using AST parsing of the expression instead of regex (govaluate expressions are simple enough that regex+intersection is sufficient; AST machinery would be over-engineering for the field-reference extraction problem).

**6. Render real AJV via CDN, not a hand-rolled JSON Schema validator.** Load `ajv@8` + `ajv-formats@3` from `https://cdn.jsdelivr.net/npm/`, matching the prototype's existing CDN pattern for Tabler Icons. Initialise as `new Ajv({ strict: true, allErrors: true })` + `ajvFormats(ajvInstance)`. The validator compiles against `regBuildPublishArtifacts(regDraft).elementSchema` (the interop-clean elementSchema per ADR 0046 and ADR 0042's strip-at-publish doctrine — already free of `x-*` keys, so `strict: true` is safe and catches malformed authoring at compile time). Validator is memoised on the schema's hash; recompiled when the schema changes, validated against the typed payload on every keystroke. Rejected: vanilla AJV without `ajv-formats` (loses `format: 'email'`, `format: 'date'`, etc. — formats production uses; demo would be incomplete) and `@rjsf/validator-ajv8` (production uses this wrapper to convert AJV errors into RJSF's internal `RJSFValidationError` shape — useful inside RJSF, useless outside it; the prototype isn't using RJSF for its Test modal renderer, so the wrapper is dead weight) and a hand-rolled mini-validator (the precedent for `regEvalExpression` was hand-rolling govaluate because no JS govaluate-equivalent ships from the Go runtime; AJV ships as the production engine and is npm-installable — no hand-rolling justification).

**7. AJV consumes pure `elementSchema`, never the sidecars.** Per ADR 0046's three-way artefact split and the **`x-*` zero-residue cutover** doctrine in CONTEXT.md, `uiSchema` is RJSF's concern (or our renderer's, mirroring RJSF), `uiRules` is govaluate's concern (visibility), and `rules[]` is govaluate's concern (cross-field). AJV sees only the JSON Schema half of the bundle. Boundary test enforceable in code: *"the object passed to `ajv.compile()` must have no key starting with `x-` at any depth"*. Rejected: passing the legacy `x-*`-laden schema through AJV with `strict: false` (regressive — relies on AJV's tolerance for unknown keywords; the cutover deliberately separated UI concerns into co-versioned artefacts so this tolerance is no longer needed; using it would muddy the demonstration of what production AJV is actually consuming) and stripping `x-*` keys at validation-time as a defensive shim (the publish boundary already strips them per ADR 0042 sub-decision 8 and ADR 0046 sub-decision 6; double-stripping at validation-time would imply the publish strip is unreliable, which would be a bigger problem).

**8. L1 inline errors render under each input — primary text + attribution chip + hover popover.** Per-field layout:

```
[input box, red ring when :invalid]
⚠ <operator-friendly message>            [Schema · <keyword>]
```

The primary message is operator-friendly translated from the AJV `keyword` + `schemaPath` (e.g., `pattern: "^\d{4}-\d{2}-\d{2}$"` → `"Use format YYYY-MM-DD."` for the operator; `"must match pattern \"^...\""` is reserved for the chip's hover popover). The chip `Schema · pattern` (or `Schema · required` / `Schema · format/email` / etc.) names the engine and AJV keyword — small, low-contrast at rest. Hovering the chip opens a popover with the raw AJV record: `{ keyword, schemaPath, rawAjvMessage, → Edit in Schema tab }`. Sarah uses the popover to verify the engine output and (optionally) navigate to the authoring surface; operators in production would see only the primary message. The chip is the Sarah-teaching marker; the popover is the Sarah-debug depth. Rejected: showing raw AJV text inline (`must match pattern "^..."` is unreadable to operators) and showing operator-friendly text only without any attribution (Sarah loses the engine-verification loop; she can't tell whether the error came from her `pattern` or her `format` or her `required` without re-reading the schema).

**9. All inline errors are visible from modal open — no touch-to-reveal.** Test-as-operator's Submit button is non-functional in the broader prototype sense (it does not submit a message — there is no backend); the modal is Sarah's testing snapshot of *what the operator would hit right now*. Showing errors only after blur or after submit-click would defeat that purpose — Sarah would have to interact with every field to see what AJV catches. The govaluate right panel already renders all failing rules at open (against the synthesised sample payload); inline AJV mirrors that. Rejected: blur-trigger / submit-only / first-input triggers (the canonical RJSF answers for *operator* surfaces, where the form opens calm and errors emerge as the operator interacts — but Test-as-operator is Sarah's surface, not the operator's, and her question is *"what's there?"* not *"what's there only after I touch it?"*) and a Sarah-toggle for "show all schema errors" on a hidden-by-default base (adds a control whose default state nobody benefits from; show-all-at-open is the right default for this surface).

**10. Wire the Submit button to simulate submit-time validation — three cascade cases.** The button's current `type="button"` + no-handler state changes: it gets an `onclick` that runs a cascade simulation. Three cases by current state:

(1) **Inline AJV still has errors:** the FloatingPanel pops with a one-line summary `"<N> Schema error(s) — fix the shape errors below first."` and a `↓ Jump to first` arrow that scrolls-and-focuses the first AJV-failing field. The panel does *not* duplicate the AJV error list (it's already inline). govaluate does not run — production cascade is preserved (AJV must pass before govaluate fires server-side).

(2) **AJV clean, govaluate has failing rules:** the FloatingPanel pops bottom-fixed inside the modal with one entry per `(rule, referenced-field)` pair (see sub-decision 14), formatted as `<b>${fieldName}:</b> ${rule.on_failure}` to mirror the production [`FloatingPanel`](../../../dex-monorepo/ui/libs/src/components/FloatingPanel/FloatingPanel.tsx) primitive's message anatomy. Click any entry → smooth-scroll the form to the field + 400ms amber pulse highlight on the input.

(3) **Both engines clean:** the Submit button transitions to `"Submitted ✓"` for 1.5s with a brief check-mark animation, then resets. No panel; no modal close. This is the felt-success state Sarah has been working toward — Doherty-style perceived-quality reward.

The cascade order (AJV before govaluate, gate at AJV failure) teaches Sarah the production runtime ordering by making it the *experiential* default — she encounters case (1) and sees that govaluate doesn't even get a chance until AJV passes. Rejected: making Submit submit-for-real to a fixture backend (out of scope; would imply a fixture-message lifecycle and conflate the validation demo with submission plumbing) and running both engines unconditionally on Submit-click (collapses the cascade into a single moment, defeating the "AJV gates govaluate" teaching point that's the production reality).

**11. L2 right panel stays as Sarah's authoring HUD — always visible, exposes expressions.** The existing right-side panel from ADR 0039's Test-as-operator design is preserved. It shows one card per `rules[i]`: rule name (clickable to scroll to a referenced field), raw `expression` text (Sarah's authored artefact, never shown to operators in production), `PASSES / FAILS / ERROR` status badge, `on_failure` (only when FAILS), and a click-to-field link. The panel header reads `"Validation rules · <count>"` — using ADR 0038's canonical glossary term, no parenthetical "(debug)" decoration. The panel updates live on every keystroke (govaluate evaluates against the typed payload). Rejected: removing the right panel in favour of FloatingPanel-only (Sarah loses the authoring-debug loop — she'd never see her expression text or `ERROR` status, both of which are authoring-only concerns operators don't experience; production govaluate runs server-side and reports only `on_failure` to operators, so FloatingPanel-only would erase the entire admin-side debug surface) and showing the expression text in the FloatingPanel too (defeats the operator-faithful intent — operators never see govaluate expressions in production; surfacing them in FloatingPanel would break the surface-per-audience separation).

**12. L4 aggregations: header Schema chip (always-on) + conditional `<legend>` chips (β).** Two summary surfaces:

- **Modal-header Schema chip** in the modal's header strip (next to the complexity pill and Acting-as banner): `Schema · <N> errors` — counts current AJV errors. Click → scrolls to the first AJV-failing field. The Rules chip is *not* added to the header — its live count already lives in the right-panel header (sub-decision 11), and duplicating it would just double-count the same number.
- **Conditional `<legend>` chips:** each fieldset's `<legend>` gains a count chip `Schema · <N>` *only when the section contains ≥1 AJV-failing field*. Clean fieldsets render no chip — quiet baseline, signal appears precisely where it's needed. govaluate counts are not section-attributed at this tier because cross-field rules may reference fields across sections (no clean section anchor).

Rejected: section chips always-on including zero (trains Sarah to ignore them, defeating their purpose), section chips for govaluate (anchor is ambiguous when expression references multi-section fields), and a single combined form-level summary device that mixes engines (collapses Q4's Sarah-teaching attribution back into a flat list — the FloatingPanel's anti-pattern).

**13. L1 errors with no `regDraft.fields[]` match attach to a schema-level error surface at the top of the form.** When AJV emits an error whose `instancePath` doesn't map to a top-level field in `regDraft.fields[]` — typically from JSON-editor-authored constructs like `oneOf` polymorphism, `additionalProperties: false` at root, `$ref` resolution failures — the error renders into a small "Schema-level errors" surface immediately above the form fields (above the first fieldset). Each error gets the same primary-text + attribution-chip + popover anatomy as inline errors. Production RJSF pattern (RJSF's `__errors` emit at root). The surface stays empty (un-rendered) when no such errors exist — invisible to the common field-builder-only authoring case. Rejected: dropping these errors from the visible UI and surfacing them only in the L4 header count (Sarah testing a JSON-editor-authored polymorphism would see "Schema · 1 error" with no way to inspect it inline) and best-effort attaching to the deepest `instancePath`-prefix-matching field (often wrong when no field is even close; misleading attribution is worse than honest non-attribution).

**14. Multi-field L2 rules emit N FloatingPanel entries — one per referenced field.** A rule like `end_date > start_date` references two fields; the FloatingPanel renders two entries (`<b>End date:</b> End date must be on or after start date.` and `<b>Start date:</b> End date must be on or after start date.`), each with `onClick` to scroll-and-focus its own field. The production [`FloatingPanel`](../../../dex-monorepo/ui/libs/src/components/FloatingPanel/FloatingPanel.tsx) primitive supports this naturally (`messages: FloatingPanelMessage[]` with per-message `onClick`). Both sides of the comparison are one click away. Rejected: emitting one entry per rule with a designated "primary field" (arbitrary which side of the comparison gets the click; operators often need to fix the "other" side and the picked-primary leaves them re-reading the rule to find it) and emitting one entry with multiple inline click targets in the message text (departs from the production anatomy; the primitive's contract is one `onClick` per `FloatingPanelMessage`).

**15. Rules with no `fieldRefs[]` emit a FloatingPanel entry without `onClick`.** Helper-only expressions like `today() > "2026-12-31"` or `len(line_items) > 0` (when line_items isn't a draft field) have no field anchor for navigation. The FloatingPanel entry renders the `on_failure` message without an `onClick`; per the primitive's behaviour ([`FloatingPanel.tsx:157`](../../../dex-monorepo/ui/libs/src/components/FloatingPanel/FloatingPanel.tsx)) the message renders as `<div>` (vs `<button>` when `onClick` is set). Production-faithful — no synthesis of a fake target. Rejected: scrolling to top-of-form as a default destination (arbitrary; pretends a navigation answer exists when it doesn't) and dropping the entry entirely (operator would never see the rule's failure at all — wrong by construction).

**16. `ERROR`-status rules (expression fails to compile) block faux-submit and surface explicitly in the FloatingPanel.** Production govaluate emits `ERROR` when an expression can't be parsed (typo, undefined helper, type mismatch). The right panel already shows the `is-errored` badge per rule. In the FloatingPanel, the rule renders as `<b>Rule "${rule.name}":</b> Rule has an authoring bug — see Rules tab to fix.` with `type: 'warning'` (vs `'error'` for normal FAILS) so the visual treatment is distinct from operator-visible cross-field failures. The faux-submit blocks — case (2) of the cascade treats ERROR identically to FAILS for the gate. Rationale: Sarah is testing precisely to catch her own authoring bugs *before* publish; an `ERROR` rule would either skip silently in production (govaluate's failure mode) or break a downstream consumer expecting the rule to fire — both are worse than blocking publish. The FloatingPanel surfacing makes the ERROR loud enough that Sarah can't ship by accident. A future cross-cutting publish pre-flight gate (out of scope) should refuse to publish elements with ERROR-status rules; this ADR's FloatingPanel surfacing is the test-time complement. Rejected: ERROR-rules invisible in FloatingPanel (the right panel alone already shows them, but the right panel is Sarah-HUD; the FloatingPanel surfacing makes the "this would break the operator's submit" lesson experiential) and ERROR-rules treated identically to FAILS without distinct labelling (operators would see "Rule X failed" with no actionable path; Sarah needs to know it's an authoring bug, not a runtime failure).

**17. FloatingPanel placement, animation, dismiss, live-update behaviour — bounded by production primitive, extended where the prototype can do better.** The FloatingPanel-equivalent in the Test modal sits bottom-fixed inside `.reg-test-modal` body (not viewport — modal is the boundary), slides up from below the footer in ≈220ms with `var(--easing)`, and preserves the production primitive's two-state model (expanded panel ↔ collapsed-to-icon-button + close-X). When collapsed, the icon button carries a count badge (`⚠ 2`) — a small extension on the production primitive whose collapsed state shows only the type icon. Each Submit click re-evaluates and re-renders the panel from current state (closed-X from prior click is reset). Click-to-field is smooth-scroll + 400ms amber pulse on the input — pulse is the prototype extension on top of production's scroll-only. The panel updates live during typing (errors drop off the panel as Sarah fixes them); when the list shrinks to zero the panel transitions to a one-line `"All cross-field rules now pass. Click Submit to confirm."` for ≈2s then auto-collapses to icon — empty-state celebration, no production analog. Auto-sizes: small when ≤3 messages, large when >3 (production exposes size as caller choice; the prototype reads error count to choose). All transitions honour `prefers-reduced-motion`. Rejected: rendering FloatingPanel at viewport-bottom-right (escapes the modal boundary; modal-inside-modal-escape feels wrong) and inline-into-body (defeats the *floating* part of the primitive — operators would scroll past it). The DESTRUCTIVE type is hardcoded for all validation panels; the prototype has no current need for the STANDARD-type informational variant.

**18. `oneOf` polymorphism in `elementSchema` renders with a picker chip + discriminator-driven re-render.** Sarah authors `oneOf` (from the future JSON-editor surface per ADR 0039 §5 — not yet built, but the design must accommodate it) when she wants *operator-runtime polymorphism* (the operator picks a variant at runtime; the published `elementSchema` keeps the `oneOf`). Distinct from `oneOf` *as an authoring tool* (where Sarah uses `oneOf` to express a choice and resolves it at publish time — published `elementSchema` is flat). The two patterns coexist; the picker chip is the visual marker that distinguishes them. Test-as-operator behaviour for the preserved case:

- Render variant matching the current discriminator field's value (the variant whose `properties.${discriminator}.const` matches the payload's discriminator value).
- Render a small picker chip at the top of the polymorphic region — labelled with the current variant's `title` (or fallback to the discriminator value), with an explicit dropdown for variant switching.
- On variant change (either via the picker chip OR by Sarah typing a new discriminator value into the form), re-render the polymorphic region and re-run AJV against the new variant.
- AJV always validates against the full `oneOf` — any variant's pass is a pass.

The picker chip is the Sarah-teaching marker. It tells her *this is an EForm-runtime polymorphism, not an authoring resolution* — without it, the variant-switch would feel like normal field editing and erase the operator-time-vs-authoring-time distinction. When `oneOf` has no clear discriminator (e.g., variants distinguished by `minimum`/`maximum` constraints rather than `const` keys), the chip says `Variant: 1 of N` with index numbers, and the discriminator-driven re-render falls back to picker-only (no input change drives a re-render). Rejected: render-first-variant-no-picker (Sarah can only test one variant from the modal; high friction to test the other), best-effort all-variants-stacked (operators in production wouldn't see all variants — faithfulness compromised), and refuse-to-render schemas with `oneOf` (over-defensive; Sarah's authoring did pick this polymorphism deliberately, the modal owes her a test of it).

**19. High-stakes wizard stays decorative — the stepper renders visually but the cascade is identical to simple.** Per ADR 0025 the Composer renders as a single page for simple and a 3-step wizard (Fill → Review → Submit) for high-stakes. The Test modal's stepper at the top of high-stakes-element preview is preserved as visual signal of "this is what the production wizard's shape looks like", but the button click (whether labelled `Submit` or `Continue to Review →`) triggers the same cascade simulation per sub-decision 10. No step-2 read-only Review pane is simulated; no step-boundary AJV gate is simulated. Rationale: the wizard's Review step exists for *commit-confirmation* per ADR 0025, not for validation timing. AJV runs the same way in both shapes; govaluate fires at the same logical moment (final submit) in both shapes. Simulating the wizard navigation adds UX scaffolding without adding validation-cascade demonstration. Rejected: minimal Review-step simulation with step-boundary AJV gating (would teach the AJV-before-govaluate cascade *order* more viscerally, but the cascade order is already taught by case (1) of the simple flow — duplicating it in high-stakes is decoration over necessity) and full wizard simulation with back-edit cycles (out of scope; this ADR is about validation rendering, not wizard navigation).

**20. Boundary tests as code, not just docs.** Following angle-gauge-ui's pattern of `*.test.ts` suites enforcing the contract (`helpRegistryContentQuality`, `triageTerminology`, `helpPlainLanguage`, `helpSourcePorting`), this ADR mandates analogous tests:

- **Engine vocabulary fences:** L2-FloatingPanel HTML contains no string matching `/[><=]/` or `must (match|have|be)` (operator-faithful surface; AJV/govaluate syntax leakage is forbidden).
- **AJV input cleanliness:** the object passed to `ajv.compile()` has no key starting with `x-` at any depth (post-cutover invariant; regressing the **`x-*` zero-residue cutover** is caught at test time).
- **Contract enforcement:** every L1 record passed to the inline renderer has the required fields per sub-decision 4; missing-field records are rejected (not silently swallowed).
- **Plain-language policy:** L1 primary inline message strings + L2 FloatingPanel `on_failure` strings contain no expression syntax, no JSON pointers, no AJV-vocabulary keywords (operator-readable check, mirrors angle-gauge-ui's `helpPlainLanguage.test.ts`).
- **L3 visibility honouring:** when a field is hidden by `uiRules.visibility`, no L1 inline error renders for it (test passes the value `false` to a `visibility` expression and verifies the inline-error DOM is empty for the hidden field).

Tests run under the existing `node --test` harness in `portal-app/tests/`. Rejected: documenting the boundaries in markdown only (angle-gauge-ui's experience names this as the failure mode that produced their framework — docs are aspirational, tests are enforceable; the discipline only sticks when the discipline is executable).

## Consequences

**Architectural:**

- A new module `portal-app/scripts/register-element-test-as-operator-validation.js` (or comparable filename) owns the four-layer surface assembly. Pure functions: `regBuildL1Records(elementSchema, payload) → L1Record[]` (consumes AJV output), `regBuildL2Records(rules, payload) → { sarahHud: L2HudRecord[], operatorFloating: L2FloatingRecord[] }` (consumes govaluate output, projects to two surfaces), `regBuildL3VisibilityMask(uiRules, payload) → Set<fieldPath>` (consumes govaluate-evaluated visibility expressions), `regBuildL4Aggregations({ l1Records, l2Records, fieldsBySection }) → L4Records[]`. The Test modal's render function consumes these and lays out the surfaces per the contracts in sub-decision 4.
- `regRenderTestModal()` ([`register-element.js`](../../portal-app/scripts/register-element.js)) extends: in addition to the existing right-panel govaluate render, it now compiles AJV against `elementSchema`, runs it on every payload change, and renders L1 inline + L4 header chip + L4 conditional `<legend>` chips. The Submit button gains an `onclick` that runs the cascade simulation per sub-decision 10. A new FloatingPanel-equivalent renders bottom-fixed inside the modal per sub-decision 17.
- `portal-app/index.html` gains two CDN script tags before `register-element.js`: `ajv@8` + `ajv-formats@3` from jsdelivr.
- A new tests file `portal-app/tests/test-as-operator-boundaries.test.js` enforces the boundary contracts per sub-decision 20.
- The L1 + L4 surfaces consume `regBuildPublishArtifacts(regDraft).elementSchema` (post-cutover interop-clean schema). The right panel + FloatingPanel consume `regBuildPublishArtifacts(regDraft).rules` (govaluate rules array). The visibility mask consumes `regBuildPublishArtifacts(regDraft).uiRules.visibility`. No layer reads from `regDraft.fields[]` directly except where field-name → DOM-node resolution is needed (the layers read the published artefacts; the renderer is the only consumer of the field-builder model).
- AJV's compiled validator is memoised on the schema's stringified hash; recompiled only when the schema changes (Sarah edits a field's type/required/pattern/etc.). Typical Sarah session: one compile per session; many validate-against-payload calls per keystroke. Compile cost is bounded; per-keystroke validate cost is sub-millisecond for typical field counts.

**Behavioural / contractual:**

- After this lands, Sarah's Test-as-operator modal demonstrates **all three governance layers from ADR 0038** — Schema validation (L1) live inline, Validation rules (L2) in both Sarah HUD (right panel) and operator simulation (FloatingPanel post-Submit), and conditional visibility (L3) by which fields actually render. The "what does the operator see?" question now has a complete answer for Sarah.
- The Submit button gains a *functional* semantic (sub-decision 10): faux-submit triggers the cascade simulation. It does not submit a message to any backend — there is none — but it transitions through the production-faithful timing (AJV first, govaluate second, success state on clear). Sarah experiences the cascade *order* she's been authoring for.
- **The FloatingPanel-equivalent in the prototype is a design proposal to the dex-monorepo team**, not a port. When [`ui/apps/core/web/pitstop`](../../../dex-monorepo/ui/apps/core/web/pitstop) mounts DSForm with the existing [`FloatingPanel`](../../../dex-monorepo/ui/libs/src/components/FloatingPanel/FloatingPanel.tsx) primitive, this ADR's design suggests three extensions to the production primitive: (a) count badge on collapsed icon, (b) live-update during typing rather than re-evaluate only on submit, (c) empty-state celebration on transition from N→0 errors. None of these require breaking changes to the primitive's API (`messages: FloatingPanelMessage[]` + `title?` already supports them); they are *consumer-side* composition patterns. The pitstop app's bring-up team can adopt some, all, or none.
- **Plain-language policy is now enforced in code** per sub-decision 20. Sarah authoring a `rule.on_failure` of `"end_date must be > start_date"` will fail the operator-vocabulary test (`/[><=]/` matched). The test failure points her at the lesson: `on_failure` is the operator-facing message; expression syntax belongs in `expression`, not `on_failure`. The discipline that took multiple iterations to internalise in angle-gauge-ui's clinical-help system is transferred to the validation surfaces here at first ship.
- **The Test modal becomes the canonical Sarah-debug surface for L1 + L2.** A subtle implication: where Sarah previously had to publish, copy the bundle, open admin-corev2 dev tooling, and inspect AJV output server-side to know "did my schema actually catch the malformed pattern?", she now sees that answer at every keystroke in the Test modal. The publish-import-inspect loop's iteration count drops dramatically — typical compile-cycle latency for "did I author the constraint right?" goes from minutes to milliseconds.

**Forward-flexible:**

- **JSON-editor authoring surface (ADR 0039 §5's Impl D/E).** When the JSON-editor lands as an editable surface (currently the JSON view is read-only per [`regRenderJsonPreview`](../../portal-app/scripts/register-element.js)), Sarah's ability to author `oneOf`/`allOf`/`$ref`/`format: "uuid"` constraints expands. Sub-decision 13's schema-level error surface and sub-decision 18's `oneOf` picker-chip behaviour are both designed for this future state — they sit unused for purely field-builder-authored schemas and activate automatically when JSON-editor constructs appear. No further ADR amendment needed when the editor lands; the layers absorb the new constraints by their existing contracts.
- **Step-boundary AJV gating for high-stakes wizard.** Sub-decision 19 deliberately leaves the wizard decorative. If a stakeholder demo or operator-research finding reveals that the *step boundary* itself is a load-bearing teaching moment (i.e., operators benefit from feeling "shape errors block Continue to Review →" as distinct from "shape errors block final Submit"), the wizard simulation can be added as a follow-on ADR. The cascade per sub-decision 10 already maps to step boundaries semantically; the implementation work is wizard navigation and step-state machinery, not validation engine work.
- **`querySchema` for PULL flows** (per ADR 0046 sub-decision 9). When `querySchema` authoring lands as a separate Review-tab surface, Test-as-operator can extend to simulate PULL-flow filter validation — likely as an L1-analogue surface scoped to the pull-request filter payload rather than the message payload. The four-layer framework absorbs the extension cleanly: `querySchema` is the L1-shaped artefact for a different payload shape; the framework's "one engine, one authoring surface, one contract per layer" rule lets it coexist with the message-payload L1 without confusion.
- **Production FloatingPanel adoption.** When `ui/apps/core/web/pitstop` mounts DSForm and wires the FloatingPanel primitive into [`Form.tsx`](../../../dex-monorepo/ui/libs/src/components/Forms/Form.tsx) (currently `showErrorList={false}` reserves the slot), the dex team can reference this ADR's sub-decisions 14, 15, 16, 17 as the consumer-side composition patterns the prototype demonstrated. The three extensions on the primitive (count badge, live-update, empty-state celebration) translate directly to React component-level changes without touching the primitive itself.
- **Backwards compatibility with pre-cutover Element versions.** Per CONTEXT.md *"Pre-cutover compatibility boundary"* + *"Bundle-shape rollback only"*, pre-cutover `x-*`-laden Element versions are immutable legacy history and are *not* valid Test-as-operator surfaces — Sarah testing a version she's authoring is testing a post-cutover version by construction. The Test modal does not need to handle the `x-*`-laden shape. If a legacy version is opened in Test-as-operator (e.g., via "+ New version of…" from a pre-cutover element), the modal initialises against the new version's post-cutover bundle, not the legacy snapshot — the snapshot is read-only history, not a Sarah-authoring surface.

## Alternatives considered

**Port FloatingPanel as-is (no layered framework).** Adopt the dex-monorepo FloatingPanel primitive's flat-list anatomy directly: one panel surfacing both AJV and govaluate errors with no engine attribution, no count split, no live-update, no section locality. Rejected: this is the failure mode angle-gauge-ui's framework was designed to prevent. The prototype's job per ADR 0043 §9 ("prototype-vs-production seam") is to *demonstrate the contract* production should adopt — porting the existing primitive's limitations directly would propose nothing. The ADR's "better than" position requires the layered framework as the design vocabulary.

**Two ADRs — one for AJV layer, one for FloatingPanel surface.** Split this work into a Schema-validation-rendering ADR and a separate cross-field-rules-rendering ADR. Rejected: the layered framework's value is the *single coherent taxonomy* — extracting one layer in isolation would lose the boundary discipline (where does L1 stop and L2 begin? what's the escape valve? which surfaces are Sarah-only?). The framework only works as a whole.

**Amend ADR 0038 instead of writing a new ADR.** Add the rendering decisions as a sub-section of the three-governance-layers ADR. Rejected: ADR 0038 is about *governance* (what the three layers are at the engine/artefact level); this ADR is about *surfacing* (how the prototype renders them to Sarah and to simulated operators). Different problems; bundling them would obscure both. The framework discipline (one ADR, one decision, clearly bounded) is the same discipline this ADR proposes for the surfaces — apply it consistently.

**Use `@rjsf/validator-ajv8` instead of vanilla AJV.** Production uses the RJSF wrapper; matching it gives maximum parity. Rejected per sub-decision 6: the wrapper's value is translating AJV `ErrorObject` into RJSF's `RJSFValidationError` shape — useful inside RJSF's render loop, useless outside it. The prototype isn't using RJSF for its Test modal renderer (the renderer is hand-rolled per the current pre-DSForm prototype state), so the wrapper is dead weight. Engine parity is what matters for the demo and engine parity is preserved by using the underlying AJV directly.

**Bundle the rule-to-field linking (sub-decisions 5, 14) into a separate follow-on instead of this ADR.** The link is technically additive over today's right-panel — the panel works without it. Rejected: the rule-to-field link is the closing move of Sarah's authoring debug loop (rule fails → which field is implicated?). Splitting it out would ship a half-experience and require a second ADR for what's mechanically a derivation function (`regExtractRuleFields`) plus a click handler. Cohesion wins.

**Defer the polymorphism design (sub-decision 18) until the JSON-editor ships.** `oneOf` can't currently be authored anywhere in the prototype, so the polymorphic-rendering work has no live consumers. Rejected: the framework's value compounds when surfaces are designed *with* their future extensions in mind. Sub-decision 13's schema-level error surface and sub-decision 18's picker-chip pattern both sit dormant for purely field-builder-authored schemas and activate automatically when JSON-editor constructs appear. Designing them now means the JSON-editor work has a Test-modal answer waiting; designing them later risks bolt-on patterns that don't fit the framework.

## Open questions

- **When does `ui/apps/core/web/pitstop` mount DSForm + FloatingPanel?** This ADR's prototype-side proposals are independent of the dex-monorepo timing, but the moment of production adoption is where the three extensions (count badge, live-update, empty-state celebration) need to make the cross-repo case. The pitstop bring-up team owns that decision and timing.
- **Should the boundary tests in sub-decision 20 run as part of the existing `npm run test` or as a separate `npm run test:validation-surfaces` invocation?** Pure organisational question — pick the convention that matches the existing test grouping in `portal-app/tests/`. Resolve at implementation time.
- **The L3 visibility behaviour for govaluate rules referencing hidden fields** — when `rules[i].expression` references a field that `uiRules.visibility` has hidden, what's the engine's input value for that field? Three options: `undefined` (current `regCollectTestPayload` would yield this if the input doesn't render), the field's last typed value (preserved when hidden but reappearing), or `null` explicitly (signals "intentionally absent"). govaluate semantics in production likely yield `undefined`; the prototype should match. Resolve at implementation time; the layer separation in this ADR is independent of the chosen semantic.
- **Whether Sarah can test pre-cutover legacy Element versions in Test-as-operator** — currently no, per *Forward-flexible*. If user research surfaces a need to "test what an old version's operator would see" (for audit/compliance contexts), a legacy-rendering mode for the Test modal would need its own scope decision. Not blocking this ADR.
