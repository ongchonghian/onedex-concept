# Data Element bundle export to admin-corev2 DataElement registry

> **Status:** accepted (2026-05-28)
> Builds on [ADR 0026](./0026-agreement-snapshot-immutability-schema-upgrade.md) (Element version immutability — the snapshot rule the registry must honour), [ADR 0038](./0038-data-element-governance-three-independent-layers.md) (the three-layer governance the bundle carries), [ADR 0039](./0039-data-element-registration-admin-authored-single-page-flow.md) (the registration flow whose Publish action this ADR extends), [ADR 0042](./0042-element-spec-sheet-onramp-deterministic-parser-audit-log-provenance.md) (the strip-at-publish + parallel-audit-log doctrine for authoring provenance), [ADR 0043](./0043-element-persistence-schema-driven-composer-workspace-v6.md) (the workspace persistence and the schema-walker shared module the bundle reads from). Closes the open question implicit since registration first shipped: *what artefact does a published Element version produce, and how does it cross the portal-to-platform boundary?*

## Context

The portal-app prototype publishes a fully-authored Element version to `workspace.dataElements` per ADR 0043 — `elementSchema` + `uiSchema` + `uiRules` + `composeComplexity` + `rules` + `pack` + `governance` + `authoringMetadata` — and the catalogue surfaces it as pickable for new Agreements. But the version never leaves the prototype browser. There is no path today by which Sarah's authoring effort becomes *a real DataElement in production*, where the dex-monorepo Go backend can validate PUSH / PULL / STORE / RECEIVE payloads against it.

The production landscape this ADR plugs into:

- **admin-corev2** (TypeScript Lambda, separate repo) owns the **DataElement registry** — Postgres-backed storage of `DataElement` and `DataElementVersion` rows, exposed via `POST /api/v1/data-element-versions` (gated by `authSuperSGTraDexAdmin`), `POST .../promote` for the `DRAFT → ACTIVE` lifecycle transition, and `GET .../:id/versions/:version` for read. The accepted body shape is `IDataElementVersion[]` ([`admin-corev2/src/repositories/dataElementVersion.ts:57`](../../../admin-corev2/src/repositories/dataElementVersion.ts:57)) with strict server-side validation: `changeType ∈ {INITIAL, MAJOR, MINOR, PATCH}`, `version` matches `\d+\.\d+\.\d+`, `(dataElementId, version)` is unique, semver ordering is enforced.
- **dex-monorepo** (Go services + React/TS UI) reads from the registry via `AdminConnectorService.GetDataElementVersionDetail` ([`be/sharelib/services/admin_connector/admin_connector_service.go:1546`](../../../dex-monorepo/be/sharelib/services/admin_connector/admin_connector_service.go:1546)). The producer's `validate.go:83` uses the fetched `elementSchema` for payload validation in the PUSH/PULL/STORE/RECEIVE pipelines. The Go backend does not duplicate admin-corev2's CRUD capabilities.
- **Form rendering** lives at `dex-monorepo/ui/libs/Forms/Form.tsx` — `@rjsf/core` v5.23.2 with custom widgets; the `uiSchema?: Record<string, unknown>` prop is RJSF-native. **No production app in `ui/apps/*` mounts DSForm yet** — `ui/apps/core/web/pitstop`, `tradex`, `buildex`, `healthdex` are Nx-welcome scaffolds. Production form rendering today still lives in legacy `pitstop-ui` with a hand-rolled `ValidatePayloadFunction` (no RJSF, no `uiSchema` concept) — being retired per CONTEXT.md preamble.

Two structural mismatches between the prototype's authored shape and admin-corev2's accepted shape made the bundle question non-trivial:

(1) The prototype's RJSF cutover (CONTEXT.md *Versioned element bundle*, **Interop-clean published schema**, **`x-*` zero-residue cutover**) split publication into three artefacts (`elementSchema` + `uiSchema` + `uiRules`). admin-corev2's `IDataElementVersion` has only `schema` — no `uiSchema`, no `uiRules`. The cutover landed in the prototype ahead of the registry.

(2) The prototype authors several per-version governance attributes admin-corev2 doesn't yet model — `rules` (govaluate per ADR 0038 layer 2), `pack` (per ADR 0013), `composeComplexity` (per ADR 0025), `governance.residencyStrict` (per ADR 0039). And admin-corev2 requires several fields the prototype doesn't author cleanly — `type` (DOCUMENT/EVENT/ETR), `changeType` (INITIAL/MAJOR/MINOR/PATCH), `changeDescription`, strict-semver `version`.

A grilling session walked the dependency tree from operator gesture to file shape: who consumes the export, what files ship, what each file contains, how authoring extensions on the Review tab source the missing fields, what the import flow looks like, how re-export works under immutability. The decisions below close that tree.

## Decision

The portal-app gains an [[element-bundle-export]] — a 4-file ZIP emitted at Publish-time and re-downloadable from the catalogue — that lands cleanly in admin-corev2's existing two-step lifecycle (`POST` then `/promote`) via a documented dev-relay curl flow. Thirteen sub-decisions:

**1. The consumer is a human dev running curl against admin-corev2.** The bundle is *exported to disk* (downloaded to the operator's Downloads folder), then *handed off* to a dex-monorepo dev — via PR, Slack, email, or shared drive — who runs the IMPORT.md-documented curl flow against admin-corev2. Rejected: direct portal → admin-corev2 API calls (the prototype has no production-auth context; coupling the prototype to live admin-corev2 would compromise its local-first character) and direct writes to `pitstop-core` / `dex_orchestrator_rules` (those entities aren't where DataElements live; admin-corev2 is the registry).

**2. The render target is `dex-monorepo/ui/apps/core/web/pitstop`, not legacy pitstop-ui.** The forward-spec sidecars (`uiSchema.json`, `uiRules.json`) are RJSF-shaped per the prototype's cutover. Legacy `pitstop-ui` uses a custom `ValidatePayloadFunction` with no RJSF; backporting `uiSchema` semantics there would either require porting pitstop-ui to RJSF (a project of its own) or re-merging UI directives into `elementSchema` as `x-*` keys (a cutover regression forbidden by the **`x-*` zero-residue cutover** entry). Explicit non-scope: legacy pitstop-ui. Forward target: `dex-monorepo/ui/apps/core/web/pitstop`, currently an Nx scaffold awaiting build-out.

**3. This task ships portal-app export only; admin-corev2 + dex-monorepo work is named follow-on.** Adding `uiSchema` / `uiRules` columns to admin-corev2, surfacing them via `AdminConnectorService`, mounting DSForm in `ui/apps/core/web/pitstop`, and building a `querySchema` authoring surface for PULL flows are each separate efforts in their own repos with their own ADRs. The portal's job per ADR 0043 §9 (*"prototype-vs-production seam"*) is to *demonstrate the contract* that production will implement — not to *be* production. Rejected: bundling cross-repo work into one task (multi-repo merge dependency stalls prototype velocity; obscures the prototype-side decision under integration plumbing).

**4. Bundle ships full-future-target shape, not narrow-today.** The on-disk artefact carries the contract admin-corev2 *will* accept once it grows columns — `uiSchema`/`uiRules` sidecars and inline forward-spec governance fields — not just what fits today's `IDataElementVersion` interface. The bundle is the spec for production; the import script narrows at the curl boundary. Rejected: narrow-today (would require restructuring the export every time admin-corev2 grows a column; encodes the wrong target shape) and portal-mirror (would ship `regDraft` raw with no admin-corev2 alignment, forcing the dev to translate at import time).

**5. Four-file bundle, fixed shape.** `dataElementVersion.json` (the canonical POST body, single-element array `[{...}]` matching `IDataElementVersion[]`), `uiSchema.json` and `uiRules.json` (forward-spec sidecars per sub-decision 6), `IMPORT.md` (the dev-facing curl handoff). `jsonTemplate` and `csvTemplate` are omitted — admin-corev2 server-derives them from `schema` at POST time ([`admin-corev2/src/repositories/data-element.ts:401-402`](../../../admin-corev2/src/repositories/data-element.ts:401)). `querySchema` is omitted per sub-decision 9. `authoringMetadata` is stripped at export per sub-decision 11.

**6. `uiSchema` and `uiRules` ship as [[forward-spec-sidecar]]s outside the POST body.** Both are RJSF-shaped artefacts the cutover deliberately separated from `elementSchema` per **Interop-clean published schema**. admin-corev2 has no column slot for either today; IMPORT.md documents that the dev does *not* include them in the curl payload. When admin-corev2 catches up, IMPORT.md updates to broaden the curl; no portal change needed. Rejected: re-merging UI directives into `elementSchema` as `x-*` keys (direct cutover regression, violates the **`x-*` zero-residue cutover** doctrine; the published artefact's serialiser unit test pins the contract per ADR 0042 sub-decision 8) and shipping the UI artefacts to a separate destination (creates partial-failure modes where elementSchema lands but UI artefacts don't, violating the **Versioned element bundle** atomicity rule).

**7. Per-version governance attributes inline into `dataElementVersion.json` as forward-spec keys.** `rules`, `pack`, `composeComplexity`, `governance.residencyStrict` ride inside the POST body the registry already accepts (admin-corev2 ignores unknown keys; once columns land, no payload change). Rationale: these are *missing columns on `IDataElementVersion`*, not separate artefacts requiring different runtime moments. Sidecaring them (as the cutover sidecared UI directives) would imply contractual separation that doesn't exist — the runtime moments are different but the *storage shape* converges on one row. Rejected: sidecar files per governance attribute (loses single-source-of-truth for the import payload; future maintainers can't predict where new governance attrs land) and hybrid inline/sidecar (no principled basis for the split).

**8. The Review tab gains four metadata fields.** Sarah authors `type` (3-chip: `Document` / `Event` / `ETR` — stable across versions of the same `dataElementId`; first-publish-only), `version` (strict semver `1.0.0` text input — replaces the legacy `v1.0` free-form prototype field; auto-incremented from prior version on `+ New version`), `changeType` (4-chip: `Initial` / `Major` / `Minor` / `Patch` — auto-defaulted from the schema diff against the prior published version), and `changeDescription` (textarea — required on version bump, optional on greenfield). The fields surface in a new "Version metadata" block on the Review tab, before the Publish CTA. Smart Start assist (per [ADR 0040](./0040-smart-start-assist-grounded-cross-tab-suggestion-engine.md)) gains a follow-on paragraph: when a grounding source names the element type (a Confluence page heading says "Bill of Lading" → ETR suggestion; an xlsx schema looks event-shaped → EVENT suggestion), assist may suggest `type`. Workspace schema bumps v6 → v7 to add `meta.type`, `meta.semverVersion`, `meta.changeType`, `meta.changeDescription` to the `dataElements[id@version]` record; archive-and-rebuild on first read per the existing pattern (schemas v2/v3/v4/v5/v6).

**9. `querySchema` is deferred; PULL flows for newly-registered elements degrade until a follow-on authoring surface lands.** querySchema is the schema for the *pull-request filter payload* (`be/producer/internal/dto/package_filters.go:29`, `get_config.go`, `config_adapter.go:GetQuerySchema`) — a separate authoring artefact from `elementSchema`. The prototype has no concept of querySchema today; authoring it requires its own Review-tab surface (or new tab) and its own ADR. IMPORT.md explicitly names that PULL filter behaviour for the imported element will default to empty-filter until querySchema is authored separately. PUSH / STORE / RECEIVE remain fully functional. Rejected: auto-deriving querySchema from elementSchema (too risky — auto-exposing all fields as PULL filters creates privacy and performance bombs; auto-inferring only "simple" fields silently omits the actually-queried key) and inlining querySchema authoring into this task (doubles scope; PULL filter authoring deserves its own grilling against ADR 0038's three-layer separation).

**10. Auto-download on Publish + catalogue re-download.** The Publish CTA fires `regPublish()` → writes to `workspace.dataElements` → builds the ZIP → triggers browser download in one gesture. A success toast names the artefact path. Subsequent re-downloads happen via a per-row "Re-download bundle" affordance on the catalogue, gated by `canManageDataElements`. The re-download is read-only — emits a `bundle-re-exported` audit-log event but doesn't mutate the workspace record. Rationale: Publish is the natural materialization moment; re-download covers env-replay (dev → qa → prod) and lost-file recovery without violating ADR 0026's immutability (re-emission is a function of the immutable record, not a republish). Rejected: separate Export affordance with no auto-download (forces Sarah to click twice for the always-needed first artefact) and auto-download with no re-download path (breaks the multi-env story).

**11. Bundle ships `status: 'DRAFT'`; IMPORT.md documents two curls.** `dataElementVersion.json` explicitly carries `status: 'DRAFT'`. IMPORT.md's flow:
1. `POST /api/v1/data-element-versions` with the bundle's body → creates the version as `DRAFT` (admin-corev2 default; `publishedDate=NULL`).
2. `POST /api/v1/data-element-versions/:id/versions/:version/promote` with `{targetStatus: 'ACTIVE'}` → transitions to ACTIVE; triggers email notifications and audit-trail logging on admin-corev2's side.

Rationale: the prototype's Publish ≠ admin-corev2's ACTIVE. Sarah's Publish is "authored, finished, packaged"; admin-corev2's ACTIVE is "live in the platform." Two separate authorisation gates in two separate systems. The /promote endpoint exists precisely so an env-specific smoke-test step can sit between import and live-status. Rejected: `status: 'ACTIVE'` on POST (even if accepted, bypasses /promote's email-notification and audit code paths — production-wrong) and omitting `status` (unnecessarily ambiguous about what the bundle claims).

**12. `authoringMetadata` is stripped at export — audit trail stays in workspace.** The provenance carried in `workspace.dataElements[*].auditTrail` per ADR 0043 sub-decision 3 — Smart Start suggestions accepted/edited/rejected, on-ramp source file/hash/row, publishing user — never leaves the prototype browser. The bundle on disk has no `authoringMetadata.json`. Rationale: ADR 0042 sub-decision 8 establishes provenance as audit-log-only at the publish boundary; the export is a publish boundary. A reviewer of the bundle PR who needs to understand authoring history opens the prototype workspace. Rejected: shipping `authoringMetadata.json` as a review-only sidecar (correct-per-ADR-0042-intent but introduces a fifth file the registry has no consumer for; the audit story is better served by the prototype's workspace `auditTrail` than by an on-disk parallel) and embedding `x-source` per-field on elementSchema (direct ADR 0042 violation; the serialiser strip-test forbids it).

**13. IMPORT.md is `$BASE_URL`-parameterized; ZIP via JSZip; deterministic regeneration.** The dev sets `BASE_URL` once via an exported env var to pick dev/qa/prod (matching admin-corev2's own docs' `$TOKEN` idiom in [`local-testing-curls.md`](../../../admin-corev2/docs/local-testing-curls.md)). The bundle ZIP is built client-side via JSZip (~50KB dep; precedent set by SheetJS for ADR 0042's spec-sheet on-ramp) with filename `${elementId}-${semver}.zip` (lowercase, hyphen-separated, semver-suffixed; `@` from the workspace key replaced by `-` for Windows filename safety). Re-downloads always regenerate from `workspace.dataElements[id@version]` with stable-sorted JSON keys (`JSON.stringify(obj, Object.keys(obj).sort(), 2)` or equivalent) — never a cached blob. Rationale: stable serialization gives byte-equality without the storage cost of caching ~10–100KB per published version; the workspace record per ADR 0026 immutability is the single source of truth.

## Consequences

**Architectural:**

- Workspace schema bumps v6 → v7 to add Review-tab metadata fields (`meta.type`, `meta.semverVersion`, `meta.changeType`, `meta.changeDescription`) to the `dataElements[id@version]` record. Existing v6 snapshots archive-and-rebuild per the established pattern.
- `regPublish()` ([`register-element.js:9467`](../../portal-app/scripts/register-element.js:9467)) extends: after persisting to `workspace.dataElements`, builds the 4-file bundle, invokes JSZip, triggers browser download. Emits a `bundle-emitted` audit-log event.
- A new module `portal-app/scripts/element-bundle-export.js` (or comparable filename) owns the bundle assembly: pure-function `buildBundle(versionRecord) → Blob` consuming the workspace `dataElements[id@version]` shape and producing the ZIP. Stable serializer lives here.
- The catalogue's per-row affordance set extends with a "Re-download bundle" link, gated by `canManageDataElements`. Click rebuilds the ZIP from current `workspace.dataElements`; deterministic.
- The Review tab ([`register-element.js`](../../portal-app/scripts/register-element.js) Review-tab handler) gains a "Version metadata" block — `type` chip group (DOCUMENT/EVENT/ETR), `version` semver text input, `changeType` chip group (INITIAL/MAJOR/MINOR/PATCH), `changeDescription` textarea. Auto-defaults populate from the schema diff at Review-tab entry; Sarah confirms or overrides.
- The prototype's legacy `meta.version` free-form field (`v1.0`) is retired in favour of `meta.semverVersion` (`1.0.0`); the existing `meta.category` free-text field is retired in favour of `meta.type` (enum-constrained). Workspace migration translates legacy fixture data where possible (`'v1.0'` → `'1.0.0'`, free-text category → `DOCUMENT` default).
- JSZip is added as a vendor dependency under the same idiom SheetJS was added for ADR 0042.

**Behavioural / contractual:**

- After this lands, Sarah's Publish gesture produces a *real, hand-offable artefact*. The portal-to-platform gap that ADR 0043 §9 named as a "prototype-vs-production seam" gains a concrete crossing point.
- A dex-monorepo dev who receives a bundle can import it into admin-corev2 with two curls — no Excel relay, no Swagger YAML edit, no manual SwaggerHub copy, no 4-POSTMAN-call relay. The 9-step legacy CSM relay (per ADR 0038 Context) is compressed to two documented curls.
- Once imported and promoted, the registered DataElement is immediately usable by dex-monorepo's PUSH / PULL / STORE / RECEIVE pipelines — the same Go backend that already consumes admin-corev2 via `AdminConnectorService` picks it up transparently. PULL filter behaviour degrades to empty-filter until a follow-on querySchema authoring surface lands.
- The bundle is *honest about its forward-spec shape*. A reviewer who opens the ZIP sees the contract production *will* implement; IMPORT.md names which fields the registry consumes today and which sit in the bundle waiting. No silent pollution of `elementSchema`; no hidden coupling.
- The Publish-time auto-download means every published version's artefact lands on Sarah's machine without a separate gesture. The catalogue re-download covers env-replay and lost-file recovery.

**Forward-flexible:**

- **admin-corev2 follow-on (separate ADR in that repo).** Grow `DataElementVersion` with `uiSchema` (JSONB) and `uiRules` (JSONB) columns; extend `IDataElementVersion` interface; accept the fields on `POST /data-element-versions`. The bundle's `uiSchema.json` and `uiRules.json` sidecars are already shape-ready; IMPORT.md updates to add `--data-binary @uiSchema.json` (or merge into the POST body) — no portal change. When the additional inline governance fields (`rules`, `pack`, `composeComplexity`, `governance.residencyStrict`) gain column slots, IMPORT.md needs zero update because they already ride in the POST body.
- **dex-monorepo follow-on (separate ADR in that repo).** `AdminConnectorService.GetDataElementVersionDetail` surfaces the new fields when admin-corev2 returns them. `ui/apps/core/web/pitstop` builds out the EForm rendering page, mounting `ui/libs/Forms/DSForm` with `schema={elementSchema}`, `uiSchema={uiSchema}`, and a `uiRules`-driven custom field-template for conditional visibility. Until that app is built, the sidecars sit in the bundle as documented spec.
- **querySchema authoring follow-on (separate ADR in this repo).** A new authoring surface for PULL filter contracts — likely a fifth tab on the registration canvas or an extension within an existing tab. The bundle gains a fifth file `querySchema.json` (inline in `dataElementVersion.json` if admin-corev2 keeps the existing column, sidecar if a separate endpoint is decided). The current ADR's bundle shape is forward-compatible — adding a fifth file is non-breaking.
- **Registry-side audit-trail-import follow-on (speculative).** If admin-corev2 grows an audit-trail-import endpoint, the bundle could optionally start shipping `authoringMetadata.json` (reversing sub-decision 12). The strip-at-export decision is per-this-ADR doctrine, not an irreversible architectural commitment.
- **Bulk export (speculative).** The current bundle is per-element-version. A future "Export all elements in pack X" affordance would ship a multi-element ZIP or a manifest-pointing-to-N-ZIPs. The current bundle shape (4 files, fixed names) is preserved as the per-element unit.
- **CI-friendly import.** The `$BASE_URL`-parameterized IMPORT.md is consumable by CI pipelines (set env vars, run the two curls). Future automation in dex-monorepo's deploy pipeline could replace the manual dev relay with a CI job that watches a bundle drop location.
