# Element spec sheet on-ramp: deterministic parser, audit-log-only provenance, refit three-way merge

> **Status:** accepted (2026-05-23)
> Builds on [ADR 0039](./0039-data-element-registration-admin-authored-single-page-flow.md) (the single-page registration flow this on-ramp slots into), [ADR 0040](./0040-smart-start-assist-grounded-cross-tab-suggestion-engine.md) (Smart Start assist's grounded suggestion contract that consumes the parser's sidecar metadata), [ADR 0041](./0041-smart-start-refit-mid-edit-structural-restatement.md) (the refit pipeline the `+ New version` mode reuses), [ADR 0026](./0026-agreement-snapshot-immutability-schema-upgrade.md) (Element version immutability — re-imports produce new versions, never mutate published artefacts), [ADR 0038](./0038-data-element-governance-three-independent-layers.md) (the three-layer governance the published artefact must respect).

## Context

A grilling session walked through the worked example of [`drp-schema.json`](../../drp-schema.json) — a published JSON Schema that was authored by hand from `DRP & DFS Field Sequences For E-Form.xlsx`, a business-requirements spreadsheet where **each row defines one field** of the DRP data element (business term, definition, mandatory/optional, technical name, type, validation rule). The published schema had drifted out of sync with the xlsx (the xlsx had added `sp_hci` after the schema shipped) because there was no obvious affordance to feed the updated xlsx back into the element — re-authoring by hand from a 49-row spec sheet is brutal enough that the dominant outcome was *"don't update."* The four existing Smart Start seed on-ramps (Fork, Sample, Plain English, Form/PDF) all fail this artefact for distinct reasons: the spec sheet has no records to infer from (defeating Sample), is not an existing element (defeating Fork), is too structured to need NL parsing (defeating Plain English), and is not lossily-extracted prose (defeating PDF/Form's VLM path). A deterministic xlsx parser is wanted.

Two separately-grounded constraints shape what gets shipped: (1) the published `elementSchema` artefact is **interoperability infrastructure** — consumed by other DEX systems, counterparties, and downstream tooling on stacks the portal does not control — so the published shape must stay close to standard JSON Schema vocabulary plus production-established advisory extensions, never carrying portal-internal authoring metadata. Production evidence corroborates: a scan of all 165 elementSchemas in [`sgtradextech-data-element-dev.json`](../../../local-dev/data/dynamodb/sgtradextech-data-element-dev.json) shows zero authoring-provenance `x-*` keys; the only `x-*` extensions present are 5 × `x-uitype` and 3 × `x-uioptions` on the eBL DCSA element's `segregation_groups` field — render-directives, not provenance. (2) The element-version-snapshot rule from [ADR 0026](./0026-agreement-snapshot-immutability-schema-upgrade.md) means existing Agreements remain bound to the prior schema even after a new version publishes; the staleness-refresh case must produce a **new Element version**, never overwrite a published one, and the refresh path must preserve Sarah's hand edits per [ADR 0041](./0041-smart-start-refit-mid-edit-structural-restatement.md)'s "preserve existing work" contract.

## Decision

A fifth Smart Start seed on-ramp ships: **Element spec sheet** — sibling to Fork / Sample / Plain English / Form/PDF — with a deterministic XLSX/XLS/CSV parser, an audit-log-only provenance trail, and a three-way refit-mode merge that preserves Sarah's hand edits on conflict. Ten sub-decisions:

**1. The on-ramp is concept-named *Spec sheet* (canonical: *Element spec sheet*).** The launcher tile reads "Spec sheet" with sub-label *"Each row defines one field — XLSX, XLS, CSV"*. The CONTEXT.md term is *Element spec sheet* to avoid the bare-noun ambiguity of "spec." Rejected names: *XLSX import / Spreadsheet* (function-named — breaks the concept-named on-ramp pattern of Fork / Sample / Plain English / Form/PDF), *Requirements sheet* (collides head-on with [ADR 0040](./0040-smart-start-assist-grounded-cross-tab-suggestion-engine.md)'s *Confluence requirements page* grounding source — two artefacts both called requirements-something with different roles), *Data dictionary* (resurrects retired `SIDEBAR_ITEMS_PLATFORM` IA vocabulary), *Field sequence* (the source-filename idiom — too narrow, implies ordering only while the artefact carries types, mandatory/optional, and validation).

**2. The on-ramp drives both `+ New element` (seed mode) and `+ New version` (refit mode), with one parser and two downstream merge paths.** Seed mode lands the parsed output on an empty canvas — identical to how the other four on-ramps behave. Refit mode diffs the parsed output against the prior published Element version's schema and surfaces the diff through the existing refit drawer per [ADR 0041](./0041-smart-start-refit-mid-edit-structural-restatement.md). Hiding the on-ramp from `+ New version` would re-create the asymmetry that produced the staleness problem — there'd still be no path back from an updated xlsx to an existing element. The shared parser keeps both paths deterministic; the merge logic is the only thing that branches.

**3. Multi-sheet workbooks resolve to one element per import trip via a sheet picker.** The on-ramp modal surfaces a sheet-picker chip row when the dropped file has ≥2 sheets; Sarah picks one (DRP *or* DFS) and the parser ingests that sheet only. Multi-element batch (one xlsx → N elements) was rejected because it forces a multi-element draft state and a new multi-publish step that the existing single-element flow contract ([ADR 0039](./0039-data-element-registration-admin-authored-single-page-flow.md)) does not carry. Auto-pack inference (detecting that DRP + DFS share a header set and emitting them as a **Data element pack**) was also rejected — pack membership is Sarah's editorial decision per the [CONTEXT.md] **Data element pack** entry, not a parser inference. The workbook is cached in `sessionStorage` keyed by `sha256(file)` so re-opening the on-ramp for the sibling sheet inside a session is a one-click step (sheet picker pre-shows the workbook); a post-publish catalogue toast surfaces *"You imported DRP from a workbook with 2 sheets. Import DFS now?"* with a deep-link, so the sibling-import case is obvious without being paternalistic. In refit mode, the sheet picker filters by an overlap heuristic (`≥80%` overlap between the sheet's `Technical Data Field Name` set and the prior version's property names) — if exactly one sheet qualifies, the picker is hidden and the qualifying sheet is auto-selected; if zero qualify, the modal surfaces *"This workbook's sheets don't match {ElementName}'s prior fields — are you sure this is the right element to bump?"* as a guardrail against accidentally re-importing the wrong element's spec.

**4. The parser is deterministic-only; all inference lives in Smart Start assist / refit.** Each xlsx row maps to exactly one seed field via a fixed column-to-property mapping: `Technical Data Field Name` → property key; `Business Term` → `title`; `Business Definition` → `description`; `Mandatory / Optional` → top-level `required[]` membership; `Data Field Type` → JSON Schema `type` (with the type-mapping table below); `Data Validation Rule` → parsed `minLength` / `maxLength` / `minimum` / `maximum` / `pattern` where the prose matches the regex set. **No parse-time inference is performed.** Cartesian decomposition of `*_le` / `*_re` eye-pair fields (signalled by the *Classification = Odd/Even* column in the xlsx), conditional-required translation (the validation column's `NOT NULL if sp_uen = NULL` prose → JSON Schema `anyOf`), and enum extraction from definition prose (`[i.e. Selection: 1 - Female; 2 - Male]`) all live downstream in Smart Start assist / refit. The boundary is enforced by ADR 0040 §17's shift-left structural-classification contract: a parse-time inference engine inside the on-ramp would re-create the two-engine reconciliation problem §17 was written to prevent. The parser preserves source signals that no other grounding source carries (the *Classification* column, the original validation prose, the original definition prose) as sidecar metadata on each seed field so Smart Start assist can read them as grounding inputs.

   **Type-mapping table** (xlsx → JSON Schema):
   | xlsx `Data Field Type` | JSON Schema output |
   |---|---|
   | `string` | `"type": "string"` (mandatory) or `"type": ["string", "null"]` (optional) |
   | `int8` | `"type": "integer"` with `minimum` / `maximum` parsed from validation column |
   | `Boolean` | `"type": "boolean"` (mandatory) or `"type": ["boolean", "null"]` (optional) |
   | `date-time` | `"type": "string"` + `"format": "date-time"` |
   | `array` | `"type": "array"` with `items` scaffold (default: object items, no children — assist suggests the items shape later) |
   | unknown | preview shows a warning chip on the row; field is emitted as `"type": ["string", "null"]` as a safe default; assist's review flags it |

   The optional-anything nullability pattern (`["string", "null"]`, `["boolean", "null"]`, `["object", "null"]`) is the production-canonical shape — all 165 elements in [`sgtradextech-data-element-dev.json`](../../../local-dev/data/dynamodb/sgtradextech-data-element-dev.json) follow it; the parser matches.

**5. Per-field seed sidecar shape: nested `x-source` object.** The parser writes one `x-source` object per seed field carrying `{ kind: 'spec-xlsx', file, fileHash, sheet, row, headerRow, classification, validationProse, definitionProse }`. The nested form (one `x-source` key per field, structured inside) was chosen over a flat `x-source-*` prefix scheme because it matches how other on-ramps will eventually shape their own provenance (`x-source.kind = 'pdf-vlm'` for Form/PDF, `x-source.kind = 'fork'` for Fork, etc.) — a single union type, not N parallel flat schemes. `row` is 1-indexed to match the xlsx UI. `fileHash` doubles as the workbook-cache key from sub-decision 3. The sidecar is **internal to authoring** — it rides on `regDraft` through the editing session, is consumed by Smart Start assist and the refit diff engine, and is **stripped at publish time** per sub-decision 8.

**6. Suggestion provenance shape: `source.suggested.from` carries a parser-row pointer plus a signal slot.** When Smart Start assist proposes a downstream transformation grounded in the parser's sidecar (an enum promotion on `patient_id_gender` grounded in the `definitionProse` pattern; a Cartesian decomposition grounded in the `classification = Odd/Even` signal; an `anyOf` translation grounded in the `validationProse`), the suggestion envelope's `source.suggested.from` carries `{ kind: 'spec-xlsx', file, sheet, row, signal: 'definitionProse' | 'classification' | 'validationProse' | 'row-diff' }`. The `signal` slot names *which* sidecar field grounded the suggestion — auditable, debuggable, and future-readable. This extends [ADR 0040 §32](./0040-smart-start-assist-grounded-cross-tab-suggestion-engine.md)'s universal suggestion envelope, not replacing it.

**7. Refit-mode merge is three-way with Sarah-wins on conflict.** When the parsed output (L2) is reconciled with `regDraft` (L1) against the prior published Element version's schema (L0), the diff engine emits one suggestion entry per non-silent diff:

   | L0 | L1 | L2 | Behaviour |
   |---|---|---|---|
   | absent | absent | present | **`add`** suggestion, default-accept |
   | present | unchanged (= L0) | changed | **`modify-untouched`** suggestion, default-accept |
   | present | changed by Sarah | changed differently | **`edit-conflict`**, **default to Sarah's value**, explicit *"take xlsx version (discards your edit)"* affordance |
   | present | deleted by Sarah | still present | **`delete-conflict`**, **default to Sarah's delete**, explicit *"re-add"* affordance |
   | present | absent (Sarah's local addition; not in L0) | absent | **silent — preserve Sarah's addition** |
   | any | == L2 | == L1 | **silent — agreement** |

   `Sarah-wins on conflict` is the load-bearing UX choice — [ADR 0041](./0041-smart-start-refit-mid-edit-structural-restatement.md)'s "preserve existing work" contract requires that a re-import never silently overwrite a deliberate edit. The default-accept on adds + untouched-modifies is what makes the staleness-refresh case one-click (Sarah's motivating example — `sp_hci` appears in the new xlsx, never existed in L0 or L1, surfaces as one `add` suggestion, one click). The diff entry shape is `{ field, kind, l0, l1, l2, sarahTouched: bool, source: { suggested: { from: { kind: 'spec-xlsx', sheet, row, signal: 'row-diff' }, at } } }` — slotting into refit's existing suggestion drawer per [ADR 0041 sub-decision 3](./0041-smart-start-refit-mid-edit-structural-restatement.md). Rejected: xlsx-wins (violates ADR 0041 head-on), never-overwrite-anything (over-preserves — locks out the exact updates the on-ramp exists to enable), always-suggest-with-no-defaults (high enough friction to discourage refit use).

**8. `x-source` is audit-log-only; the published Element version's schema carries no authoring provenance.** Per the [CONTEXT.md] **Interop-clean published schema** principle, the published `elementSchema` artefact is an interoperability contract consumed by other DEX systems. Authoring provenance is irrelevant to that contract — production has zero `x-source` / `x-provenance` / `x-imported-from` keys across 165 elements. The serialiser strips `x-source` from every property body on publish. The audit log's `element-version-published` event carries the same provenance in parallel — `{ versionId, publishedAt, publishedBy, fields: [{ name, source: { imported: { kind: 'spec-xlsx', file, fileHash, sheet, row }, suggestionsAccepted: [...] } }] }`. Refit's cross-version diff annotations in the *next* re-import (where v(N+1)'s import diffs against v(N)) read the prior version's row references from the audit log, not from the schema. The Go model layer ([`be/sharelib/services/models/data_element.go`](../../../dex-monorepo/be/sharelib/services/models/data_element.go)) treats `ElementSchema` as opaque `JSON` and does not enforce the principle — the serialiser is the only gate, so a unit test on the serialiser pins the contract: *"published artefact contains no `x-*` keys outside the known-advisory allowlist."*

**9. Render-directive `x-*` namespace drift is acknowledged and explicitly out of scope.** Production ships render-directives as `x-uitype` and `x-uioptions` (the eBL DCSA element); the portal prototype invents `x-presentation` for the same role per CONTEXT.md's *Presentation panel* entry. The Spec-sheet on-ramp does *not* author render-directives at parse-time (sub-decision 4), so its wire output is unaffected by whichever namespace wins. The drift is flagged in CONTEXT.md *Flagged ambiguities* for a separate grilling and remains unresolved here — when Sarah uses the Presentation panel post-import to add render-directives, they go out under whichever namespace the panel currently uses, and the future namespace reconciliation will sweep both spec-sheet-imported and non-imported elements alike.

**10. Capability gate: `canManageDataElements`, same as the rest of registration.** No new capability needed; the spec-sheet on-ramp is part of the same authoring act as the existing four on-ramps per [ADR 0039 §4](./0039-data-element-registration-admin-authored-single-page-flow.md). The CONTEXT.md `canManageDataElements` entry's existing language already covers it.

## Consequences

**Architectural:**

- `regDraft.fields[i]` shape extends with an optional `x-source` sidecar carrying the parser provenance. The sidecar is internal-only; the serialiser at publish time strips it. A serialiser unit test guards the contract (*"published artefact contains no `x-*` keys outside the known-advisory allowlist of `x-uitype`, `x-uioptions`, and — until namespace resolves — `x-presentation`"*).
- The audit log's `element-version-published` event shape extends with per-field `source.imported` and `source.suggestionsAccepted` slots. Backward-compatible; older events without these slots remain valid (they describe non-imported elements).
- A new module `portal-app/scripts/register-onramps-spec-sheet.js` (or equivalent) loads SheetJS lazily on first open of the on-ramp; the SheetJS dependency does not bloat first paint.
- The launcher modal in [`portal-app/index.html`](../../portal-app/index.html) gains a fifth on-ramp tile in both `openNewElementFlow()` and `openNewVersionFlow()`.
- The refit drawer ([ADR 0041 sub-decision 3](./0041-smart-start-refit-mid-edit-structural-restatement.md)) gains a new suggestion-source kind: `kind: 'spec-xlsx-diff'`. Its cards render the diff-entry shape from sub-decision 7; the *Apply* action takes a diff entry and mutates `regDraft` accordingly.
- `sessionStorage` gains a workbook cache: `sessionStorage[`spec-sheet-workbook:${fileHash}`]` carries the parsed workbook structure (sheet names + parsed seeds per sheet) for the session. Cleared on tab close. No persistence beyond the session.

**Behavioural / contractual:**

- Sarah's hand edits in refit mode are inviolable defaults on conflict; the on-ramp can never overwrite them without an explicit click. The conflict-resolution UI must surface the *take xlsx version* affordance prominently enough to be discoverable but not so prominently that mis-clicks happen.
- Sibling elements (DRP + DFS from one workbook) are imported as separate trips through the on-ramp. The sibling-element ergonomic is carried by `sessionStorage` workbook caching plus a post-publish nudge — not by a multi-element-batch flow.
- Re-imports always produce new Element versions; they never mutate published artefacts per [ADR 0026](./0026-agreement-snapshot-immutability-schema-upgrade.md). Existing Agreements snapshotted against v1.0 of DRP remain bound to v1.0's schema even after v1.1 publishes with the new `sp_hci` field.

**Forward-flexible:**

- The parser's column-to-property mapping is the seam where future xlsx schema dialects can plug in. If a DEX adopts a different spec-sheet column convention (different header labels, different validation-prose grammar), a per-DEX mapping plugin can be added without changing the parser core. v1 ships with the DRP/DFS dialect baked in; multi-dialect support is deferred.
- The `signal` slot in `source.suggested.from` (sub-decision 6) is the seam where future parser sidecars (e.g., a future *Examples* column carrying example values) can ground suggestions without changing the envelope.
- The audit-log-only provenance pattern (sub-decision 8) is the model for how the *other* on-ramps should evolve. Today PDF/Form may carry implicit provenance differently; future cleanup can normalise all five on-ramps onto the same `source.imported` audit-event shape.

## Considered alternatives

- **Extending the Sample on-ramp with xlsx detection** instead of adding a fifth on-ramp — rejected because Sample's contract is *infer fields from records*; spec-sheet rows are field-definitions, not records. The header-row heuristic ("does row 1 contain `Technical Data Field Name`?") would technically work but it makes Sample's inference logic carry two unrelated parsers behind one door, violating the on-ramp-per-shape design from [ADR 0039 §3](./0039-data-element-registration-admin-authored-single-page-flow.md).
- **Treating the xlsx as a grounding source for Smart Start assist** (peer to the Confluence requirements page from [ADR 0040](./0040-smart-start-assist-grounded-cross-tab-suggestion-engine.md)) rather than as an on-ramp — rejected because the xlsx parse is deterministic; demoting it to a grounding-only role would force Sarah to author from a blank state and accept suggestions one by one, wasting the determinism. The xlsx-as-grounding-source role is still useful and additive later for cross-source reconciliation — but is a separate concern.
- **Inference-rich parser** (Cartesian-decompose at parse-time, translate prose to `anyOf` at parse-time, extract enums at parse-time) — rejected because it re-creates the two-engine reconciliation problem [ADR 0040 §17](./0040-smart-start-assist-grounded-cross-tab-suggestion-engine.md) was written to prevent, and because pre-decomposed seeds break the refit-mode diff (the diff would compare post-decomposition shapes against L0's flat shape non-deterministically).
- **Persisting `x-source` on the published Element version's schema** — rejected because the published artefact is an interoperability contract and production has zero authoring-provenance precedent across 165 elements. The audit log is the durable, decoupled home for provenance.
- **Xlsx-wins / never-overwrite / always-suggest** refit merge policies — rejected on the analysis in sub-decision 7. Three-way Sarah-wins is the only policy that simultaneously honours [ADR 0041](./0041-smart-start-refit-mid-edit-structural-restatement.md)'s preserve-existing-work contract and makes the staleness-refresh case one-click.
