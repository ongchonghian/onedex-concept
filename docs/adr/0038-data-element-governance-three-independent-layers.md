# Data Element governance is three independent layers

> **Status:** accepted (2026-05-20)
> Names a separation that already exists in the dex-monorepo runtime systems and surfaces it as first-class structure in the new admin registration UI. Foundational for [ADR 0039](./0039-data-element-registration-admin-authored-single-page-flow.md).

## Context

The legacy Data Element creation/update process (captured in *update process-190526-231734.pdf*) is a nine-step relay: CSM prepares an Excel → CSM raises an SR → SR approved → Dev edits Swagger YAML in the repo → PR → merge → Leads manually copy the YAML into SwaggerHub per environment → Leads make four POSTMAN calls to mockserver (`/getSwaggerDataElements`, `/usercasepersonas`, `/verificationFlow`, `/personas`) → DevOps restarts the relevant pitstops. The pipeline hides what the dex-monorepo runtime already separates into three different systems:

1. **JSON Schema validation** — `@rjsf/validator-ajv8` on the frontend renders forms and runs structural validation; the same schema is enforced at submission. This is the *shape* layer: types, required, format, nesting, polymorphism (`oneOf`).
2. **Expression rules engine** — `Knetic/govaluate` at `be/sharelib/mock/mock.go` evaluates expression strings against payload params. Header rules at `be/sharelib/services/config_cache/extra_header.go` extend the same shape with goja JS scripts and a `Rule struct` carrying patterns. This is the *cross-field behaviour* layer: `end_date > start_date`, "if X = 'foo' then Y is required", `total === line_items.sum(...)`.
3. **Orchestrator rules entity** — `dex_orchestrator_rules.go` persists `RECEIVING` / `SENDING` rules as JSONB. The validator framework at `be/aggregator/internal/consumer/validator` plugs these in with `IValidator` / `IValidationManager`, priority ordering, and `FailFast` / `CollectAll` strategies. This is the *routing* layer: where the message goes post-submission based on payload content.

Three separate runtimes, three separate evaluators, three separate authoring conventions today — all of which a CSM in the legacy flow had to coordinate through Excel + Dev relay + per-env manual replay. The new portal surface must name and separate these layers, not paper over them. Conflating them inside a single "JSON Schema editor" is the YAML mindset; surfacing them as one tab each reflects what the runtime already does and lets each layer have its own authoring affordances.

The decision below establishes the conceptual model. [ADR 0039](./0039-data-element-registration-admin-authored-single-page-flow.md) decides how the model is surfaced in the registration UX.

## Decision

A Data Element carries **three independent governance layers**. Each layer has a distinct author surface, evaluator, runtime moment, and lifecycle.

| Layer | Authored as | Evaluator | Runtime moment | Lifecycle scope |
|---|---|---|---|---|
| **Schema** | JSON Schema (field-builder + read-only JSON preview) | AJV + RJSF (`@rjsf/validator-ajv8`) | Form-render time at Composer; same schema enforced at submit | Per **Element version** snapshot |
| **Validation rules** | govaluate-style expressions, each with `name + expression + on_failure` | govaluate engine (`be/sharelib/mock/mock.go`) | Composer submission, after AJV passes | Per Element version attachment |
| **Routing rules** | `dex_orchestrator_rules` JSONB (`RECEIVING` / `SENDING`) | orchestrator, via validator framework at `be/aggregator/internal/consumer/validator` | Post-submission orchestration | Per-DEX scope, cross-version |

Six sub-decisions:

**1. Schema layer owns *shape*, never *behaviour*.** Conditional schemas (`oneOf` for true polymorphism — union types where the same field can be one of N shapes) live in JSON Schema and are authored via the JSON-editor side of the Schema tab. *Behavioural* conditionals — "if field X = 'foo', field Y is required" — live in Validation rules instead. JSON Schema's `if/then/dependencies` exist but are deliberately not used for behavioural conditions because (a) they're verbose, (b) they're poorly supported by builder UIs, (c) the govaluate engine already runs at submission and has a more readable syntax. The Schema tab field-builder exposes only the constructs it can render cleanly (types, required, per-field validation, nesting up to 3 levels); the JSON editor handles polymorphism and anything else exotic.

**2. Validation rules own *cross-field behaviour*.** Every rule is `(name, expression, on_failure, applies_at)`. The expression DSL is the existing govaluate syntax — what ops have been writing in code today, now authored by DEX admins in a dedicated tab. `applies_at` is `validation` in v1; `routing` is reserved for a future bridge to the orchestrator layer. The Rules tab is **always visible** from registration day one — opting in via a "Show advanced" toggle would tell admins rules are an afterthought, which contradicts both the runtime reality (rules exist in production today) and the safety story (cross-field violations are how high-stakes documents become silently wrong).

**3. Routing rules are out of scope for the registration UI in phase 1.** The Routing tab is a visible placeholder framed as *"managed by orchestrator admin · phase 2."* Three reasons: (a) `RECEIVING` / `SENDING` JSONB authoring is a different mental model from validation rules and bundling them dilutes both demos; (b) routing intersects [ADR 0028](./0028-routing-is-not-an-agreement-property.md) in subtle ways that need their own grilling; (c) routing rules are cross-version (one rule applies to all versions of the element), while Schema and Validation rules are per-version — placing them in a per-version-author flow without resolving that scope mismatch would mis-shape the UX.

**4. The three layers are independent, not pipelined.** A new element with a schema, zero validation rules, and no routing rules is valid. A new element with just one validation rule and no schema is *not* valid (the schema is the foundation that the rule references field names against). Routing rules can be authored independently for an existing element without re-publishing. The dependency is one-way: Validation rules reference Schema fields by name; nothing references the reverse. Routing rules reference Schema fields by JSON-path; nothing references the reverse.

**5. Each layer evaluates at its native runtime moment, not in a shared pipeline.** AJV runs as the operator types in the Composer (live feedback on shape). govaluate runs at Submit (cross-field check before the message is accepted). Orchestrator rules run after the message is in the system (routing). Conflating these into a "validate everything at submit" pipeline would force AJV's per-keystroke feedback to wait for govaluate, and would force every routing decision to block submission — both regressions on the existing runtime.

**6. The three-layer model articulates what already exists; it does not invent new runtime.** The validator framework, the govaluate engine, the orchestrator entity, and AJV + RJSF are all in the codebase today. This ADR names the separation and lifts it into product vocabulary so the portal can surface it. No backend changes are implied.

## Why this is consistent with existing doctrine

- **ADR 0025 (compose_complexity attribute).** ADR 0025 established that a Data Element carries an admin-owned attribute (`compose_complexity`) that affects Composer rendering. That ADR was the first formal recognition that elements carry *more than just schema*. ADR 0038 generalises the pattern: schema is one layer, complexity is a separate (and orthogonal) governance attribute, validation rules are a third layer, routing rules are a fourth concern. The "elements have governance attributes beyond JSON Schema" precedent is already set.
- **ADR 0026 (snapshot immutability).** Element versions are immutable per ADR 0026. The three-layer model respects that: Schema is per-version (snapshotted), Validation rules are per-version (snapshotted with the element), Routing rules are cross-version (per-DEX, not snapshotted). Existing Agreements continue to evaluate against the Schema + Validation rules they snapshotted; routing rules apply at orchestration time per the orchestrator's then-current state. Snapshot semantics are preserved.
- **ADR 0034 (functional prototype).** The Rules tab in the registration prototype evaluates rules live against the Smart Start sample payload — Sarah types a rule, sees pass/fail against a real payload. This requires shipping a govaluate equivalent in JS for the prototype (or faking it; per [ADR 0039 §Q10] the prototype uses a real evaluator for the deterministic case). ADR 0038's three-layer model is what makes that demo possible — Rules are first-class enough to deserve their own tab with their own runtime preview.
- **Aligns with the be/ runtime, not against it.** The dex-monorepo already runs AJV + govaluate + orchestrator as three separate systems. The portal vocabulary that ADR 0038 introduces matches what the backend already does — no new naming for engineers to learn, no portal-only abstraction that obscures runtime behaviour.

## Considered options that were rejected

**All-in-JSON-Schema** — express cross-field rules via JSON Schema's `if/then/else`, `dependencies`, `dependentRequired`, and value-comparison via custom keywords (`x-comparison`). Rejected: (a) JSON Schema cannot cleanly express value comparison (`end_date > start_date`) without custom keywords that no validator supports out-of-the-box; (b) the govaluate engine already runs in production for exactly this, so JSON Schema's awkward expression would duplicate runtime work; (c) admins would still need to learn an awkward JSON Schema dialect rather than the govaluate syntax that ops already write. Strictly weaker on both authoring and runtime axes.

**All-in-rules-DSL** — express shape, behaviour, and routing in one rules language (govaluate or similar). Rejected: AJV + RJSF is the working form-rendering stack and replacing it would break the Composer's per-keystroke feedback loop and rebuild a production-grade form renderer from scratch. The "one DSL to rule them all" instinct loses to the reality that AJV is good at shape and govaluate is good at behaviour; using both lets each do what it does well.

**Two-layer model: Schema + Rules** (folding routing into "Rules with `applies_at: routing`"). Rejected: surfaces fine in copy but creates two scope mismatches in the UX. Routing rules are cross-version and per-DEX; Schema/Validation rules are per-version. Routing rules are authored against JSON-paths and orchestrator predicates; Validation rules are authored against field names and operator-submitted payloads. Forcing both into one "Rules" tab requires the admin to mentally context-switch on `applies_at` for every rule, which is exactly the kind of conflation the three-layer model exists to prevent. The current ADR keeps Routing as a separate visible placeholder so the eventual phase-2 authoring surface can have its own mental model.

**Routing in the registration flow today** — surface a real Routing-rules authoring UI in phase 1, not just a placeholder. Rejected for scope per [ADR 0039 §Q10] and because Routing rules need their own grilling against ADR 0028 (routing-is-not-an-Agreement-property) before they're surfaced in any portal flow. The placeholder honestly names that the layer exists and is governed elsewhere today.

**Hide the layered model and call it "schema editor"** — present the registration UI as a single "schema editor" surface with everything inside (fields + cross-field rules + routing) presented as collapsible sections of one big form. Rejected: this is the YAML mindset, the very thing the new portal is replacing. The legacy relay race exists *because* admins thought of all governance as "the schema." Naming the three layers explicitly in the UI is how the new product teaches the new mental model — and once an admin learns that Schema, Rules, and Routing are different things with different evaluators, they think correctly about every subsequent element they author.
