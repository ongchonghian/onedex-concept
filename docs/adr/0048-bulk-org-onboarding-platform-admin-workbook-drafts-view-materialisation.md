# Bulk org onboarding via Platform-Admin workbook with Drafts-view first-login materialisation

> **Status:** draft (2026-05-29; amended 2026-05-30 — see *Amendments* below)
> Builds on [ADR 0015](./0015-onboarding-via-design-discipline-not-tours.md) (the no-tours / empty-state-driven onboarding discipline this ADR honours by *not* building a dedicated landing surface), [ADR 0017](./0017-migration-onboarding-panel-redirects-and-draft-migration.md) (the "Automated draft migration" precedent — staged work materialises into the user's standard Drafts view at first login, no special route, inline panel for one-time framing), [ADR 0042](./0042-element-spec-sheet-onramp-deterministic-parser-audit-log-provenance.md) (the deterministic-parser + per-row authoring sidecar + strip-at-publish + parallel-audit-log doctrine this ADR's workbook ingress reuses, with the post-cutover noun `authoringMetadata` per ADR 0046), [ADR 0041](./0041-smart-start-refit-mid-edit-structural-restatement.md) (the three-way merge contract the staging-side refit reuses on Platform-Admin re-uploads, with Platform-Admin-wins-on-conflict instead of Sarah-wins), [ADR 0014](./0014-counterparty-picker-hybrid-with-enrolment-signal.md) (the counterparty resolution model + enrolment signal the staging parser must honour at parse time), [ADR 0026](./0026-agreement-snapshot-immutability-schema-upgrade.md) (the element-version snapshot rule the staging-to-materialise window must navigate), [ADR 0007](./0007-agreement-lifecycle-state-machine.md) (the Pending → Active → Ended lifecycle the materialised Agreements join unchanged), [ADR 0029](./0029-user-org-affiliation-as-n-to-m-with-embedded-dex-roles.md) (the n-to-m org-affiliation model the atomic materialise creates), [ADR 0010](./0010-lifecycle-reminder-pattern-not-framework.md) (the reminder cadence pattern stalled-onboarding falls back to), and [ADR 0046](./0046-data-element-bundle-export-to-admin-corev2-registry.md) (the `authoringMetadata` noun and the strip-at-publish-boundary doctrine, now applied at the stage-to-materialise boundary rather than the publish boundary). Closes the open question implicit since ADR 0017 named the draft-migration pattern: *how does a brand-new org's onboarding leverage the same primitive — pre-populated work appearing in the user's Drafts view at first login — when the source isn't a legacy portal but a Platform Admin authoring on the prospect's behalf?*

## Context

CONTEXT.md's vocabulary frames a clean steady-state: an operator logs in, picks a counterparty, chooses an element, fills the Agreement wizard, submits. The doctrine in [ADR 0015](./0015-onboarding-via-design-discipline-not-tours.md) settles the brand-new-user case for *empty* state — empty-state Inbox does the work, no tour, no checklist. What neither addresses is the **populated-but-not-yet-claimed** case: a prospect org engaging with SGTradex before having any account at all, going through offline due diligence (KYC documents, jurisdiction checks, signatory verification), and emerging with a real onboarding intent that involves N specific Agreements with M specific counterparties. Today the Platform Admin (`Super SGTradex Admin` / `SGTradex Admin` per [ADR 0029](./0029-user-org-affiliation-as-n-to-m-with-embedded-dex-roles.md)) completes KYC, creates the `organization` record, provisions one or more `pitstop` rows, invites the org admin user — and the org admin lands on an empty Inbox per ADR 0015, then authors every Agreement from scratch. The work the Platform Admin gathered during due diligence (which counterparties, which elements, what terms) lives in email threads and spreadsheets, lost to the operator.

Two structural realities shape this ADR's design choices:

(1) **The activation gap is asymmetric.** The Platform Admin is going to do the due diligence work either way — the question is whether their gathered context survives into the operator's first session, or evaporates. Capturing it in a structured artefact is a small additional cost for the Platform Admin and a peak-end first-impression win for the operator (whose mental model of the platform forms in the first 10 minutes post-invite per the Doherty-threshold and peak-end heuristics). The lever is *not* "reduce Platform Admin labour" — it's "convert Platform Admin labour from invisible to consequential."

(2) **The pattern precedent already exists.** [ADR 0017](./0017-migration-onboarding-panel-redirects-and-draft-migration.md) established "Automated draft migration" — pre-populated Drafts appearing in the user's standard Drafts view at first login, framed by a one-time inline panel, no special route. That pattern was built for users migrating from legacy `admin-ui` / `pitstop-ui`; its shape is identical to what's needed here. The ingress changes (Platform Admin workbook upload vs. legacy-portal scrape), and the trigger changes (org-creation moment vs. first-login post-cutover), but the user-facing primitive is the same.

A grilling session walked the decision from "is logging into the app the only onboarding channel?" through "what affects what?" to "in-Inbox vs dedicated URL?" through "atomic-across-batch vs forward-references for cross-onboarding counterparties." The decisions below close that walk.

## Decision

A new platform channel ships: the **Org onboarding workbook** — a Platform-Admin-authored XLSX/CSV artefact that pre-stages an org's onboarding shape (Org profile, Pitstops, initial Users, Direct Agreements, Service-Provider Agreements) during the offline KYC window, commits atomically into real platform records when the Platform Admin approves KYC, and surfaces to the new org admin as a populated Drafts view enriched with a one-time onboarding shell at first login. ## Amendments — 2026-05-30

Grilling against the existing portal's Participants surface, `+ Invite` modal, sign-up flow ([`admin-ui/src/pages/signup/sign-up.jsx`](../../../admin-ui/src/pages/signup/sign-up.jsx)), and `data-exchange.jsx` invite-integration uncovered five corrections the initial ADR didn't honour. The sub-decisions below have been edited in-place; this section is the single index of what changed and why.

1. **`Materialise` retires as a verb; `KYC approval` replaces it.** The original verb was engineer-flavoured. The KYC verdict is the gate; the cascade (Pitstop, User, Draft creation) is *contingent* on KYC approval per the existing prototype's lifecycle. Sub-decision 7 amended. Workbook screen CTA reads *Approve KYC and onboard* / *Stage as Pending KYC* / *Reject KYC*.
2. **Workbook now supports three verdicts** — Approved (cascade fires), Pending (Org row only, contingent records deferred until later KYC clearance), Rejected (Org row only in `kyc-rejected` status, no cascade). Sub-decision 7 + new vocabulary in CONTEXT.md.
3. **Discoverability moves from sidebar entries to Participants header CTAs.** The existing portal's `+ Invite` button on Participants is the canonical org-admin entry point for *peer-invite*; the workbook upload is its platform-admin sibling — *+ Onboard new org* — on the same Participants header, role-gated by `canManageNetworks`. The retired sidebar items (*Org onboarding*, *Rejected onboardings*, *Declined*) leave the IA. Sub-decision 1 amended; new sub-decision pattern follows [ADR 0039 §2](./0039-data-element-registration-admin-authored-single-page-flow.md)'s catalogue-CTA discovery.
4. **`Publish` retires as the operator's CTA verb; `Send` replaces it.** Per [ADR 0007](./0007-agreement-lifecycle-state-machine.md), an operator-initiated commit creates a *Pending* Agreement that the counterparty must Accept to make Active. Calling that act *Publish* implied a terminal state and misled the demo. The end-state card prose updates accordingly — *N Agreements sent — they'll go live as counterparties accept them*, not *N Agreements active*. Sub-decisions 8, 10, 16, 17 amended.
5. **Rejected onboardings retire as a parallel staging archive; `kyc-rejected` becomes a real `ORG_DEX_MEMBERSHIP.status`.** The existing model already keeps Pending KYC orgs as full `ORGS` rows (Pacific Container Lines on the Participants directory). Rejected orgs should follow the same pattern — a real `ORGS` row with `kyc-rejected` status, hidden from the default Participants filter behind a *Show inactive* toggle (platform-admin-only). The user-facing chip says *Onboarding deferred* for diplomatic reasons. Sub-decision 13 amended; `workspace.rejectedOnboardings` retires. New CONTEXT.md term `kyc-rejected`.
6. **`Decline` (operator-side, Draft level) retires the sidebar entry; becomes a status filter on Drafts.** Pattern-consistent with the *Rejected onboardings → Participants filter* move. Verbs stay asymmetric: *Reject KYC* (Org-level platform-admin decision) vs *Decline* (Draft-level operator decision) — distinct concepts, distinct scopes. Sub-decision 11 amended.
7. **Evidence + reason capture on the workbook decision UI.** The Platform Admin uploads an artefact of the offline KYC agreement (required for Approved + Rejected, optional for Pending); writes an *Internal reason* (audit-only) + optional *Org-facing message* (only delivered if the *Send notification* toggle is on). The audit needs honesty; the org-facing needs diplomacy; mixing them compromises both. New sub-decision 18.
8. **Re-engagement of a `kyc-rejected` org** is a fresh workbook upload that updates the existing row's status (not a new row, not a special *Reconsider* affordance). The audit log carries the rejection event; the Participants detail surface shows a *Previously declined* history link. Captured in the amended sub-decision 13.
9. **New capability: `canInviteParticipant`** — coined to gate the existing Invite modal (currently ungated, an existing leak that surfaces it for Etr/Operation/Tech Users). Granted to `Super Admin` + `Admin User` only. The `canManageNetworks`-gated *+ Onboard new org* CTA is mutually exclusive at the persona level — platform-tier roles never see *+ Invite participant*; org-tier admins never see *+ Onboard new org*.



**1. The channel is Platform-Admin-only, bulk-only, internal.** Capability gated by `canManageNetworks` (per the existing ROLE_CAPABILITIES — Super SGTradex Admin + SGTradex Admin). No self-serve channel, no public template marketplace. The submitter is a trusted internal actor; abuse / spam / unverified-identity vectors don't apply. **Discoverability** is a header CTA on the Participants directory page — *+ Onboard new org* — mirroring [ADR 0039 §2](./0039-data-element-registration-admin-authored-single-page-flow.md)'s catalogue-CTA pattern (*+ New element* / *+ New version* on Data elements). Sits next to the existing peer-invite CTA *+ Invite participant* (which is gated by the new `canInviteParticipant` capability for org-tier admins); the two CTAs are mutually exclusive at persona level — platform-tier never sees Invite, org-tier never sees Onboard. **Rejected**: a dedicated sidebar entry (would resurrect what [ADR 0017](./0017-migration-onboarding-panel-redirects-and-draft-migration.md) retired via the SIDEBAR_ITEMS_PLATFORM removal); self-serve pre-KYC upload by prospects themselves (abuse vector + no audit chain to the submitter).

**2. The artefact is an Org onboarding workbook with five named sheets.** One XLSX file carries the entire onboarding shape:

| Sheet | Defines | Row cardinality |
|---|---|---|
| `Org` | Profile: UEN, legal name, jurisdiction, primary DEX, business address, contact | exactly 1 row |
| `Pitstops` | Endpoint configurations the org will operate | N rows (≥1) |
| `Users` | Initial users — first row is the org admin (invitee); subsequent rows are additional operators | N rows (≥1) |
| `Agreements (Direct)` | Direct Agreements per CONTEXT.md *Direct Agreement* | N rows (≥0) |
| `Agreements (SP)` | Service-Provider Agreements per CONTEXT.md *Service-Provider Agreement* | N rows (≥0) |

Two Agreement sheets, not one, because the row shapes differ materially (a Direct Agreement has counterparty + role; an SP Agreement has data-owner + service-provider + counterparty + flow-direction). One sheet trying to encode both via conditional columns invites parse-time confusion and obscures the discriminator (which per CONTEXT must never surface as a user-facing noun but is *load-bearing* at parse time). The workbook is a single file because the unit of onboarding is one org; multi-org per workbook is rejected on the same grounds ADR 0042 sub-decision 3 rejected multi-element batch — pack membership / batch grouping is an editorial decision the parser should not infer. The consortium-batch case (sub-decision 14) is handled by multiple workbooks coordinated by the Platform Admin, not by encoding multiple orgs in one file.

**3. The parser is deterministic-only; all inference deferred.** Each row maps to exactly one staged record via a fixed column-to-property mapping. No NL parsing, no VLM extraction, no enrolment-fuzzy-match. Unresolved references (an unknown UEN in a counterparty column, an unknown `element_id` in an Agreement row) produce **parse-time warnings** that block the upload from advancing to *stage* until resolved by the Platform Admin in the upload UI. This mirrors ADR 0042 sub-decision 4 (*"all inference lives in Smart Start assist / refit"*) — the parser surfaces signals, never invents data, and the resolution affordance is a human gesture. Rejected: parse-time fuzzy counterparty resolution (re-creates the two-engine problem ADR 0040 §17 warned against; resolution must be one engine acting once, not the parser guessing and the materialiser correcting).

**4. Per-row `authoringMetadata` stamped on each staged record.** The parser writes one `authoringMetadata` object per staged row — `{ kind: 'org-onboarding-workbook', file, fileHash, sheet, row, headerRow, stagedBy, stagedAt }`. Carries the same role `x-source` carried pre-cutover in ADR 0042: provenance for downstream refit-diff, audit-log feed at materialisation, debugging seam. Lives on the staged record only — **stripped when the record materialises into a real `consent_agreement` / `organization` / `pitstop` row** per the post-[ADR 0046](./0046-data-element-bundle-export-to-admin-corev2-registry.md) *strip-at-publish-boundary* doctrine, with the same provenance recorded on the corresponding audit-log event (`org_created`, `agreement_created`, etc.). The staging boundary is the analogue of ADR 0042's publication boundary; the principle transfers cleanly.

**5. Snapshot-at-stage with reconcile-at-materialise (Platform-Admin-side, never operator-side).** Per [ADR 0026](./0026-agreement-snapshot-immutability-schema-upgrade.md), Agreement creation snapshots element IDs. A staged Agreement at T1 references `bunker-invoice@1.0.0`; Element version `1.1.0` publishes at T2 (mid-staging-window); KYC completes at T3. **Three-step contract**: (a) snapshot the version named in the workbook at stage time, recorded on the staged record; (b) at the materialise gate (just before atomic org-creation), surface a *Version reconcile* affordance to the Platform Admin listing every row whose snapshotted version has moved, with per-row accept-current / keep-snapshotted decisions; (c) on Platform Admin sign-off, materialise with the operator-confirmed version pins. The operator must never see a broken or version-drifted Draft — that breaks the peak-end first impression beyond repair. The reconcile is the Platform Admin's responsibility, not the operator's. Mirrors ADR 0041's refit pattern but on a different axis (cross-window rather than cross-version-of-same-element).

**6. Counterparty resolution at stage time, not at materialise.** Per [ADR 0014](./0014-counterparty-picker-hybrid-with-enrolment-signal.md), the picker's enrolment signal is the platform's authoritative source of "is this counterparty real and reachable today?" The workbook's counterparty columns resolve against this signal *at parse time*. Three outcomes: (a) **resolved + enrolled** — the row stages cleanly; (b) **resolved + not enrolled** — the row stages with a `pending-counterparty` flag; the materialised Agreement renders in the operator's Drafts view in a *Waiting on counterpart onboarding* group (sub-decision 9), disabled for publish, with an honest UX label; (c) **unresolved** — parse-time warning; Platform Admin must reconcile before staging completes. The "Waiting on counterpart" case is *not* a graceful fallback — it is an honest state that surfaces the cross-organisation dependency to the operator, who deserves to know which Agreements are blocked on which other onboardings.

**7. KYC is the gate; the contingent cascade fires only on Approve. Three verdicts, three outcomes.** Platform Admin completes offline KYC, runs the version reconcile per sub-decision 5, and commits a verdict from the workbook preview screen. The verdict picker is a three-way radio with explicit copy on each option; the primary CTA below adapts:

   | Verdict | CTA copy | What writes |
   |---|---|---|
   | **Approved** | *Approve KYC and onboard* | One transaction: `ORGS` row (status=active), N `pitstop` rows, M `user_org_affiliation` rows per [ADR 0029](./0029-user-org-affiliation-as-n-to-m-with-embedded-dex-roles.md), K `agreement_draft` rows per [ADR 0007](./0007-agreement-lifecycle-state-machine.md). Welcome email goes to the org admin invitee *after* commit (per the existing portal's *"welcome email is sent when KYC is approved and the agreements have been created by the platform admin"* contract). |
   | **Pending** | *Stage as Pending KYC* | Only the `ORGS` row (status=pending), no Pitstops, no Users, no Drafts. The Org appears in Participants with the existing *Pending KYC* chip (same shape as the existing peer-invite path). Welcome email is **deferred** — sent later when Platform Admin revisits and approves. |
   | **Rejected** | *Reject KYC* | Only the `ORGS` row with `ORG_DEX_MEMBERSHIP.status = 'kyc-rejected'`. User-facing chip *Onboarding deferred* per CONTEXT.md. Optional notification email if Platform Admin ticks *Send notification* on the decision UI (sub-decision 18). |

   The cascade is atomic when it fires — either everything writes or nothing does. Per-row writes during cascade and async cascade are both rejected: the operator must never land on a partial onboarding (peak-end ruined by missing pieces). The verb *Materialise* retires entirely from the ADR vocabulary — the gate semantic is *KYC approval*, not *materialisation*.

**8. First-login lands on the standard Drafts view enriched with a one-time onboarding shell — not a dedicated URL.** Welcome email links to `/portal/<dex>/drafts?onboarding=1` (deep-link query param ensuring the shell renders even if the operator has visited Drafts before — rare on first login but possible if they auth via a different path). The shell renders three enrichments on top of the standard Drafts view: (a) a one-time inline welcome panel mirroring [ADR 0017](./0017-migration-onboarding-panel-redirects-and-draft-migration.md)'s *"What's changed"* discipline (auto-dismiss on engage, not on timer, with set-up-by attribution); (b) a goal-gradient progress bar showing *N of M ready to send* across all `from-onboarding` Drafts; (c) the standard Drafts view's *group by counterparty* mode defaulted on. **The action verb is *Send*, not *Publish*** — per [ADR 0007](./0007-agreement-lifecycle-state-machine.md), the operator-initiated commit produces a *Pending* Agreement that the counterparty Accepts to make Active; calling that act *Publish* implied a terminal state and is wrong. The shell disappears once all onboarding Drafts have been resolved (sent or declined) plus a 7-day grace, after which the Drafts view reverts to its everyday rendering. **No dedicated `/onboarding` URL** — the rejected pattern is documented in *Considered alternatives* below. The mechanism for the shell's lifecycle is the staged-from-onboarding marker on each Draft, not a separate UI mode flag.

**9. Drafts grouped by counterparty with a *Waiting on counterpart* sub-group.** Per the proximity-grouping principle from [ADR 0035](./0035-inbox-three-axis-taxonomy.md)'s sibling discussions and Gestalt UX-proximity-grouping doctrine, the populated Drafts view groups rows by counterparty (each group headed by counterparty name + enrolment chip per [ADR 0014](./0014-counterparty-picker-hybrid-with-enrolment-signal.md)). A trailing group titled *Waiting on counterpart onboarding* holds rows whose counterparty was staged with `pending-counterparty` per sub-decision 6. Rows in this group render Publish-disabled with a tooltip naming the blocking counterparty; resolution happens when that counterparty's own onboarding completes (the cross-onboarding dependency mechanic is part of the deferred consortium-batch grilling — sub-decision 14). The group is visible but not actionable; the operator sees their full picture without being misled into thinking blocked work is their problem.

**10. Hybrid Publish UX: per-row review for the first 3-4, then graduated bulk affordance.** The first 3-4 Drafts in the queue surface only the per-row *Review* CTA — the operator must open each, see the standard Agreement detail page per [ADR 0019](./0019-agreement-detail-page-hero-scroll-with-sticky-header.md) in Draft mode, click Publish individually. After 3-4 successful publishes (configurable threshold), a graduated-trust affordance surfaces at the bottom of the queue: *"Publish the remaining N like the ones above?"* — opening a compressed batch summary with key terms, then a single Publish-all action. The discipline: **per-row review for IKEA-effect ownership-transfer, bulk affordance for momentum once trust is earned.** Rejected: bulk-from-row-1 (loses ownership-transfer; the operator never engages individually enough to feel ownership) and per-row-only-throughout (tedium at row 8+, drops completion rates).

**11. Decline handling: declined rows move to a status filter on Drafts (not a sidebar entry); Platform Admin gets a per-org rollup; placement is in the secondary-actions cluster, never adjacent to Send.** If the operator opens a staged Agreement and decides it shouldn't ship, they pick *Decline* — surfaced in the Agreement detail page's **secondary-actions cluster** (alongside *Save Draft* / *Delete Draft* per [ADR 0019](./0019-agreement-detail-page-hero-scroll-with-sticky-header.md)'s existing pattern), **never as a sibling button to Send**. The placement honours `ux-hicks-minimize-choices` and avoids the misclick risk of a destructive action next to a constructive one. *Decline* is **conditional** — surfaces only on `from-onboarding` Drafts, hidden on operator-authored Drafts where *Delete Draft* is the analogous action. Clicking *Decline* opens an inline confirmation requesting an optional reason; the row gains a `declined` outcome flag and **disappears from the default Drafts view**. Declined rows remain inspectable via a *Show declined* status filter on Drafts (hidden by default; toggle reveals — pattern-consistent with the *Show inactive* toggle on Participants per sub-decision 13). **A dedicated *Declined* sidebar entry is explicitly rejected** — see *Considered alternatives* — because lifecycle states should attach to existing surfaces as filters, not bloat the sidebar IA. Platform Admin sees an admin-side per-org rollup (*"Org X declined 3 of 12 staged Agreements"*) on whatever surface they monitor onboarding outcomes — never per-row notifications. Pattern-consistent with [ADR 0035](./0035-inbox-three-axis-taxonomy.md)'s *inbox-as-action-queue* doctrine: declines aren't actions for Platform Admin to take, they're signal for them to inspect later. The signal's value is calibrating KYC briefing quality (*"why did this org decline 3? did we mis-brief them?"*), not creating Platform Admin work. **Vocabulary discipline**: *Decline* is the operator's Draft-level act; *Reject KYC* is the Platform Admin's Org-level act — asymmetric verbs for distinct concepts at distinct scopes (different actors, different objects).

**12. Lifecycle-reminder cadence per ADR 0010 if onboarding stalls.** If the operator publishes some but not all staged Drafts and leaves, the [ADR 0010](./0010-lifecycle-reminder-pattern-not-framework.md) lifecycle-reminder pattern applies with a custom cadence: Day 3 (Inbox-only nudge), Day 7 (+ email), Day 14 (+ portal banner), Day 28 (+ named-consequence message — *"Counterparty X has been waiting 4 weeks"*). The escalation, multi-channel ramp, and named-consequence affordances follow ADR 0010's required pattern. The cadence is independent per onboarding-batch; it stops when the queue is fully resolved.

**13. KYC rejection writes a real Org row in `kyc-rejected` status; hidden by default; housekeeping after 90 days; audit log persists forever.** A KYC-rejected prospect is still an Org in the model — same shape as a Pending KYC org (which the existing portal already keeps as a real `ORGS` row, e.g. Pacific Container Lines visible in Participants). The Reject KYC verdict writes:

   1. **`ORGS` row** with the org's identifying details (UEN, legal name, jurisdiction) from the parsed workbook.
   2. **`ORG_DEX_MEMBERSHIP` row** with `status = 'kyc-rejected'` (a new value in the existing enum — see CONTEXT.md).
   3. **`org_kyc_event` audit row** — `{ orgUen, orgName, verdict, decidedBy, decidedAt, evidenceFileHash, internalReason, orgFacingMessage, sentNotification }`. **Persists forever**, survives housekeeping; queryable from a future audit surface.

   **No contingent records** — no Pitstops, no Users, no Agreement drafts. The org never reaches the platform's operational surface; the row exists for directory honesty + audit + future re-engagement.

   **Visibility**: the user-facing chip on the Participants directory says *Onboarding deferred* (diplomatic vocabulary; the internal state name `kyc-rejected` is technical). The chip is **hidden from the default Participants filter** behind a *Show inactive* toggle that's gated to `canManageNetworks` — only Platform Admin sees the toggle, only Platform Admin can reveal `kyc-rejected` orgs. Org-tier admins (Marcus) never see Sarah's KYC rejections.

   **Housekeeping**: after 90 days (configurable per platform policy) in `kyc-rejected` status, the `ORG_DEX_MEMBERSHIP` row is hard-deleted from the live workspace; the `ORGS` row drops if no other DEXes reference it. The `org_kyc_event` audit entry stays. Live-UI cleanup is automatic via the filter; the deletion is a separate timestamped housekeeping job. **The 90 days is a configurable platform policy, not a hard ADR commitment** — different DEXes (TX/BX/HX) may calibrate based on local compliance.

   **Re-engagement** is a fresh workbook upload that updates the existing `ORG_DEX_MEMBERSHIP` row's status (`kyc-rejected` → `pending` or `active`). Not a new row, not a special *Reconsider* affordance — same UEN, same identity, status flips. The audit log carries the rejection event as part of the org's history; the Participants detail surface for that org shows a *Previously declined — see history* link surfacing the `org_kyc_event` chain. **Rejected**: a parallel `workspace.rejectedOnboardings` shadow table (would fragment the data model from the existing Pending-KYC-orgs-in-`ORGS` pattern); hard-purge of the audit event (loses durable provenance); indefinite retention of the live row (storage drift, compliance review burden).

**14. Refit on the staging side: Platform-Admin-wins on conflict.** Between initial stage and KYC completion, the prospect's situation may change — they want to add a Pitstop, drop an Agreement, change a counterparty. The Platform Admin can either edit the staged batch in the UI directly or re-upload an updated workbook. Re-upload runs a three-way merge mirroring [ADR 0042 sub-decision 7](./0042-element-spec-sheet-onramp-deterministic-parser-audit-log-provenance.md) — L0 = initial staged batch, L1 = Platform Admin's UI edits, L2 = new upload — with **Platform-Admin-wins on conflict** rather than Sarah-wins. The L0/L1/L2 grid from ADR 0042 §7 applies verbatim with `Sarah` substituted for `Platform Admin`. The discipline preserves the principle (re-import never silently overwrites a deliberate edit) on a different axis.

**15. Cross-onboarding counterparties resolve via forward-references; no consortium-batch coordination layer.** When multiple prospect orgs are being onboarded in parallel and their staged Agreements reference each other (the "Maritime Singapore Hub joins SGTradex with 8 of its members" pattern), each workbook materialises independently as its own org's KYC clears — there is no batch grouping that holds materialisation until all referenced orgs are KYC-clear. Rows whose counterparty is another staged-but-not-yet-materialised org carry the `pending-counterparty` flag per sub-decision 6 and render in the *Waiting on counterpart onboarding* group per sub-decision 9; when the referenced org subsequently materialises, the flag clears automatically and the row becomes Publish-enabled. Three resolution moments matter:

   (a) **Counterpart materialises (happy path).** The dangling reference resolves automatically. The waiting operator receives a soft notification — Inbox card with `intent=review`, optional email per their notification preferences, no banner — naming the now-onboarded counterparty and the Agreement that's now Publish-ready. The notification follows [ADR 0010](./0010-lifecycle-reminder-pattern-not-framework.md)'s lifecycle-reminder channel-ramp doctrine at its lowest intensity, since this is a positive event surfacing newly-available work rather than a stalled-action escalation.

   (b) **Counterpart KYC fails.** Per sub-decision 13's soft-purge, the failed prospect's staged batch moves to *Rejected onboardings*. The dangling references in other orgs' Drafts transition from *Waiting on counterpart onboarding* to a new terminal label *Counterpart onboarding ended*. The row becomes declinable but not publishable; the operator decides whether to decline (per sub-decision 11's Decline view, which records the *Counterpart onboarding ended* reason in audit) or hold open in case the counterparty re-engages later.

   (c) **Counterpart never materialises (timeout).** If the referenced org's staged batch sits in *Staged* (KYC neither cleared nor rejected) for more than the 90-day audit-retention window from sub-decision 13, the dangling references are surfaced to the Platform Admin for cleanup via the admin-side report. The operator-side row enters the same *Counterpart onboarding ended* terminal label.

   Forward-references' peak-end first-impression is weaker than atomic-across-batch's on consortium-shaped onboardings — operators may land on a partial-resolution Drafts queue with multiple *Waiting on counterpart* rows. The trade is accepted because (i) consortium-shaped onboardings are not the dominant pattern, (ii) atomic-across-batch holds fast-KYC orgs hostage to slow-KYC ones in the same consortium, and (iii) the `pending-counterparty` flag is the seam where a future atomic-across-batch layer could attach without disturbing this ADR's contract — the policy can be revisited if consortium frequency grows materially.

**16. Send-click feedback sequence: row exit + toast + progress update, sequenced for clarity.** Clicking Send on a `from-onboarding` Draft triggers a four-part sequence — explicitly **not simultaneous** — so the operator's eye lands on one focal point at a time per `staging-one-focal-point`:

   (a) **Row exit animation.** The sent row leaves the Drafts list using `ease-in` easing over ~220ms (within the `timing-under-300ms` band; honours the `easing-exit-ease-in` doctrine for entries leaving a surface). Height collapses to zero; opacity fades. The next-up row slides up to fill the vacated space using the same curve. No spring physics — the row's exit is a deterministic state change, not a gesture-driven motion per the `spring-for-gestures` / `easing-for-state-change` distinction.

   (b) **Toast notification.** A bottom-right toast appears at frame 0 of (a): *"Sent to {CounterpartyName}. They'll review and accept."* The counterparty is **named verbatim** per `ux-von-restorff-emphasis` — the named entity makes the act feel consequential, not abstract. The *review and accept* copy honours [ADR 0007](./0007-agreement-lifecycle-state-machine.md)'s Pending → Active lifecycle — the Agreement is Pending until the counterparty Accepts, and the toast names that truthfully. Toast lifetime: 4s auto-dismiss; manual close available; **not stacked** — a second Send before the first toast dismisses replaces the prior toast with the new one (rather than queueing or piling).

   (c) **Goal-gradient progress bar update.** The filled portion grows from `(N−1)/M` to `N/M` using **linear** easing per `easing-linear-only-progress` — the bar represents *proportional progress*, and only linear honours that contract; ease-in/out would dramatise the increment in a way that misrepresents the cumulative arithmetic. Duration ~400ms, **starting after (a) completes** so the eye registers the increment after the row has cleared (single focal point per `staging-one-focal-point`).

   (d) **Cursor / focus continuity.** The next-up Draft row's *Review →* button retains keyboard focus; the operator can tab/space through the queue without re-anchoring. Scroll position holds — the new top-of-list row occupies the same Y-position the just-sent row occupied. This honours `none-keyboard-navigation`'s principle that keyboard-driven traversal must feel instant.

   **Reduced-motion fallback.** `prefers-reduced-motion: reduce` collapses (a) and (c) to instant snaps; (b) still appears (toast is information, not motion) per `a11y-visual-equivalent`. No sound is emitted on Send — per `appropriate-no-decorative` plus the broader portal discipline of silent UI.

**17. End-state moment when the last `from-onboarding` Draft resolves: the *Your organisation is set up* card, then graceful retirement of the onboarding shell.** The transition from "N Drafts remaining" to "0 Drafts remaining" is not silent. Immediately after the final Send (or final Decline) of a `from-onboarding` Draft, the Drafts view renders a transient **end-state card** in place of the now-empty queue:

   > **Your organisation is set up.**
   > *N Agreements sent — they'll go live as your counterparties accept them, usually within a day. Data starts flowing as soon as each Agreement is accepted and your Pitstop is connected.*
   >
   > **What's next**
   > - Configure your Pitstop endpoint → [link to Settings → Pitstops]
   > - Invite teammates → [link to Settings → Users]
   > - Set up Watch alerts on critical Agreements → [link to the Agreement-list with Watch filter]
   > - Browse the data element catalogue → [link]

   (a) **Counts in the lede are computed at render time.** *N Agreements sent* sums Sent `from-onboarding` rows; *M counterparties* counts distinct counterparties across them. Declined rows are excluded from both. **All-declined edge case**: if the operator declined every staged Draft, the lede falls back to *"Your organisation is set up. No Agreements were sent from onboarding — start fresh below."* with a *Create Agreement →* CTA. The card honours `ux-peak-end-finish-strong` either way — the *end* is always a clear success state, even when the operator's decisions reshaped what success means. The heading deliberately says *set up*, not *live* — *live* would be inaccurate since the Agreements are Pending until the counterparties Accept per [ADR 0007](./0007-agreement-lifecycle-state-machine.md).

   (b) **The *What's next* card is the contextual progressive-disclosure surface.** Per `ux-progressive-disclosure`, surfacing next-level affordances at this moment is appropriate (not paternalistic) because the operator has demonstrated by publishing (or deliberately declining) N Drafts that they understand the model. The four affordances are ordered by **likely-next-need** (Pitstop config first because data can't flow without it; teammates second because most orgs scale beyond one operator; Watch alerts third because they require a published Agreement to anchor to; catalogue browse last because it's exploration, not action) — **not** alphabetical, not feature-prominence-driven. Order is task-flow-driven per `ux-serial-position` (the most-load-bearing item first).

   (c) **Lifecycle of the end-state card.** Persists across a single visit (so an operator who navigates away and back still sees it). On visit 2, dismisses; thereafter the Drafts view shows the standard empty-state per [ADR 0015](./0015-onboarding-via-design-discipline-not-tours.md). Explicit *Got it* affordance dismisses immediately for operators who want to move on. **No timer-based dismiss** — the moment ends when the operator says it ends, not when the clock says it does.

   (d) **Sub-decision 8's shell-retirement clause is amended.** The welcome panel + goal-gradient bar + group-by-counterparty enrichments retire at the moment the end-state card renders; the end-state card itself retires per (c). The 7-day grace from §8 is repurposed: it is now the period during which the Drafts navigation entry retains a discreet *Recently onboarded* annotation (a subtle marker, not a chrome enrichment) so operators reviewing what they did during onboarding can find their sent Agreements via the standard Drafts → Sent filter.

**18. Evidence + dual-reason capture on the workbook decision UI.** The Platform Admin's verdict per sub-decision 7 isn't a bare click — the decision UI requires (and records) artefacts that make the verdict honest in the audit chain and diplomatic in the org-facing communication.

   | Field | Required for verdict | Audience | Audit |
   |---|---|---|---|
   | **KYC evidence file** (PDF/XLSX/image upload, hashed via SubtleCrypto) | Approved + Rejected (**required**); Pending (optional — KYC may still be in progress) | Internal compliance | Hash + filename recorded on `org_kyc_event`; file storage out of scope of this ADR |
   | **Internal reason** (free-text textarea) | All three verdicts (optional but encouraged) | Internal audit only; never sent to the prospect | Verbatim on `org_kyc_event.internalReason` |
   | **Org-facing message** (free-text textarea) | Optional always | The prospect, only if *Send notification* is checked | Verbatim on `org_kyc_event.orgFacingMessage` |
   | **Send notification toggle** (checkbox, default off) | Optional always; default off | Determines whether the prospect receives an email | Recorded on `org_kyc_event.sentNotification` |

   The two-field reason split is load-bearing: **the audit needs honesty, the org-facing needs diplomacy, and mixing them compromises both.** Sanctions-list hits get an honest internal reason and no org-facing message (toggle off); soft rejections (capacity / timing) get a diplomatic org-facing message (toggle on) without exposing internal calculus. The platform's default fallback when *Send notification* is off and no org-facing message is supplied: silent — no email is sent.

   The notification copy template when *Send notification* is on and no org-facing message is supplied is a platform-standard diplomatic line: *"We're unable to complete onboarding at this time. We'd welcome the opportunity to revisit when your circumstances change. Please reach out to onboarding@sgtradex.example for next steps."* The Platform Admin can override or augment via the org-facing message field.

## Considered alternatives

- **Dedicated `/portal/<dex>/onboarding` URL** as the first-login landing surface — rejected. The grilling weighed the peak-end framing benefit (a bounded canvas with clear lifecycle) against the [ADR 0015](./0015-onboarding-via-design-discipline-not-tours.md) discipline (*"Zero formal onboarding + empty-state-driven (chosen)"*) and [ADR 0017](./0017-migration-onboarding-panel-redirects-and-draft-migration.md)'s precedent (migration users with populated Drafts land in the standard Inbox/Drafts view, not a special URL). Sustaining an ephemeral-lifecycle URL is the only such pattern in the portal IA; the maintenance cost (redirect rules, orphan-state handling, bookmark semantics) is real. The peak-end machinery survives as Drafts-view enrichments per sub-decision 8 — only the bounded-canvas property is lost, which is small relative to the discipline-coherence gain.

- **Self-serve pre-KYC channel** where prospects themselves upload workbooks or interact with a magic-link form before having an account — rejected per sub-decision 1 (abuse vector, no audit chain, semantically muddy). The brainstorm explored this branch (channel A, "counterparty-initiated invite link," and channel C, "templated deep link") and converged on Platform-Admin-only for this ADR; channel C remains viable as a separate downstream activation lever once a real account exists (the org admin clicking a one-tap "Publish all staged" deep link from the invite email is the closest analogue, satisfied by sub-decision 8's `?onboarding=1` parameter).

- **Materialise into Pending Agreements instead of Drafts** — rejected. Per CONTEXT.md *Draft*, *"A draft is NOT an Agreement"*; Drafts have no counterparty visibility, no audit chain, no obligations. Materialising directly to Pending would send unconsented notifications to counterparties on behalf of an operator who has not yet seen the Agreement. The IKEA-effect ownership-transfer in sub-decision 10 *requires* the operator to actively Publish each Draft; bypassing that step destroys the design's behavioural foundation.

- **Multi-element batch per workbook** (encoding multiple prospect orgs in one file) — rejected per sub-decision 2 + 14. Editorial grouping is a Platform Admin decision; the parser shouldn't infer it.

- **Smart Start assist on the workbook** (grounded suggestions while the Platform Admin is authoring the staged batch) — rejected for v1. The use case is well-structured: Platform Admin knows what they need from KYC documents and offline conversations; assist's value (cross-tab suggestion engine grounded in Confluence + DEX reference registry + catalogue priors) doesn't transfer cleanly to a workbook ingress where the input is already structured. Defer until production usage reveals a recurring assist-shaped need.

- **Atomic-across-batch materialisation** for consortium-shaped onboardings (hold all workbooks in *Staged* until every org in the consortium clears KYC, then materialise everything in one transaction) — rejected per sub-decision 15. The strong-consistency peak-end gain (operators land on fully-resolved Drafts with zero *Waiting on counterpart* rows) is real but applies only to consortium-shaped onboardings, which are not the dominant pattern. The infrastructure cost is significant — a new consortium-batch concept above the workbook layer, an explicit grouping affordance on the admin side, and a manual-cleanup workflow for the partial-failure case (one of N orgs fails KYC). Forward-references is the correct default; atomic-across-batch can attach as a future layer at the `pending-counterparty` seam without disturbing this ADR's contract if consortium frequency grows.

- **Sidebar entries for *Org onboarding*, *Rejected onboardings*, and *Declined*** — rejected per the 2026-05-30 amendment. The existing portal's Participants page is already the platform-wide org directory (the tip reads *"All orgs onboarded to SGTradex"*) with Pending KYC orgs visible by default; orgs-onboarding work is verb-shaped governance, and the existing pattern ([ADR 0039 §2](./0039-data-element-registration-admin-authored-single-page-flow.md)) is for verb-shaped acts to be discovered as CTAs on their catalogue surface, not as sidebar entries. Adding sidebar entries here would also resurrect what [ADR 0017](./0017-migration-onboarding-panel-redirects-and-draft-migration.md) retired (SIDEBAR_ITEMS_PLATFORM). Declined follows the same pattern at Draft scope — lifecycle states attach to existing surfaces as filters, not as sidebar peers.

- **Parallel `workspace.rejectedOnboardings` shadow table for KYC-rejected batches** — rejected per the 2026-05-30 amendment and the corrected sub-decision 13. The existing data model already keeps Pending KYC orgs as full `ORGS` rows in the Participants directory (Pacific Container Lines being the canonical example). KYC-rejected orgs should follow the same pattern — a real `ORGS` row with `kyc-rejected` status — not a parallel staging artefact. The shadow-table approach fragmented the model from the existing pattern and required a separate browse surface; the corrected design absorbs both into Participants with a status filter.

- **Symmetric *Decline* vocabulary across Org and Draft scopes** — rejected per the 2026-05-30 amendment. *Reject KYC* (Org-level platform-admin decision) and *Decline* (Draft-level operator decision) are different concepts with different actors and different objects; asymmetric verbs map real semantic difference. Forcing symmetry would conflate two distinct lifecycle moments under one word.

## Consequences

**Architectural:**

- New staged-record tables (`organization_staged`, `pitstop_staged`, `agreement_staged`, plus the user-staging analogue) keyed by an `onboarding_batch_id`. Schema-wise the staged shapes mirror their materialised counterparts plus an `authoringMetadata` JSONB column per sub-decision 4 and a `pending_counterparty_uen` nullable column per sub-decision 6.
- New parser module (e.g., `portal-app/scripts/admin-onboarding-workbook-parser.js` in the prototype; corresponding production location in admin tooling) lazily loading SheetJS — same pattern as ADR 0042's `register-onramps-spec-sheet.js`.
- Drafts view ([`portal-app/scripts/app.js`](../../portal-app/scripts/app.js) and corresponding production surface) gains the onboarding-shell rendering per sub-decision 8 — welcome panel, goal-gradient bar, group-by-counterparty default — gated on the presence of `from-onboarding`-flagged Drafts in the org's queue.
- Materialise transaction (sub-decision 7) is new infrastructure on the admin side; it owns the atomicity guarantee and the invite-email-after-commit ordering.
- A new admin-side surface for the *Rejected onboardings* archive (sub-decision 13) and the per-org *Declined Agreements* rollup (sub-decision 11). These can be report-shaped, not interactive.

**Behavioural / contractual:**

- The Platform Admin's labour during KYC is now consequential — what they capture in the workbook directly shapes the operator's first 10 minutes. KYC briefing quality becomes a feedback loop (declines per sub-decision 11 are the signal).
- The operator's first-login experience is no longer ADR 0015's *empty-state-driven*; it is *populated-Drafts-driven*, with the welcome panel and shell taking the framing role that empty-state copy plays for the non-onboarded case. ADR 0015's discipline is preserved (no tour, no checklist) — the Drafts are real work, not pedagogy.
- Counterparty enrolment-signal correctness (per [ADR 0014](./0014-counterparty-picker-hybrid-with-enrolment-signal.md)) becomes operationally load-bearing — sub-decision 6's resolution depends on it being current. Drift here surfaces as parse-time warnings (sub-decision 3) rather than silent miscategorisation.
- The element-version-snapshot rule from [ADR 0026](./0026-agreement-snapshot-immutability-schema-upgrade.md) is honoured by sub-decision 5's reconcile gate; the operator never sees the cross-window version drift.

**Forward-flexible:**

- The staged-then-materialise pattern is the model for *any* future bulk-shaped onboarding (a new DEX joining, a regulatory bulk-add, a hub-of-trade consortium). The atomic materialise transaction is reusable.
- The consortium-batch deferred decision can land as a sibling ADR without disturbing this one's contract — sub-decision 6's `pending-counterparty` flag is the seam where consortium-batch coordination plugs in.
- The Drafts-view onboarding shell (sub-decision 8) is a *reusable enrichment pattern*. The next onboarding-shaped experience (e.g., a major data-element pack rollout, a DEX-wide compliance refresh) lifts the same shell over the same surface.
- The `authoringMetadata` strip-at-materialise contract (sub-decision 4) generalises ADR 0046's publish-boundary doctrine to staging boundaries. Future staging surfaces (a staged Message draft? a staged Pitstop config?) can apply the same pattern verbatim.

## Open questions

- **Consortium frequency monitoring.** Sub-decision 15 chooses forward-references with the expectation that consortium-shaped onboardings are not the dominant pattern. The Platform Admin's onboarding-outcome dashboard should expose the *Waiting on counterpart* row count per onboarding so frequency can be tracked; if a meaningful share of new org admins land on majority-pending Drafts queues, atomic-across-batch becomes a real candidate for a follow-on ADR.
- **Workbook template distribution to Platform Admins.** Internal channel only (sub-decision 1) so no public distribution; the template lives in admin-tooling docs or as an export from the admin-side upload surface itself. Worth a one-line decision but not load-bearing for this ADR.
- **What happens if the org admin user named in `Users` sheet row 1 refuses the invite?** Mirrors today's behaviour for any org-admin-invite that gets declined — out of scope for this ADR; the org exists, the Drafts exist, the org needs to nominate a different org admin via the existing org-admin-replacement flow. The staged Drafts are unaffected.

### Designer-side calls explicitly named (so they don't get silently dropped)

These are decisions sub-decisions 8–17 leave to the implementer but should be made consciously rather than fallen into by default:

- **Bulk-affordance threshold default.** Sub-decision 10 marks the threshold *configurable*; the v1 default is **3 publishes** (so the bulk affordance surfaces on row 4). Honours `ux-hicks-minimize-choices` (graduated trust earned after small-N engagement) while keeping the bulk path reachable on short onboardings — a 6-Draft batch still gets the bulk affordance after the operator engages with half.
- **Counterpart-materialise notification accumulation.** Sub-decision 15(a) names a soft notification per counterpart. When multiple counterparts resolve within a short window (≤24h), the notifications **bundle per [ADR 0036](./0036-inbox-banding-and-bundling.md)'s Inbox bundle rules** rather than rendering N separate cards — bundling key `(sourceType='onboarding-resolution', counterpartyOrgId='*', intent='review')`, title surfacing the counterparty count (*"3 counterparts are now onboarded — review N Agreements"*).
- **Welcome-panel "engage" trigger.** Sub-decision 8's *"auto-dismiss on engage"* is the explicit *Got it* click — **not** any incidental scroll or click on the underlying queue. The panel is positionally above the queue but operators can interact with rows underneath without dismissing the panel until they explicitly acknowledge it. Mirrors [ADR 0017](./0017-migration-onboarding-panel-redirects-and-draft-migration.md)'s precedent.
- **Counterparty groups beyond Miller's range.** Sub-decision 9's group-by-counterparty default holds for the common case; a single counterparty with ≥10 `from-onboarding` Drafts (rare, possible for deep bilateral onboardings) sub-groups by *Sent* / *Received* direction per `ux-millers-chunking` to keep each visual chunk inside the 5–9 range.
