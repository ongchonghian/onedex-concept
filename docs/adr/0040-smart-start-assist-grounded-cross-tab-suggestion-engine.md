## Context

The registration flow defined by [ADR 0039](./0039-data-element-registration-admin-authored-single-page-flow.md) leaves Sarah (a platform-tier DEX admin) authoring elements from a near-blank state. Even with Smart Start on-ramps (CSV / PDF / Plain English / Fork) the **seed** is structural — it produces fields, not field-meaning. Sarah still hand-translates the requesting party's intent (typically expressed in a Confluence requirements page authored by BD/PM ahead of any Jira ticket) into:
- Schema field names + types + constraints
- A Compose complexity decision
- Cross-field validation rules in govaluate expression syntax
- Pack membership

A Jira-ticket scrape over 12 months on CTD showed ~4 real new-element development requests per year, almost always with `Signoff_MasterData_DE_*.xlsx` and a signed PDF attached — and a Confluence requirements page authored upstream. The PDF alone is under-specified for schema authoring; the Confluence narrative carries the disambiguating intent. Without something that bridges PDF + Confluence + authoritative external specs (BCA, IMDA, MOH publications) into the registration tabs, Sarah's quarterly authoring job remains manual translation against governance pressure.

The grilling also surfaced that **DexConnect's existing VLM** ([AI Transformation Utility - Product Design](https://afa-cdi.atlassian.net/wiki/spaces/CDIT/pages/1843920899)) is deliberately scoped to *runtime extraction against a known target schema* — its design explicitly out-of-scopes "Pitstop Portal upload flow" and states "AI extraction should not attempt to generate DE schema fields directly." Schema *authoring* from an unknown PDF is the opposite problem and needs its own engine, with a clean migration path back into DexConnect when both pipelines have stabilised.

## Decision

Smart Start gains a cross-tab AI-augmented authoring layer called **Smart Start assist** — a deterministic, hard-grounded suggestion engine that consumes the on-ramp seed plus configured grounding sources and produces per-field, per-rule, per-pack-membership *suggestions with provenance* across the registration flow. The engine drafts; Sarah reviews and signs off; the audit chain records what was suggested, what was edited, and what was rejected.

Sixteen sub-decisions resolved during grilling:

**1. Naming: `Smart Start` is the umbrella; `Smart Start seed` is the on-ramp output (existing concept); `Smart Start assist` is the cross-tab suggestion engine (this ADR's subject).** All three are pinned in [CONTEXT.md](../../CONTEXT.md). The user-facing term in copy is *"Smart Start assist"* — never "AI assist," "LLM overlay," "co-pilot," or "AI feature" (those date poorly, anthropomorphise the engine, and create chatbot-shaped expectations).

**2. Shape B — one-pass pre-population with streaming.** After the on-ramp seeds the draft, Sarah is dropped on the Schema tab immediately with seed-populated content. The assist engine streams its enhanced suggestions in over ~5–20 seconds while Sarah is already reading; other tab badges with *"Smart Start assist — generating…"* until the pass completes, then update to *"Rules · 3 suggested"* style counts. No blocking loading screen. Rejected: lazy per-tab (latency hits flow-switch moments) and always-on side panel (over-indexes on power-user ergonomics for a quarterly task).

**3. Six grounding inputs, scoped contract.** Smart Start seed (always present); Confluence requirements page (one URL, no precedence problem); curated DEX reference registry (scoped to the URL DEX per [ADR 0001](./0001-url-anchored-dex-context.md)); catalogue priors (sibling Elements in the same DEX); URL DEX context; optional explicit sample payload. Explicitly excluded: generic web search (noise source — replaced by curated registry), operator history (surveillance-y, low value), cross-DEX sibling priors (muddies domain conventions), user identity (assist behaviour must not vary by operator).

**4. Universal suggestion envelope across all tabs.** One shape: `{ id, tab, kind, payload, sources[], confidence, caveats[], liveEval, alternatives[] }`. `tab` + `kind` discriminator pair routes UI rendering; `payload` varies by kind (`field`, `enum`, `validation-rule`, `complexity-pick`, `pack-membership`); `sources[]` is the non-negotiable grounding citation list. Closed schema — extensibility lives in `payload` per-kind, not at the envelope level.

**5. Position A grounding: every suggestion must cite ≥1 source from the six inputs. No training-data inferences.** If the engine has a hunch with no citable source, it does not emit a suggestion. *This is the load-bearing defensibility decision.* Sibling-Element citations count as grounded but are weak: a suggestion citing only catalogue priors **caps at `medium` confidence** — never reaches `high`. Rejected: Position B (quarantine zone for ungrounded suggestions — becomes either a trap or a back-door) and Position C (treating "model knows this" as a citable source — unauditable). Audit defensibility — *"where did this field come from?"* — is the failure mode the grounding constraint exists to prevent.

**6. Three-bucket confidence (`high` / `medium` / `low`), calibrated by source quality and count, not LLM certainty.** Source quality is ordered: `reference-doc > pdf-region (VLM) > confluence-prescriptive > confluence-hedged > sibling-element > sample-payload`. **High** requires ≥2 independent sources agreeing, at least one of which is a structural-truth source (PDF region or reference doc). **Caveats demote, never promote** — Confluence "should/normally" hedging language forces `low` regardless of source count. No numeric confidence (false precision, invites shopping). No per-source weights as a tunable parameter (anti-feature).

**7. Two-tier engine architecture: Option 2 in production (`dex-monorepo`), Option 1 in prototype (`design-concepts`).** Production: a new stateless backend service (`smart-start-assist`) exposing one endpoint; portal calls one endpoint; service orchestrates Confluence fetch + VLM call + LLM overlay + reference-registry lookup + sibling-element query. Service-account Confluence credentials live in the backend (not operator-personal). Migration path to Option 3 (`/suggest-schema` co-housed with DexConnect) is a deliberate v2 refactor; the portal's API contract does not change during migration. Prototype: frontend-only, browser-resident. **The prototype is the spec for the production port** — same input contract, same suggestion envelope, same per-tab payload shapes. Production lifts the engine without reshaping the API.

**8. Position C lifecycle: persisted in draft, auto-invalidated on material input changes.** Suggestions live alongside the in-flight element draft (autosave) with a `regDraft.assist` field plus an `assistRunFingerprint` (hash of inputs). A "material change" — PDF re-uploaded, Confluence URL changed, reference registry config changed, sample payload swapped, URL DEX changed — invalidates the fingerprint and surfaces a *"Re-run Smart Start assist"* affordance. Editing a single schema field manually is **not** a material change; rejecting/accepting a single suggestion is not. Three corollaries: suggestions are creation-timestamped; edited suggestions become `modified from suggestion` (both versions kept); rejected suggestions persist in the audit log graveyard. Rejected: ephemeral (stochasticity destroys audit defensibility) and persisted-without-auto-invalidation (manual cache invalidation is a known anti-pattern).

**9. Two-tier audit chain.** The Element version (positive specification) carries minimal provenance per field/rule: `{ suggested, accepted, sources[], confidence, suggestedAt, assistVersion, confluencePageId?, confluencePageVersion? }` — a *closed* schema. The audit log (append-only, event-shaped) carries the full historical trail: `suggestion-emitted`, `suggestion-accepted`, `suggestion-edited`, `suggestion-rejected`, `assist-run-triggered`, `material-change-detected`, `element-published`. The audit log persists past Publish. A regulator querying the Element version alone can reconstruct *where each field came from*; querying the audit log reconstructs *how the element was authored*. Rejected: minimal-only Element provenance (loses defensibility against the published artefact alone) and full-audit-on-Element (bloats immutable record; immutability per [ADR 0026](./0026-agreement-snapshot-immutability-schema-upgrade.md) becomes harder).

**10. Loud degradation under failure.** Each grounding source has explicit per-failure-mode behaviour: VLM unreachable → fall back to Tesseract with banner; Confluence unreachable → engine runs without it, banner explains, suggestions sourced elsewhere cap at `medium`; reference registry unreachable → same; LLM overlay itself is critical-path — when it fails, the entire assist run fails cleanly with *"Continue manually or [Retry]"*. **The grounding constraint holds under degradation** — if degradation means a suggestion can no longer cite ≥1 source, the suggestion is dropped, not surfaced with a "best guess" label. `degradedSources: ['confluence']` is stamped on the audit log entry. Sarah never sees silent partial output.

**11. Confluence convention: assume State B, ship a template alongside the engine.** Today's element-requirements pages are loosely structured at best. The engine handles State B by default (LLM-driven section detection across heterogeneous prose). Alongside the engine, ship a *"Data Element Requirements"* Confluence template (sections: *Background & Stakeholders*, *Fields of interest*, *Validation expectations*, *Compliance regime*, *Sample artefacts*) and a *"Suggested by Smart Start assist"* badge that appears on Confluence pages used as grounding sources — making the value of template adoption visible to BD/PM authors. Over 2–3 quarters, the corpus converges to template-aligned (State A) and engine confidence on Confluence sources rises naturally.

**12. Reference registry: JSON-in-repo v1, per-DEX, section-level granularity, immutable versioned entries.** Schema: `{ dexes: { <dexSlug>: { publishers: [{ id, name, documents: [{ id, title, version, publishedAt, sourceUrl, sections: [{ id, title, excerpt, tags[] }] }] }] } } }`. Owned by platform-tier roles under `canManageDataElements`. Updates are manual, PR-reviewed — never auto-synced from regulator websites. Registry entries inherit snapshot immutability from [ADR 0026](./0026-agreement-snapshot-immutability-schema-upgrade.md): editing an existing section in-place would silently change the meaning of every Element citing it; instead, bump `version` and mark previous as superseded. **The registry is consultable** — Sarah can hover citations for excerpts, expand to side panels, navigate to publisher source URLs. Not just engine-internal. Empty registries are a normal state for early-stage DEXes; assist runs anyway with relevant suggestions capped at `medium`. v2 migration to Confluence-hosted is optional and deferred. In design-concepts: `portal-app/scripts/fixtures/dex-references.json`.

**13. Sequential per-tab ordering: `Schema → Complexity → Pack → Rules`.** The four passes share a single LLM session/context; each subsequent pass sees prior outputs. Rules is last because it has the most field-name dependencies on Schema — sequential ordering prevents rules from referencing fields the Schema pass did not produce (those would fail the grounding constraint and get dropped). Complexity runs second because it's cheap and unblocks visible-progress UX. Partial-failure tolerance flows through a dependency graph: Schema fails → Rules cannot run (depends on fields); Complexity fails → Rules can still run. Completion event stamps `assistRunFingerprint` with a `partial: true` flag when applicable. No `routing-rule` kind in v1 — routing remains out-of-portal per [ADR 0038](./0038-data-element-governance-three-independent-layers.md); reserved space, not implemented. Rejected: all-at-once (cross-tab failure granularity worse, prompt becomes a monster) and per-tab independent runs (loses cross-tab consistency).

**14. Provenance UI: Shape A inline, with three primitives.** Suggestions render in the existing tab layouts (no dual-pane, no modal). Three universal UI elements carry provenance across all tabs: **(a) provenance chip** — small marker per suggestion with confidence dot (`high` solid / `medium` half / `low` outline), sparkle icon, source-count badge; **(b) provenance popover** — opens on chip click, shows each source as a typed entry with excerpt + back-link, plus confidence + live-eval result + caveats + Accept/Edit/Reject/Open audit detail actions; **(c) caveat banner** — thin yellow banner above a suggestion's payload (not inside the popover) when caveats are populated, because caveats demote confidence and must be visible without opening a popover. **No bulk "accept all" affordance in v1** — direct violation of per-suggestion grounding/review contract. Same components across all four tabs. Tab labels carry live counts: *"Schema · 12 fields (3 modified, 1 rejected)."*

**15. Same capability gate as the rest of the registration flow.** `canManageDataElements` ([CONTEXT.md:113](../../CONTEXT.md:113)) covers Smart Start assist invocation alongside the existing CTAs (`+ New element`, `+ New version`, `+ New pack`). Forward-flexible: future split into a dedicated `canUseSmartStartAssist` sub-capability is non-breaking but unnecessary in v1 (no operational segmentation justifies it). Cost-control concerns (LLM budget, rate limiting) belong in quotas, not capability. Audit-log `actor` on `suggestion-emitted` events is `engine`; on `suggestion-accepted`/`suggestion-edited`/`suggestion-rejected` events the actor is the human operator.

**16. Phase 1 scope (prototype, Shape C).** Three on-ramps trigger assist — PDF/Form, Sample, and Plain English. Fork stays seed-only (parent already carries schema + rules + pack; marginal value of assist on Fork is low and muddies the demo story). One canonical full-depth demo element (Environmental Site Observations, [CTD-10435](https://afa-cdi.atlassian.net/browse/CTD-10435) on SGBuildex) with authentic Confluence fixture + 1 BCA reference doc with 2–3 sections + 3–5 sibling Elements + canned VLM extraction + canned LLM overlay producing ~8 Schema fields + 1 Complexity pick + 1 Pack + 3 Rules. Two DEX skeleton elements (HX lab result + TX vessel arrival) with minimal fixtures — just enough to demonstrate per-DEX boundary visually. A real-API-key toggle on the PDF on-ramp drives **both** VLM extraction **and** LLM overlay calls live (operator pastes Anthropic Claude vision key into `localStorage` with explicit "demo only · your key · your cost · cleared on tab close" warning) — the prompt structure must be stable enough to make a defensible real call, not a throwaway. Phase 2 deferrals: real backend service, real Confluence fetch, service-account credentials, Confluence template artefact ship, audit log UI surface, Fork-on-ramp assist.

**17. Structural-aware extraction contract (amendment, 2026-05-22).** PDF/Form extraction passes through three enforced layers — the change strengthens the seed-time engine so radio-clusters, tables, and muxed inputs do not arrive as flat fields. Each layer's contract is schema-enforced, not prose-enforced, because LLMs follow prose loosely.

  **Layer 1 — VLM spatial+semantic extraction with structural classification.** Per detected region, VLM emits `{ textContent, boundingBox: { page, y1, x1, y2, x2 }, structuralRegion }` where `structuralRegion` is one of a **closed 13-value vocabulary**: `standalone`, `single-boolean`, `signature-block`, `attachment-zone`, `composite-input.{date|phone|postal|generic}`, `radio-cluster`, `checkbox-cluster`, `table`, `likert-matrix`, `sub-form-group`, `repeated-pattern`, `static-disclaimer`, `unclassified`. Bounding boxes are mandatory on every region, normalised per page, and carry through to the audit log (not the Element-version artefact — keeps the immutable record closed-schema per sub-decision 9). The `unclassified` value forces affected fields to `low` confidence regardless of source count (caveats demote, per sub-decision 6).

  **Layer 2 — LLM schema engineering with mandatory dual output and self-audit.** The LLM consumes the Layer 1 region array and emits a strictly-typed JSON response containing four keys: `jsonSchema` (the data validation artefact); `x-presentation` (per-field library-agnostic presentation hints — closed vocabulary: `text | numeric | checkbox | radio | multiselect | data-grid | likert-scale | fieldset | repeater-block | composite-{date|phone|postal|generic} | file-upload | signature | disclaimer-text`, with per-field `labels` and likert-matrix-specific `{ rowLabels, optionLabels }` sub-namespaces); `x-presentation-order` (canonical render-order array merging input keys with synthetic `_static_N` disclaimer keys); and `refitSuggestions[]` (self-audited structural-restatement candidates the LLM noticed in its own output, shaped per sub-decision 4's universal envelope with `kind: "structural-restatement.merge-to-{table|enum|array-object|likert}"` and `mergedFromFieldIds`). The LLM is **strictly forbidden** from producing `<noun>_<n>_<attr>` flat numbered keys; outputs containing this pattern are rejected by the constraint validator and re-prompted once with the violation surfaced. Second failure → suggestion dropped per sub-decision 10's loud-degradation principle.

  **Layer 3 — deterministic name-pattern check folded into Layer 2's validator.** An earlier shape considered Layer 3 as an *active AST transform* (silently fold detected `item_1_*`, `item_2_*` patterns into `array<object>`). Rejected: silent mutation violates [ADR 0039 §80](./0039-data-element-registration-admin-authored-single-page-flow.md)'s deliberation invariant and breaks the audit chain (Sarah sees folded output with no signal a fold happened, so a regulator querying the audit log later cannot distinguish "extracted as array<object>" from "extracted flat, secretly folded"). Instead, the name-pattern regex check (`/^[a-z_]+_\d+_[a-z_]+$/i`) is part of Layer 2's validator and triggers re-prompt with explicit violation messaging. Patterns that survive Layer 2 (because VLM and LLM both agreed they were genuinely independent fields) surface as `refitSuggestions` entries for Sarah's deliberation through [ADR 0041](./0041-smart-start-refit-mid-edit-structural-restatement.md), never as silent transforms.

  **The `x-presentation` sidecar contract.** Strict library-agnostic. Composer downstream owns the `hint → Mantine/MUI/RJSF widget` mapping. **No `uiSchema` key ever appears in the artefact** — that direct coupling was rejected because it binds the immutable Element version (per [ADR 0026](./0026-agreement-snapshot-immutability-schema-upgrade.md)) to a specific UI library's vocabulary, making a future library swap a schema migration with downstream Agreement-snapshot consequences. `x-presentation-order` is the canonical render-order array because JSON Schema's `properties` is an object with implementation-defined key order — a separate ordered array is necessary to interleave synthetic disclaimer keys with property keys deterministically. Labels live in `x-presentation.{field}.labels` (not in a mutable per-DEX vocabulary registry) — a label is part of the *meaning* of an enum value at the moment of signing; mutable registries introduce retrospective semantic drift, which breaks the audit story ("show me what was displayed when this was signed").

  **`structuralRegion` → `jsonSchema` + `x-presentation.hint` mapping (Layer 2 enforcement table):**

  | structuralRegion | jsonSchema constraint | x-presentation hint |
  |---|---|---|
  | `standalone` | `type: "string"` or `"number"` (Layer 2 infers from label semantics — unit suffix, numeric keypad icon, "$" prefix) | `text` or `numeric` |
  | `single-boolean` | `type: "boolean"` | `checkbox` |
  | `radio-cluster` | `type: "string", enum: [...]` — `enum` values come from the per-field `options[]` contract documented below | `radio` |
  | `checkbox-cluster` | `type: "array", items: { enum: [...] }` — same `options[]` contract; `selectionMode: "multiple"` | `multiselect` |
  | `table` | `type: "array", items: { type: "object" }` | `data-grid` |
  | `likert-matrix` | `type: "object", properties: { ... }` (Layer 2 validates *shared enum across all rows*; else demote to `sub-form-group`) | `likert-scale` |
  | `sub-form-group` | `type: "object", properties: { ... }` | `fieldset` |
  | `repeated-pattern` | `type: "array", items: { type: "object" }` | `repeater-block` |
  | `composite-input.{sub}` | `type: "string", pattern: "..."` (caveat: caps at `medium` confidence per sub-decision 6 — visual muxing can be downstream-semantic; verify) | `composite-{sub}` |
  | `attachment-zone` | `type: "array", items: { format: "uri" }` (Layer 2 detects singular vs plural — "Upload a document" → `maxItems: 1`) | `file-upload` |
  | `signature-block` | `type: "string", format: "data-url"` (additionally hard-lights the high-stakes indicator on the Compose complexity tab per [ADR 0039 §5](./0039-data-element-registration-admin-authored-single-page-flow.md) — **no auto-selection** per [ADR 0039 §80](./0039-data-element-registration-admin-authored-single-page-flow.md); Sarah still picks) | `signature` |
  | `static-disclaimer` | *No property emitted.* Synthetic `_static_N` key in `x-presentation` + `x-presentation-order` only. Markdown body. | `disclaimer-text` |
  | `unclassified` | `type: "string"` (fallback) | `text` (capped at `low` confidence) |

  **Cost model: Phase 1 ships with zero additional LLM calls.** The structural self-audit is folded into Layer 2's existing call as an additional output channel — the LLM already holds the full context (Layer 1 regions + bounding boxes + the schema it just engineered) and self-reports structural patterns it noticed without a separate inference pass. Phase 2 may split into a dedicated structural-classifier call if Layer 2 self-audit proves too weak; deferred until Phase 1 usage data exists. The emitted `refitSuggestions[]` surface through the [ADR 0041](./0041-smart-start-refit-mid-edit-structural-restatement.md) drawer.

  **Enum-options extraction contract (amendment 2026-05-22).** The original §17 contract said `radio-cluster` and `checkbox-cluster` regions map to JSON Schema `enum` — but did not specify *where the option values themselves come from*. In practice the Layer 2 LLM emitted `enum: []` (empty options array), leaving Sarah to retype every visible choice by hand. The corrective: every field whose `type` is `enum` (regardless of which structural region it came from) **must** carry `options[]` with one entry per visible choice, plus a `selectionMode` discriminator. Each option carries `{ value, label, hasFreeTextBlank }`:
  - `value` — the wire-stable code (slug-cased for free text labels like "Chinese" → `"chinese"`; preserved as-is for codes already shown on the form like `"WP"` or `"PSA01"`).
  - `label` — the visible text exactly as it appears on the form. Lands in `x-presentation.{field}.labels[value]` per sub-decision 17's existing sidecar contract.
  - `hasFreeTextBlank` — true *only* when this specific option has an adjacent blank line, underscore rule, or input box that the operator would fill in when they pick it (e.g., `Others ______`, `Insurance Name ______`). Default false. **This is a CRITICAL signal** — see the companion-field synthesis below.

  `selectionMode` discriminates `"single"` (radio-cluster — JSON Schema `{type: "string", enum: [...]}`) from `"multiple"` (checkbox-cluster — JSON Schema `{type: "array", items: {enum: [...]}}`). The mode is about *intent*, not widget: a form using checkboxes for a pick-exactly-one set is still `"single"`. When in doubt and the visual is checkbox-shaped, the prompt defaults to `"multiple"`.

  An enum field with zero options is treated as a misclassification — Layer 2 must re-emit with a different type (likely `"string"`) rather than ship an empty options array.

  **Companion-field + cross-field rule synthesis for the "Others ____" pattern (amendment 2026-05-22).** Real-world forms routinely encode conditional fields visually: an enum option followed by a blank line means *"if this option is selected, fill in the adjacent text."* Examples on the Innoquest specimen-collection form: `Bill To: ... ☐ Insurance Name _____`, `HIV Screening: ... ☐ Others _____`. Without an explicit contract, Layer 2 emits the enum without the companion field and without the conditional-required rule — the operator has to spot the pattern manually and hand-author both.

  The corrective: for every enum option flagged `hasFreeTextBlank: true`, the seed builder synthesises **two coordinated artefacts**:

  1. **A companion text field** named `<base>_<option_value>_specify` (e.g., `bill_to_insurance_name_specify`, `screening_purpose_others_specify`), of type `string`, placed in the same group as its parent enum. The field carries a `_companionFor: { base, option }` provenance hint linking it back to the option it serves. Required is `false` at the per-field level — the conditional requirement is expressed as a cross-field rule below, not as a per-field `required: true`.

  2. **A govaluate-style cross-field validation rule** on the Rules tab, of the form: *"the rule passes when the enum does not hold the flagged option, OR the companion field is non-empty."* For `selectionMode: "single"`: `<base> != "<option>" || (<companion> != "" && <companion> != null)`. For `selectionMode: "multiple"`: `!contains(<base>, "<option>") || (<companion> != "" && <companion> != null)`. The rule's `on_failure` message names the option label and the companion field explicitly: *"When `<option label>` is selected, `<companion>` must be filled in."*

  Generalises beyond literal "Others" — works for any enum option with an adjacent blank, irrespective of the option's label. Three or more flagged options on one enum produce three or more companion fields and three or more rules, all coordinated. Multiple companion fields per option (e.g., the Innoquest `Insurance Name ____ Member / Policy No ____` pattern, where one selection drives two trailing inputs) is **Phase 2** — Phase 1 ships with one companion per flagged option.

  This sub-contract sits cleanly inside the three-layer architecture: Layer 1 (VLM) detects and emits `hasFreeTextBlank`, Layer 2 (LLM) preserves it in the suggestion payload, and the seed-builder (deterministic, JavaScript) materialises the companion field + rule. No additional LLM call. Audit-defensibility per sub-decision 9 is preserved — the rule carries `source.suggested: "engine"` per the existing provenance contract; Sarah can edit/delete the rule, and the audit log captures the act.

  **Per-field Presentation panel (amendment 2026-05-22, third in §17).** The mapping table above gave each `structuralRegion` a *derived* default hint. In practice the derived default doesn't always fit — a `string` description field needs `textarea`, a 30-option `enum` needs `dropdown` instead of 30 radio buttons, a `boolean` toggle needs `switch`. The original §17 contract had no override channel; Sarah's only escape was to drop into the JSON view or change the data type (which loses the schema work). The corrective is to expose a per-field **Presentation panel** in the schema authoring UI and to expand the closed hint vocabulary with **presentation-alternative hints** — bounded library-agnostic alternatives within a single semantic type. The contract preserves the library-agnostic guarantee while restoring Sarah's authoring agency.

  **Expanded hint vocabulary — 7 new entries.** Each new entry is a presentation alternative within a single semantic type. The original 13-entry vocabulary holds; the 7 new entries are *opt-in* — Sarah picks them from the per-field Presentation panel's hint dropdown when the derived default doesn't fit. The full alternatives map (the closed list of allowed hints per field type, listed default-first) is:

  | Field type | Allowed hints (default first) |
  |---|---|
  | `string` | `text`, `textarea` |
  | `number` / `integer` | `numeric`, `slider` |
  | `boolean` | `checkbox`, `switch` |
  | `enum` (single-select) | `radio`, `dropdown`, `segmented` |
  | `array<enum>` (multi-select) | `multiselect`, `checkboxes` |
  | `array<object>` | `data-grid`, `repeater-block` |
  | `object` | `fieldset`, `card` |
  | `date` / `datetime` / `composite-input.*` / `signature-block` / `attachment-zone` / `static-disclaimer` | *no override* — derived hint is structurally tied to extraction or to a sub-type; not exposed as a dropdown |

  The new entries — `textarea`, `slider`, `switch`, `dropdown`, `segmented`, `checkboxes`, `card` — extend the closed vocabulary by 7. `repeater-block` was already in the vocabulary; what's new is that it becomes user-overridable for `array<object>` rather than auto-picked. Composer downstream maps each hint to a library-specific widget (e.g., `radio` → Mantine `<Radio.Group>`, `dropdown` → `<Select>`). The library-agnostic guarantee from the original §17 contract holds — none of the new entries reference Mantine/MUI/RJSF-specific keys.

  **Lazy-override mechanism — `field.presentation.hintOverride`.** The state lives on a `field.presentation` substructure mirroring the wire's `x-presentation.{field}` shape 1:1. `hintOverride` is **lazy** — unset means "use the derived default"; set means "force this hint instead." The serialiser writes the *resolved* hint (override-or-derived) into `x-presentation.{field}.hint`, so Composer downstream never needs to know whether the value came from a derive or an override — it sees the final answer only. The parser inverts: it reads the wire hint, runs `regDeriveHint(field)`, and populates `hintOverride` *only if* the parsed value differs from the derived default AND is in the allowed-alternatives list for the field's type. Stale overrides (e.g., from a field whose type changed) silently fall through to the derived default rather than producing invalid output. Reset = `delete field.presentation.hintOverride`.

  This pattern keeps autosaved drafts compact (most fields never need an override; their `presentation` substructure stays undefined), preserves the "derived vs overridden" semantic across round-trip, and avoids magic-value comparisons in the resolver. The full substructure carries:

  ```
  field.presentation = {
    hintOverride?,                   // closed-vocabulary string or undefined
    originAnnotation?,               // editable live value (composite-input)
    originAnnotationFromSeed?,       // immutable snapshot from VLM extraction
    rowLabels?,                      // likert row labels (Phase 2 editor)
    optionLabels?                    // likert option labels (Phase 2 editor)
  }
  ```

  **`originAnnotation` provenance — snapshot + live, with sparkle indicator.** Composite-input fields carry an origin string (*"Original form: 6 boxes"*) extracted by VLM. Sarah needs to edit it (e.g., to rewrite as a user-facing helper *"Enter 6-digit postal code"*) without losing the original. The pattern: at seed-construction time, the seed-builder sets BOTH `presentation.originAnnotation` (live, editable) AND `presentation.originAnnotationFromSeed` (immutable) to the same string. The Presentation panel renders a sparkle "Auto-extracted" indicator when the two are equal; on Sarah's first edit, the strings diverge and the sparkle drops. A Phase-2 *"Revert to extracted"* button restores the live value from the snapshot. The snapshot stays in-memory draft state only — it is NOT carried on the wire (the published Element artefact carries only the live value, which is what Composer renders). This is intentional: the snapshot's job is helping Sarah author, not informing downstream consumers; the audit log captures editing history if defensibility queries arise.

  **Display-time applicability — persisted override vs effective override (amendment 2026-05-23, UX-46).** The Lazy-override mechanism above describes *wire-shape* behaviour: stale overrides resolve to derived on serialisation. It does not constrain *UI predicate* behaviour, and the early implementation left an asymmetry — `field.presentation.hintOverride` was treated as "override active" by the Presentation panel toggle's theme-tinted background and its "custom override active" tooltip regardless of whether the persisted value was still applicable to the current field type. A `string` field with a stale `textarea` override (left behind from a prior `string` → `number` → `string` round-trip) lit up the toggle as "custom override active" even though the serialiser correctly resolved the wire to `text`. The toggle was lying about the wire. The same asymmetry affected the origin-annotation divergence check, which counted `field.presentation.originAnnotation !== originAnnotationFromSeed` as an override regardless of whether the field type was still `composite-input`.

  The corrective splits the concept into two terms with distinct contracts:

  - **Persisted override** — the value stored in `field.presentation` (the operator's last expressed `hintOverride` pick, or `originAnnotation` text). Survives type changes by design. Is what enables the type-change-and-back round-trip case: pick `textarea` on `string`, switch to `number` (override now stale, wire resolves to `numeric`), switch back to `string` → the `textarea` pick is restored without re-entry. Never mutated by type change.

  - **Effective override** — the override *currently in force* for the field's *current* type. Equals the persisted value when applicability holds; equals "none" when the persisted value is stale.

  Predicates that drive UI affordances (toggle tint, tooltip wording, any 30-field-scan summary) check applicability at *display time*: `regHasPresentationOverride(field)` validates `field.presentation.hintOverride` against `regAlternativesFor(field).includes(override)` before counting it, and validates `field.presentation.originAnnotation` divergence only when `field.type === 'composite-input'`. The persisted value is never deleted on type change — the predicate is the gate, not the state mutation.

  The doctrine generalises beyond these two predicates to any future UI signal sourced from `field.presentation`. The rule is **state preserves; display gates.** Rejected: cleaning state on type change (loses the round-trip case, contradicts the "lazy" framing of the override mechanism above). Rejected: keeping the predicate as-is and adjusting the tooltip copy to hedge (the toggle's whole job is single-glance scanability across a 30-field draft — a hedged tooltip is no defence against a misleading tint).

  **Panel closes on field-type change (amendment 2026-05-23, UX-46).** Field-type change is a destructive operation already — the handler in [`register-element.js`](../../portal-app/scripts/register-element.js) at the type-select change listener wipes `children`, `validation.itemType`, `validation.enumValues`, `validation.subType`, etc., and warns via `regTypeChangeWarning` before proceeding. When the Presentation panel is open for a field whose type changes, the panel's row composition would otherwise mutate in place — the origin-annotation row appearing when switching to `composite-input`, the labels-summary row appearing for `enum`, the hint dropdown swapping to the resolved-hint chip for locked types. Mutating panel structure in response to upstream changes is a poor reactive contract: continuity of context is lost, and both sighted-mouse and screen-reader affordances would need significant chrome (an `aria-live` region, fade-in motion, or an explicit acknowledgement banner) to communicate the reshape honestly.

  The cleanest contract is to **close the panel on type change**. The type-change handler removes the field's id from `_regPresentationOpenIds` before re-rendering. Sarah re-opens the panel to see the new shape — a fresh context for the new field type. No live region needed (nothing reshapes in place), no motion work (no reshape to animate), no risk of misleading mid-flight states. Honours the same posture as the existing `regTypeChangeWarning` confirm modal: type change is a meaningful break, not an incremental edit. Rejected: in-place re-render with `aria-live` announcement (chrome cost not justified for the relatively rare type-change-while-panel-open case); in-place re-render with explicit acknowledgement banner (most chrome, defensible only if operator-deliberation-audit weighted it heavier — `regTypeChangeWarning` already captures the deliberation moment on the type-change side).

  **Audit event — `presentation-override-set`.** A new event kind joins the closed audit vocabulary from sub-decision 9: every time Sarah picks a non-derived hint (or resets back to derived), `presentation-override-set` fires with payload `{ fieldId, fieldName, previousHint, newHint, reason: 'override' | 'reset' }`. This makes regulator queries answerable: *"did the operator deviate from auto-derived presentation? when? on which field?"* The same defensibility property that justifies §17's broader contract justifies this audit slot — presentation choice in regulated environments (trade, healthcare, government) is materially load-bearing for how operators interpret data-entry semantics, and capturing the deliberation moment is cheap insurance.

  **UI surface — icon-triggered peer expander.** Sarah's entry point is a small `ti-adjustments-horizontal` (sliders) icon button in every field row's right-side cluster, between the group picker and the delete button. Click toggles a peer expander below the row showing the hint dropdown (when alternatives exist), the originAnnotation editor (when composite-input), an enum-labels summary with anchor-jump to the existing pick list editor (deduplicates the labels editing surface), and a read-only order indicator. The icon's background is theme-tinted when any override is active so Sarah can scan a 30-field draft and instantly spot custom presentation. Multiple expanders can be open simultaneously (no auto-collapse) — Sarah is in control. This UI shape was the result of trading off the original "always-visible 1-line summary chip" idea (24px of vertical chrome per row × 30 fields = 720px of dead space) against a popover (z-index traps in nested-dropdown UIs); the icon-trigger pattern preserves data density without sacrificing the editing affordance.

  **Wire shape unchanged.** The Presentation panel adds NO new keys to `x-presentation` beyond what was already in this section. The override resolution is internal to the draft state; the wire output continues to look like `x-presentation.{field}.{hint, labels?, originAnnotation?, rowLabels?, optionLabels?}` as documented above. Downstream consumers (Composer, regulators) see a single resolved hint and don't need to know about the override mechanism.

  **Boolean-matrix anti-pattern — two-lever fix (amendment 2026-05-22, fourth in §17).** Real-form testing surfaced a recurring failure mode: when VLMs encounter dense checkbox grids (rows × columns of boolean cells — Innoquest's *Nature of Specimens* table being the canonical example), spatial reasoning collapses and the model emits a `sub-form-group` containing N isolated boolean fields instead of a single `table` region. Downstream, Layer 2 dutifully translates the `sub-form-group` into a flat JSON `object` with N boolean properties. The result: the Presentation panel cannot offer `data-grid` ↔ `repeater-block` overrides (those are only valid for `array<object>`), and Sarah is left with an ugly card of N toggles. **We deliberately do NOT solve this by adding `boolean-matrix` to the closed vocabulary** — adding presentation alternatives for failure modes invites the vocabulary to grow without bound and tightly couples the artefact to UI quirks. Instead, we use two complementary levers that preserve the closed vocabulary while fixing the underlying classification.

  **Lever 1 — VLM prompt constraint (preventative, universalised UX-28).** A new clause in `smartStartPrompts_extractFieldsFromPdf` makes 2-D matrix detection a hard rule. The clause is **domain-agnostic** — anchored purely to spatial geometry, not vocabulary like "specimens" or "tests" or "checkboxes". Verbatim: *"Scan for intersecting 2-D grid layouts. A true matrix exists when ALL THREE spatial signals are present: (a) a left-edge column of row labels (text identifiers), (b) a top-edge row of shared column headers (text identifiers), AND (c) intersection cells containing interactive elements (checkboxes, empty input boxes, score circles, or sign-cells). When all three are present, you MUST classify the entire bounding box strictly as `structuralRegion: "table"`. Do NOT classify as `sub-form-group` even if the row labels look like sub-form headings. The geometric intersection structure dominates over per-cell semantics. This rule applies equally to inspection sheets, logistics manifests, lab forms, surveys, and any other tabular intake pattern."* By anchoring to *intersecting 2-D grid layouts* + *shared column headers*, the VLM is forced to recognise the structure regardless of language or domain shorthand on the form.

  **Lever 2 — refit self-audit safety net via Cartesian-product detection (corrective, universalised UX-27).** Even with the prompt strengthened, VLMs occasionally produce the failure shape. The corrective is a deterministic post-extraction scanner that operates on the AST footprint rather than domain semantics. A collapsed matrix leaves a very specific mathematical signature in the property names: when the VLM fails and flattens a matrix, it typically concatenates row and column identifiers (e.g., `plain_clinic`, `plain_lab`, `edta_clinic`, `edta_lab`, …). The scanner detects this by:

  1. **Scan objects** with > 4 children.
  2. **Type homogeneity** — verify ≥80% share the dominant type (any type, not just boolean — works for string, number, boolean cells alike).
  3. **Cartesian-product proof** — decompose each property name on its LAST `_` into `{prefix, suffix}`. Require ≥2 distinct prefixes AND ≥2 distinct suffixes. Verify ≥80% of the `prefix × suffix` combinations are actually present in the property set. **Only when this 2-D Cartesian footprint is detected do we emit the suggestion.**
  4. **Emit `structural-restatement.merge-to-table`** with a proposed `array<object>` shape: one item-level property per detected suffix (typed by the dominant type), plus a `row_label` string property. Seed the array with one entry per detected prefix so Composer can rehydrate the fixed-row taxonomy on first render.

  The Cartesian check is **load-bearing** — without it, a naive "≥80% same type" trigger would mis-fire on legitimate clusters that lack the matrix footprint: Terms & Conditions blocks, feature-flag groups, consent screens. Those have boolean homogeneity but no prefix × suffix structure (each name has a unique suffix). The scanner has been verified on three orthogonal cases — Innoquest lab matrix (10×2 booleans, fires), T&C consents (5 booleans, correctly silent), logistics manifest (5×3 booleans, fires) — confirming domain-agnostic universality.

  The scanner runs on every autosave settle (cheap O(n) scan, no inference call) and on manual *Re-run structural review* clicks, matching the existing refit trigger tiers per [ADR 0041 §2](./0041-smart-start-refit-mid-edit-structural-restatement.md). Confidence is capped at `medium` (engine-only detection per §5's grounding contract); `low` when coverage is below 95%, with a caveat naming the partial-coverage ratio.

  **Why this beats adding `boolean-matrix` to the vocabulary.** A `boolean-matrix` hint would be a UI workaround that papers over the structural misclassification — the data still lives in a flat `object`, validation rules can't reference rows generically, and the presentation override would be the only thing distinguishing it from a regular fieldset. By forcing the data to take the canonical `array<object>` shape, every downstream concern (validation, rendering, audit, refit, future re-extraction) operates on the right shape. The hint vocabulary stays at 20 entries (13 original + 7 presentation alternatives from the third amendment); growth happens only when a new SEMANTIC type emerges, not when a UI rendering quirk surfaces.

  **The fixed-row case (Phase 2 note).** The Innoquest table has FIXED rows (Plain, EDTA, Fluoride are a known taxonomy — operators don't invent new sample types). The current Lever 2 proposed shape includes a `_seedRows` field carrying the original row labels, but Composer's array<object> rendering currently treats arrays as user-extensible. A Phase-2 enhancement could add a `fixedRows: true` flag (or extend `data-grid` semantics) to lock the row set at extraction time. Defer until Phase 1 usage data confirms the case is common enough to warrant the extra surface.

## Consequences

**Architectural:**
- The prototype's `portal-app/scripts/smart-start-assist.js` module (new) becomes the spec for the production `smart-start-assist` backend service. Same input contract, same envelope. A future port lifts logic without reshaping the API.
- The `regDraft` state shape gains an `assist` substructure and an `assistRunFingerprint`. Autosave at [`register-element.js:240`](../../portal-app/scripts/register-element.js:240) area extends to persist suggestions across reloads.
- A new fixtures directory (`portal-app/scripts/fixtures/smart-start/`) carries: per-DEX reference registries, per-element canned VLM responses, per-element canned LLM overlay responses, per-element authentic Confluence excerpts.
- The per-Element version's `source.*` provenance fields become part of the published artefact schema. This is a one-way commit — once shipped, removing fields is a breaking change.

**Process / operational:**
- A Confluence template (*"Data Element Requirements"*) is authored and published as a paired deliverable. Without it, the engine's State B fallback works but the adoption flywheel for State A does not start.
- The reference registry is launched with at least one BCA document curated for SGBuildex; HX (MOH) and TX (MPA/IMDA) registries land with skeleton content. Filling them out is platform-admin operational work going forward.
- Live LLM calls in the prototype are bounded but real — ~4 sequential calls per registration run plus 1 VLM extraction. Stakeholders running the demo with the API-key toggle pay their own way; cost stays out of dex-monorepo until production.
- The prompt structure (per-tab system prompts + JSON-mode response contract) becomes a Phase 1 deliverable. Prompt versioning is tracked via `assistVersion`, stamped on every Element version produced.

**Forward-compatibility / migration paths:**
- Option 3 (DexConnect co-house) remains the long-term architecture. When DexConnect adds a `/suggest-schema` endpoint, the production `smart-start-assist` service becomes a thin façade — or merges entirely — without touching the portal's API contract.
- The audit log's append-only event shape is forward-compatible with a future audit-log UI surface (queryable per element, per draft, per operator). Data captured from v1 onward; UI lands later.
- Adding a fifth tab (e.g., `Routing`) is a payload-kind addition, not an envelope change. The dependency graph in sub-decision 13 extends naturally.

## Considered alternatives

- **Skip assist; rely on Smart Start seeds alone.** The seed is structural — Sarah still hand-translates intent. The Jira scrape showed `Signoff_*.xlsx` is self-sufficient for Schema but rules/pack/complexity remain manual. Net leverage is small.
- **Build a separate "AI assistant" feature outside the registration flow.** Drives the wrong mental model (Sarah switches between "register" and "ask the assistant"). The grilling converged on *Sarah reviews a draft the engine pre-assembled*, not *Sarah chats with an assistant*. Inline is correct.
- **Wire DexConnect's existing VLM directly into the portal as the OCR engine.** DexConnect's design explicitly out-of-scopes schema authoring and the portal upload flow. A wrong-target reuse would either fail or force defining a target schema before authoring it — exactly the problem registration is supposed to solve. Migration to a `/suggest-schema` endpoint co-housed with DexConnect remains the right *v2* shape; v1 ships independently.
- **Generic web search as a grounding source.** Open-web noise (SEO spam, outdated regulatory drafts, LLM-generated content that itself hallucinates form structures) contaminates the suggestion chain. The LLM cannot distinguish gold from noise. Replaced by the curated DEX reference registry.
- **All-at-once LLM call producing all four tabs in one response.** Cross-tab failure granularity is worse; the prompt becomes a monster; rules-referencing-nonexistent-fields is a real risk that sequential ordering prevents structurally.
- **Bulk "accept all high-confidence" affordance in v1.** Trains Sarah to skip review; loses the audit-defensibility property the whole design rests on. Could land later if data shows it's actually wanted; not v1.
- **Frontend-only architecture in production (Option 1 everywhere).** API keys in the browser, frontend-paid LLM cost, no central telemetry — operationally awful. Acceptable only in the design-concepts prototype where there is no backend.

## Future work (Phase 2 and beyond)

- Migration to Option 3 (DexConnect co-house) when DexConnect adds `/suggest-schema`.
- Audit-log UI surface: queryable views per element draft, per operator, per timespan; "show me everything Smart Start assist did across the last quarter" queries.
- Fork on-ramp gains assist (Confluence + reference grounding for rules/pack/complexity on top of the forked schema).
- Confluence-hosted reference registry as an optional migration target if JSON-in-repo curation becomes painful.
- `routing-rule` payload kind once routing-rule authoring lands in the portal (currently out-of-portal per [ADR 0038](./0038-data-element-governance-three-independent-layers.md)).
- Per-DEX prompt-tuning: as the engine accumulates accepted/rejected suggestions per DEX, the corpus becomes few-shot examples for sharper future runs (the "bidirectional learning loop" surfaced during grilling).
- Capability split: `canUseSmartStartAssist` as a dedicated gate, parallel to the existing forward-flexibility note on `canManageDataElements`.
- **Per-DEX app-maintained LLM-friendly wiki as a fifth grounding source.** Idea raised after this ADR was accepted: an app-team-curated, markdown-or-similar, retrieval-enabled knowledge base scoped per DEX (`wiki/bx/`, `wiki/hx/`, `wiki/tx/`) carrying *app-specific conventions, patterns, and accumulated learnings* that none of the four current sources hold — e.g., "manpower attendance: the `nric` field needs the K modulus check," "lab reports: `sample_date` always precedes `test_date`," "decision log: we don't put `description` fields on enums." Distinct from Confluence (upstream BD/PM-authored intent) and from the reference registry (external authority docs); the wiki carries platform-admin tribal knowledge. The capability shift this enables is **discovery mode** alongside the existing lookup mode — the engine can semantically search the wiki for relevant entries given the seed + Confluence + reference context, surfacing citations the operator wouldn't have known to attach. Treated as **additive**, not a replacement: Confluence and the reference registry retain their roles unchanged. Provisional source-ranking placement: below Confluence in the same prescriptive/hedged buckets (`reference-doc > pdf-region > confluence-prescriptive > wiki-prescriptive > confluence-hedged > wiki-hedged > sibling-element > sample-payload`). Adoption risk is real (same shape as the Confluence-template flywheel — wiki with no content provides no value); two mitigations worth considering when this lands: (1) auto-seed wiki entries from published Elements with accepted suggestions, so the wiki grows as a side effect of normal registration work; (2) an optional "capture one learning" affordance on the Review tab. Deferred to Phase 2 because Phase 1's scope is already substantial (real VLM + real LLM, three on-ramps, three DEXes); shipping the wiki before knowing whether the four-source design is sufficient is over-extension. When this lands, it gets its own ADR (provisionally 0042 — ADR 0041 is now reserved for Smart Start refit) grounded in real Phase 1 usage data.
