# Data Element registration is an admin-authored single-page flow

> **Status:** accepted (2026-05-20)
> Builds on [ADR 0038](./0038-data-element-governance-three-independent-layers.md) (three-layer governance), [ADR 0025](./0025-data-element-compose-complexity-attribute.md) (compose_complexity), and [ADR 0026](./0026-agreement-snapshot-immutability-schema-upgrade.md) (snapshot immutability). Decides the registration UX: capability gating, IA, on-ramps, flow shape, vocabulary, inference, demos. Resolves a fourteen-question grill-with-docs session against the dex-monorepo (PDF: legacy nine-step relay) and the existing portal-app surface.

## Context

Legacy Data Element creation/update is a nine-step relay between four actor types: CSM prepares an Excel → CSM raises an SR → SR approved → Dev edits Swagger YAML in repo → PR → merge → Leads manually copy YAML into SwaggerHub per environment (2.0 → DEV/TEST/UAT, 1.0 → PRE-PROD/PROD, 1.1 → stable pre-prod) → Leads call four POSTMAN endpoints on mockserver (`/getSwaggerDataElements`, `/usercasepersonas`, `/verificationFlow`, `/personas`) → DevOps restarts the relevant pitstops to fetch new config. Source: *update process-190526-231734.pdf*.

The relay is doing several different jobs that the new portal architecture has already separated:
- The CSM-as-originator role dissolves — registration becomes a **platform-admin-tier** action authored directly by a Sarah-class DEX admin (per [ADR 0029](./0029-user-org-affiliation-as-n-to-m-with-embedded-dex-roles.md) `platformRole`, [ADR 0030](./0030-demo-persona-resolution-via-persona-pill-and-url-dex.md) persona-category `platform-admin`).
- The "use case" and persona-graph (`producedBy` / `consumedBy` / `providedOnBehalf`) concepts have been refactored away — replaced by Agreements + Data Elements as the atomic units, with persona resolved at compose/message time via counterparty + org roles ([ADR 0028](./0028-routing-is-not-an-agreement-property.md), [ADR 0031](./0031-counterparty-attribution-dispatch-rule.md)).
- Multi-environment SwaggerHub version pinning is replaced by **snapshot immutability** ([ADR 0026](./0026-agreement-snapshot-immutability-schema-upgrade.md)) — new version = new immutable record, no in-place edit.
- Pitstop config refresh is replaced by reactive scope capture at the moment of friction ([ADR 0033](./0033-reactive-pitstop-scope-capture-with-inference.md)).
- The three back-end systems the relay wires together (AJV JSON Schema, govaluate expression rules, orchestrator JSONB rules) are surfaced as three independent layers by [ADR 0038](./0038-data-element-governance-three-independent-layers.md).

What remains is the question of **how** Sarah authors a Data Element in the portal. The grilling pinned that the legacy framing of "register" as a one-off enrolment is wrong (versions get published, not registered), that the surface must reuse the unified IA (a separate platform-admin sidebar `SIDEBAR_ITEMS_PLATFORM` was retired on 2026-05-17 — see [CONTEXT.md:359](../../CONTEXT.md)), and that the prototype lives at `portal-app/` running offline against a localStorage workspace per [ADR 0034](./0034-prototype-to-functional-auto-demo-runner.md).

This ADR captures the fourteen sub-decisions that the registration UX resolves.

## Decision

A Data Element registration UX is a **single-page admin-authored flow**, surfaced as CTAs on the existing Data elements catalogue page, gated by a single capability, with four schema-seeding on-ramps converging on a five-tab review-and-publish canvas. Fourteen sub-decisions:

**1. Capability-gated CTAs on the existing Data elements page; no new nav, no `/admin/` route.** The "Data elements" sidebar item at [`portal-app/scripts/app.js:7995`](../../portal-app/scripts/app.js:7995) — already marked *"Read-only reference for everyone"* — gains two admin CTAs visible only when the resolved persona holds the `canManageDataElements` capability. No separate sidebar IA; the [`SIDEBAR_ITEMS_PLATFORM`](../../CONTEXT.md:359) experiment was retired and Sarah uses the unified IA with capability gating. URL stays `/portal/<dex>/data-elements/…`.

**2. Two CTAs reflect two distinct governance acts.** `+ New element` (greenfield identity) and `+ New version of…` (bump on an existing identity). These are two different conversations: a new element introduces a new identity into the catalogue; a new version supersedes an existing one for *future* Agreements (existing Agreements keep their snapshot per [ADR 0026](./0026-agreement-snapshot-immutability-schema-upgrade.md)). Mirrors [ADR 0004](./0004-unified-agreement-with-two-create-entry-points.md)'s "two distinct CTAs > one branching wizard."

**3. Smart Start: four schema-seeding on-ramps converging on the canvas.** Sample data (CSV/JSON), Existing form (PDF/Word/eForm), Plain English, Start from existing schema. Three sit under `+ New element`; the fourth (Start from existing) is the canonical seeder for `+ New version`. All four converge on the same five-tab canvas downstream.

**4. ETR registration explicitly out of scope in phase 1.** ETRs (eBL, eBoL) keep their own dedicated workspace with transfer/surrender/shred operations (see [`index.html:1890`](../../portal-app/index.html:1890) narrative copy). The registration UI is framed as *"for regular documents — ETR registration is a separate flow"* in subtitle/chip. Forward-compatible with a sibling `+ New ETR` CTA when the ETR-workspace ADR is written.

**5. Four-tab canvas: Schema · Compose complexity · Rules · Review.** The flow shape is fixed; tabs are not optional. *Schema* uses a field-builder primary surface (top-level fields and nested objects up to 3 levels; per-field validation native to JSON Schema) with a read-only JSON Schema preview panel; conditional schemas (`oneOf`) live in the JSON view only. *Compose complexity* is a prominent dedicated step with two large cards (Simple / High-stakes) showing Composer-render snippets; indicators light up based on the schema (signature field, personal identifiers, regulatory keywords) but the admin makes the call manually — **no auto-suggest pre-selection**. *Rules* is the govaluate-expression tab per [ADR 0038](./0038-data-element-governance-three-independent-layers.md), always visible from day one, evaluating live against the Smart Start sample payload. *Review* is the terminal tab; it is the only place the `Publish v1.0` button lives, and it carries the **pack assignment sidecar** (inline-optional, smart pack-fit suggestion via schema field-name overlap per sub-decision 6 below) alongside the publish button. Pack is deliberately **not its own tab** — it's a sidecar inside Review, because pack assignment is optional governance hygiene that benefits from being asked at the moment of commit, not as a separate step.

**6. compose_complexity is a deliberate human choice, not an auto-suggested default.** The admin's thumbprint is the safety gradient per [ADR 0025](./0025-data-element-compose-complexity-attribute.md)'s "individuals can't accidentally downgrade a high-stakes document." Indicators inform; they do not pre-select. Auto-suggestion would dilute the admin governance to clicking *Accept*. Walking back an earlier instinct toward auto-suggest is itself a load-bearing call here.

**7. Live Composer preview alongside the canvas, plus on-demand full-screen test modal.** A right-rail skeleton (~30% width) renders the Composer shape live as Sarah edits — adding a field shows the input appear, toggling complexity transforms the form into a 3-step wizard, adding a rule fires validation against the sample payload. The skeleton is read-only-ish (interactive form, no propagation back to the field-builder). A *"Test as operator"* button opens the full Composer in a modal with faked operator persona resolution per [ADR 0030](./0030-demo-persona-resolution-via-persona-pill-and-url-dex.md). Skeleton is hideable for small screens. Reuses the existing Composer renderer from [`index.html:1340+`](../../portal-app/index.html:1340).

**8. Publish via dedicated Review tab; no modal warning.** Snapshot immutability deserves friction; [ADR 0015](./0015-onboarding-via-design-discipline-not-tours.md) deserves *structural* friction not modal-as-warning. The Review tab collapses everything Sarah authored into a single summary view — header, schema preview, complexity card, rules with pass/fail badges against sample payload, pack chip — plus a *"What happens at publish"* panel naming the three structural consequences (schema becomes final for new Agreements; existing Agreements unaffected; new version = new immutable record). `Publish v1.0` button at the bottom; click → toast + jump to catalogue with new element highlighted. No celebration modal. Autosave to localStorage between tabs — called *"work in progress"* in copy, never *"Draft"* (which is reserved for Agreement drafts per [CONTEXT.md:49](../../CONTEXT.md:49)).

**9. Vocabulary: three-tier.** Catalogue CTAs `+ New element` / `+ New version` (matches the existing `+ New Agreement` topbar pattern at [`index.html:8032`](../../portal-app/index.html:8032)). Flow page title echoes the CTA (`New element` / `New version of Concrete Cube Test`). Terminal commit button `Publish v1.0` / `Publish v1.1` (aligned with [ADR 0026](./0026-agreement-snapshot-immutability-schema-upgrade.md)'s publish vocabulary). *"Register"* is retired as legacy CSM-era language and explicitly avoided in the UI. CONTEXT.md adds the canonical noun *Element version*.

**10. Hybrid inference by determinism for Smart Start.** Real for deterministic + cheap: CSV/JSON sample inference (~50 lines of JS), Start-from-existing (object spread). Faked-with-theatrical-streaming for stochastic + expensive: Plain English (canned outputs streaming at ~30 chars/sec over ~3s, pre-loaded example prompts). Two-stage hybrid for PDF/Word: Stage 1 (text extraction) is real Tesseract.js + pdf.js lazy-loaded (~7MB WASM, fully offline, page-by-page progress, extracted text visible to reviewer in a side panel); Stage 2 (semantic parse to schema) is canned with theatrical reveal (~1s). No external API calls, no API keys, no network — per [ADR 0034](./0034-prototype-to-functional-auto-demo-runner.md). User-provided inputs on faked on-ramps route to closest example with visible note: *"Using example output for demo — production version generates fresh schema."* Honesty pre-empts the gotcha.

**11. Residency-strict as per-element toggle in the Governance section of the Schema tab.** Ticking it locks `compose_complexity` to high-stakes (per [ADR 0025](./0025-data-element-compose-complexity-attribute.md) lines 52–53 auto-upgrade), greys out the Simple card in the complexity step with a lock icon and tooltip *"Locked by Residency-strict — see Schema tab,"* and surfaces a *"Cross-DEX use: blocked"* row on the Review tab naming the [ADR 0012](./0012-cross-dex-action-warning.md) hard-stop. If ticked after Simple was chosen, the flow auto-flips to High-stakes with a banner — never silent mutation. Forward-compatible with a future data-class registry that would source the classification by inheritance (toggle becomes read-only chip; no flow change).

**12. Two auto-demo flows authored at v1, per [ADR 0034](./0034-prototype-to-functional-auto-demo-runner.md) and [ADR 0037](./0037-auto-demo-runner-decoupled-from-markup-via-demo-anchors.md).** `portal-app/scripts/demos/register-element.js` (greenfield via Plain English on-ramp, ~90 seconds, annotates ADRs 0001/0015/0025/0026/0033/0038 and the Q-decisions of this ADR). `portal-app/scripts/demos/version-element.js` (new version via Start-from-existing on-ramp, ~45 seconds, annotates ADR 0026 specifically — v1.0 preserved, v1.1 is new snapshot for future Agreements only). Both reuse shared Smart Start fixtures under `scripts/demos/fixtures/`. Demo anchors per ADR 0037 are required infrastructure (`register-cta`, `onramp-nl`, `nl-prompt-input`, `complexity-card-high-stakes`, `rules-tab`, `review-publish-btn`, etc.). PDF/OCR on-ramp gets no auto-demo — file-input automation is awkward in scripted demos, and the Tesseract magic is photogenic enough as a live walkthrough.

**13. `canManageDataElements` is the single broad capability gate.** Granted to the platform-tier roles `Super SGTradex Admin` and `SGTradex Admin` — the platform-tier role names are intentionally DEX-agnostic per [ADR 0001](./0001-url-anchored-dex-context.md) (DEX comes from the URL, not the role name). Absent from all org-tier roles (`Super Admin`, `Admin User`, `Operation User`, `Tech User`, `Etr User`). Gates the `+ New element` / `+ New version` / `+ New pack` CTAs and future element retirement. Does not gate catalogue viewing or Composer use of elements (those have their own gates). Pattern-consistent with `canCreateAgreement`'s breadth. Supersedes the legacy admin-corev2 capability `canPromoteDataElement` — which was Super-tier-only and unused in portal-app today; the new capability **broadens** to both platform-tier roles because in the new portal model element registration is unified Sarah-class governance, not a Super-only act. Forward-flexible: future split into `canPublishDataElements` + `canManagePacks` + `canRetireDataElements` is non-breaking.

**14. Implementation order, after this ADR lands.** (a) Capability gate added to `ROLE_CAPABILITIES`; sidebar item gating verified. (b) Two CTAs on the Data elements page. (c) Schema tab + Start-from-existing on-ramp first (cheapest, validates the canvas). (d) Other three on-ramps (NL, Sample, PDF/OCR). (e) Compose complexity + Rules + Pack tabs. (f) Review tab + Publish wiring. (g) Auto-demo authoring last, once anchors and surfaces have settled.

## Why this is consistent with existing doctrine

- **[ADR 0001](./0001-url-anchored-dex-context.md) (URL-anchored DEX).** DEX is fixed by URL at registration time. The registration flow lives under `/portal/<dex>/data-elements/…` — single DEX per registration, no cross-DEX authoring in one gesture.
- **[ADR 0004](./0004-unified-agreement-with-two-create-entry-points.md) (two CTAs > branching wizard).** ADR 0039's sub-decision 2 directly applies the precedent. Two distinct conversations (new element vs new version) get two distinct CTAs, not one wizard with a branching first step.
- **[ADR 0013](./0013-data-element-picker-browse-with-groups.md) (mutable packs, per-element snapshots).** Pack assignment at registration is inline-optional; Sarah picks one pack with a smart suggestion, but pack composition continues to be managed on the pack admin page separately. Existing Agreements are unaffected by later pack edits per ADR 0013.
- **[ADR 0015](./0015-onboarding-via-design-discipline-not-tours.md) (no tours).** The Review tab IS the structural friction that makes publish-time gravity visible without resorting to modal warnings. The *"What happens at publish"* panel is design-discipline copy in a structural placement, not a popup.
- **[ADR 0025](./0025-data-element-compose-complexity-attribute.md) (compose_complexity is DEX-admin-owned).** Sub-decision 6 makes the admin's choice the safety thumbprint — auto-suggest is rejected precisely because it dilutes ADR 0025's "individuals can't accidentally downgrade" property. Indicators inform without pre-selecting.
- **[ADR 0026](./0026-agreement-snapshot-immutability-schema-upgrade.md) (snapshot immutability).** Sub-decisions 2, 8, 9, 11 all hinge on snapshot immutability. New version = new immutable record. No in-place edit. Publish is one-way. *Register* vocabulary is retired because the right verb under immutability is *publish*. Existing Agreements are explicitly named as unaffected in the Review tab's "What happens at publish" panel.
- **[ADR 0029](./0029-user-org-affiliation-as-n-to-m-with-embedded-dex-roles.md) (platformRole).** The Sarah-class persona's `platformRole` is the substrate that `canManageDataElements` reads from. Registration is gated on platform-tier role, not per-DEX-org-role.
- **[ADR 0030](./0030-demo-persona-resolution-via-persona-pill-and-url-dex.md) (persona-category resolution).** The "Test as operator" modal in sub-decision 7 opens with a faked operator persona resolution — same mechanism the rest of the prototype already uses.
- **[ADR 0033](./0033-reactive-pitstop-scope-capture-with-inference.md) (reactive capture at the moment of friction).** Pack assignment as a sidecar suggestion at the publish step is exactly this pattern — surface the question at the moment the answer is most cognitively available, not upfront in a wizard.
- **[ADR 0034](./0034-prototype-to-functional-auto-demo-runner.md) (functional prototype, offline).** Sub-decision 10's inference hybrid is the offline-runnable shape: real engines for deterministic cases, no external calls, theatrical fakes for stochastic ones. Demo runner authoring per sub-decision 12 follows ADR 0034's declarative-step model.
- **[ADR 0037](./0037-auto-demo-runner-decoupled-from-markup-via-demo-anchors.md) (demo anchors).** The anchors named in sub-decision 12 are stable `data-demo-anchor` attributes per ADR 0037's two-attribute contract — flow-agnostic, role-named, JSDOM-smoke-tested.
- **[ADR 0038](./0038-data-element-governance-three-independent-layers.md) (three-layer governance).** The five-tab flow in sub-decision 5 surfaces ADR 0038's three layers as three of the five tabs (Schema, Rules, Routing-placeholder). Compose complexity and Pack are the two ADR-0025/ADR-0013 attributes that sit alongside the governance layers.

## Considered options that were rejected

**Separate admin-app at `admin-app/`.** Spin up a new app surface for governance work, leaving `portal-app/` participant-only. Rejected: (a) duplicates chrome, persona resolution, theme, demo runner; (b) fights the unified-portal narrative; (c) the `SIDEBAR_ITEMS_PLATFORM` experiment already tested and retired a separate admin IA on 2026-05-17 (see [CONTEXT.md:359](../../CONTEXT.md:359)) — repeating it would re-burn a known-bad path.

**Self-service registration for any DEX member with admin approval queue.** Mirror the legacy SR-approval workflow inside the portal so any CSM-class user can draft and an admin approves. Rejected: (a) reintroduces the relay race the new architecture deletes; (b) conflicts with [ADR 0025](./0025-data-element-compose-complexity-attribute.md)'s DEX-admin-owned framing; (c) the safety story of *"the person who decides the safety gradient is the same person who authors the schema"* is the design point — splitting authorship from approval re-creates the gap. Forward-compatible: when an approval gate becomes politically necessary, the Review tab gains a sibling *"Send for approval"* button next to *"Publish v1.0"* — no architectural change.

**Single CTA with mode-detection (one Register button that figures out new-vs-version).** Rejected: confuses two distinct governance acts. Publishing v2 of an existing element is a higher-stakes action than registering a brand-new element (existing Agreements are affected on future picks); conflating the two risks an admin accidentally creating a version when they meant a new element. Two CTAs force the choice up front.

**ETR in the same registration flow with `is_etr` attribute.** Rejected: ETRs are architecturally different (endorsement chain, ownership records, transfer/surrender/shred). Forcing them into the same canvas leaks ETR concerns into 95% of the surface that doesn't need them, or requires switch-reveals-new-fields patterns that demo poorly. Phase-1 honesty (out of scope, sibling CTA later) is cleaner than premature unification.

**Branching wizard for simple vs high-stakes at the door.** Pick complexity at the catalogue page; each path is its own wizard shape. Rejected: [ADR 0025](./0025-data-element-compose-complexity-attribute.md) makes complexity an *attribute on the element*, not a fork in the authoring flow. The complexity choice is made *after* the schema is seeded, not before — because the schema content informs the choice. One entry point, one flow, complexity as a deliberate step inside.

**Auto-suggested compose_complexity with admin override.** Rejected after grilling — the original instinct was that auto-suggest is "helpful," but the auto-suggest pattern in a high-stakes safety gradient quietly converts the admin choice from *deliberation* to *accept-clicking*. ADR 0025's "individuals can't accidentally downgrade" relies on the admin actually deliberating. Indicators that light up based on schema content (signature field, personal identifiers) inform without pre-selecting; the admin still picks one of the two cards by hand.

**Modal-confirm publish button.** Click `Publish` → modal *"Are you sure?"*. Rejected: modal-as-warning is the warning-side of the tours that [ADR 0015](./0015-onboarding-via-design-discipline-not-tours.md) rejected. Friction should be *structural* (the Review tab IS the review), not popup-shaped. The Review tab also forward-accommodates a future approval workflow without UI redesign (sibling button), which modal-confirm cannot.

**All-real or all-fake inference.** All-real would require LLM API keys, network dependency, and mid-demo failure modes (latency, weird output) — breaks ADR 0034's offline-runnable promise and the demo's reliability. All-fake reads as theatre to sharp reviewers and loses the visceral *"drop your own CSV and watch it work"* moment. The hybrid-by-determinism cut (sub-decision 10) is the honest middle.

**OCR via cloud API.** Use Google Vision / AWS Textract / Azure Form Recognizer for real PDF parsing. Rejected: requires API keys and network access, contradicting ADR 0034's offline contract. Tesseract.js + pdf.js run fully in-browser at the cost of ~7MB WASM lazy-loaded — same real-extraction story without external dependencies.

**Element-version drafts as a system state.** Treat in-progress registration as a `DRAFT` status on the element, visible in admin queues. Rejected: conflicts with [ADR 0026](./0026-agreement-snapshot-immutability-schema-upgrade.md) (no in-place edits, no draft lifecycle on published records) and with [CONTEXT.md:49](../../CONTEXT.md:49)'s reservation of *"Draft"* for Agreement drafts only. In-flight registration is local autosave to localStorage, never a system state — and the copy calls it *"work in progress"* to avoid the term clash.

**Pack assignment as a dedicated tab.** Rejected: overstates packs' role given they're explicitly optional per [ADR 0013](./0013-data-element-picker-browse-with-groups.md). Inline-sidecar at the publish step honors optionality while still surfacing the pack-fit suggestion at the moment context is most available.

**One auto-demo only (greenfield).** Rejected: leaves the `+ New version` CTA asserted in copy but never animated. Sub-decision 2's two-CTA decision deserves two demos to validate the IA visually.

**Three auto-demos (add PDF/OCR as a third).** Rejected: file-input automation in scripted demos is awkward, and the Tesseract OCR moment works better as a live walkthrough where the demo-er can narrate *"now I'm uploading our actual legacy form."* The on-ramp is fully built and works in live use; the auto-demo would add overhead without much payoff.

**Granular capabilities (`canPublishDataElements`, `canManagePacks`, `canRetireDataElements`).** Rejected as premature. All five governance actions cluster on Sarah-the-persona today; no realistic role demands a finer cut. `canManageDataElements` mirrors `canCreateAgreement`'s breadth. Future split is non-breaking when a granular role actually arrives.
