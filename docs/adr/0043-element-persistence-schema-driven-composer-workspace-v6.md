# Element persistence + schema-driven Composer fork: workspace v6, prototype scope

> **Status:** accepted (2026-05-24)
> Builds on [ADR 0026](./0026-agreement-snapshot-immutability-schema-upgrade.md) (Element version immutability and the snapshot rule the Agreement record must honour), [ADR 0039](./0039-data-element-registration-admin-authored-single-page-flow.md) (the single-page registration flow whose Publish action this ADR completes), [ADR 0042](./0042-element-spec-sheet-onramp-deterministic-parser-audit-log-provenance.md) (audit-log-only authoring provenance — strip-at-publish contract reused here), [ADR 0038](./0038-data-element-governance-three-independent-layers.md) (the three-layer governance the Composer must respect at submission time). Closes the *"persistence to localStorage is a Phase-2 concern"* footnote at [`register-element.js:9518`](../../portal-app/scripts/register-element.js:9518).

## Context

A published Element version in the prototype today is a mirage. [`regPublish()`](../../portal-app/scripts/register-element.js:9467) appends a catalogue stub (`{ id, name, version, icon }`) to an in-memory `DATA_ELEMENTS_BY_DEX` group called *"Authored this session"* and clears the WIP autosave. The full authored artefact — `elementSchema` JSON, `composeComplexity`, the govaluate `rules`, the `x-presentation` overrides, the pack membership — vanishes the moment the operator navigates away or reloads. The comment block at line 9518 acknowledges this directly: *"persistence to localStorage is a Phase-2 concern (registration becomes a real backend action then)."* The result is a registration flow that demos cleanly in isolation but doesn't connect to anything downstream.

Three concrete gaps emerge once you trace what should happen *after* publish:

(1) **Persistence.** Nothing about the published Element version survives a reload, so a fresh session can't pick it from the catalogue. The workspace snapshot ([`workspace-storage.js`](../../portal-app/scripts/workspace-storage.js), schema v5) carries `agreements`, `messages`, `inbox`, `orgs`, `users`, `pitstops`, `agreementPacks` — but no `dataElements` collection. The hardcoded `.picker-tree` for SGTradex in the data-picker screen further means new entries don't appear in the picker even within a single session (only SGBuildex and SGHealthdex rebuild from `DATA_ELEMENTS_BY_DEX` at [`app.js:162`](../../portal-app/scripts/app.js:162)).

(2) **Agreement snapshot.** The Agreement wizard captures `wiz.de` (name string) and derives `wiz.deId` via [`elementIdFromName`](../../portal-app/scripts/pitstop.js:417) (a slug fallback). It does not capture an Element version. [ADR 0026](./0026-agreement-snapshot-immutability-schema-upgrade.md) and the [CONTEXT.md] *Snapshot* entry are explicit that an Agreement freezes `(element_id, version)` at creation; today the prototype freezes only a slug.

(3) **Composer / Message.** [`setComposerScenario()`](../../portal-app/scripts/app.js:5855) renders forms from hardcoded HTML strings in [`COMPOSE_FORM_VARIANTS`](../../portal-app/scripts/app.js:5367) (`bol`, `bunker`, `customs`, `pull`), one per demo scenario. It does not consume `elementSchema` at all. And the workspace `messages[*]` record shape ([`workspace-bootstrap.js:197`](../../portal-app/scripts/workspace-bootstrap.js:197)) carries an `element: { name, version }` display tuple but no `payload` field — so even if the Composer rendered correctly, there is nowhere to save what was submitted. The registration→agreement→message chain is broken end-to-end.

A grilling session walked the dependency tree from the storage layer outward — picking a persistence layer; picking a per-record shape; picking how the Agreement snapshots the version; picking whether the Composer renders schema-driven, scenario-driven, or both; picking how Submit saves; picking how the existing `regBuildSkeleton*` family is reused so all three surfaces (Composer Preview, Test-as-operator, real Composer) stay structurally aligned per [[feedback-three-surfaces-check]]. The decisions below close the chain.

## Decision

The workspace snapshot bumps to schema **v6** with archive-and-rebuild, gaining a `dataElements` collection. The Agreement record gains an `elementSnapshot` key. The Composer gains a schema-driven render path that reuses the registration-flow schema walker. The Message record gains a `payload` key. Ten sub-decisions:

**1. Persistence layer: workspace snapshot v6, single collection keyed by `${id}@${version}`.** A new `workspace.dataElements` map is added to the snapshot. Schema-version bump 5 → 6 archives existing snapshots per the [`archiveCorruptWorkspace`](../../portal-app/scripts/workspace-storage.js:28) pattern — same doctrine schema v3 used when unifying scene data. `DATA_ELEMENTS_BY_DEX` remains the seed fixture (read on first bootstrap); after that, the catalogue is rendered as a derived view over `workspace.dataElements ∪ seed`. Rejected alternatives: a separate `localStorage` key parallel to the workspace (cheaper to add, breaks the single-snapshot bootstrap pattern that schema v3 deliberately unified), and in-memory only with a per-session lifecycle (the existing footnote at register-element.js:9518 — known broken, the gap this ADR closes).

**2. Visibility scope: workspace-scoped (per-browser).** Published elements are visible only to the browser that published them. A faux-DEX-scoped layer (a second localStorage key that workspace-bootstrap merges from) was considered and rejected — adds a cross-key merge layer without changing what the demo proves. The Snapshot rule from [ADR 0026](./0026-agreement-snapshot-immutability-schema-upgrade.md) means Agreements that *do* reference a workspace-only element still resolve in the publishing browser; the "what's pickable in *my* catalogue" question is per-browser, which is the honest model for a prototype with no backend.

**3. Per-record shape: full Element version, sidecars stripped on persist.** Each `workspace.dataElements['${id}@${version}']` entry carries `{ id, version, name, dexId, publishedAt, publishedBy, elementSchema, composeComplexity, rules, pack, auditTrail }`. The `elementSchema` is the **interop-clean** artefact per the [CONTEXT.md] *Interop-clean published schema* principle — authoring sidecars (`x-source`, `x-provenance` from [ADR 0042](./0042-element-spec-sheet-onramp-deterministic-parser-audit-log-provenance.md)) are stripped at persist time; render-directive `x-*` extensions (`x-presentation`, `x-uitype`, `x-uioptions`) survive. A unit-level guard pins the contract: *"persisted dataElements entries contain no `x-*` keys outside the known-advisory allowlist."* `auditTrail` carries the `element-version-published` event plus any future per-version events, keyed by version so audit reads don't bloat catalogue reads.

**4. Agreement snapshot: `elementSnapshot: { id, version, source }`.** On Agreement creation the wizard captures `elementSnapshot.id` (resolved against `workspace.dataElements` or slug-fallback) and `elementSnapshot.version` (the latest version of that element at creation time, by `publishedAt`). The pre-existing `element: { name, version }` display tuple is preserved alongside for the renderer (same split-pattern schema v2 used when stamping `counterpartyOrgId` alongside the display name — see [`workspace-storage.js:21-25`](../../portal-app/scripts/workspace-storage.js:21)). The `source` slot carries `'published'` (resolved against `workspace.dataElements`) or `'seed'` (slug-only fallback for seeded fixture elements that never had a full `elementSchema` published). The discriminator drives the Composer fork in sub-decision 7.

**5. Picker rebuild: every DEX, derived from catalogue.** The hardcoded `.picker-tree` for SGTradex becomes seed-only first-paint; on workspace bootstrap, the tree is replaced with one derived from `workspace.dataElements` overlaid on `DATA_ELEMENTS_BY_DEX`. SGBuildex and SGHealthdex behaviour is unchanged. The synthetic *"Authored this session"* group ([`register-element.js:9523`](../../portal-app/scripts/register-element.js:9523)) is dropped — once persistence works, the *"session"* label is misleading; published elements merge into their declared pack group or surface in a neutral *"Recently published"* indicator on the catalogue row.

**6. Composer fork: schema-driven for `source='published'`, scenarios for `source='seed'`.** The Composer reads `agreement.elementSnapshot.source` and dispatches: `'published'` → new schema-driven path that hydrates the form from `workspace.dataElements[ref].elementSchema`; `'seed'` → existing scenario-driven path (`COMPOSE_FORM_VARIANTS` keyed by the same scenario-inference logic that lives in [`COMPOSE_SCENARIOS`](../../portal-app/scripts/app.js)). Banners (cross-DEX, pitstop-down, acting-as, multi-schema) are factored into a banner-state resolver keyed on the Agreement record — both paths feed the same banner-render code at [`app.js:5887`](../../portal-app/scripts/app.js:5887). The chip resolvers from [`pitstop.js`](../../portal-app/scripts/pitstop.js) (Acting-as, Send-from per [CONTEXT.md] *Pitstop chip*) already key on the Agreement and need no change. Rejected: rewriting all scenario forms as fixture `dataElements` entries (a much larger lift that breaks the existing demos' three-surfaces-check parity) and a single-path Composer that requires every Agreement to have a full schema (would break the seed-fixture demo path).

**7. Schema walker: hoist `regBuildSkeleton*` family to a shared module.** A new file `portal-app/scripts/schema-walker.js` owns `regBuildSkeletonInput` / `regBuildSkeletonArray` / `regBuildSkeletonObject` / `regBuildSkeletonLikert` / `regBuildSkeletonForEnum` / `regBuildSkeletonByHint` / `regBuildSkeletonInputWithValue` (currently at [`register-element.js:7643+`](../../portal-app/scripts/register-element.js:7643)) plus a new inverse `schemaWalker_readValues(rootEl, schema)` that walks the form DOM on Submit and produces the JSON object keyed by schema field path. All three surfaces — Composer Preview skeleton, Test-as-operator preview, and the real Composer (sub-decision 6) — consume the same walker. The skeleton surfaces pass `{ disabled: true }`; the real Composer passes `{ disabled: false, capturePath: <field-path> }`. This makes the [[feedback-three-surfaces-check]] structurally enforced — a new field type lands in all three surfaces simultaneously, or in none. Rejected: writing a third renderer alongside `regBuildSkeleton*` (invites drift), and parameterising the existing functions in place inside `register-element.js` (couples the Composer to the registration module).

**8. Message payload shape: schema-keyed values + rule-evaluation snapshot.** The workspace `messages[*]` record gains `payload: { schemaRef, values, submittedAt, submittedBy, rulesEval }` where `schemaRef` is `${id}@${version}` back-pointing to `workspace.dataElements`, `values` is the schema-shaped JSON produced by `schemaWalker_readValues` (sub-decision 7), and `rulesEval` is the snapshot of govaluate rule results at submission time per [CONTEXT.md] *Validation rule* and the three-layer governance table in [ADR 0038](./0038-data-element-governance-three-independent-layers.md). Submit blocks on any rule failure — matching production govaluate semantics. The Message detail view renders the saved payload using the same walker (skeleton mode) so the operator sees the saved form with the same widget choices that authored it. For the scenario-driven path (sub-decision 6), `payload.values` is recorded as an unparsed `'<scenario-form>'` placeholder — the prototype is honest about what it can't reconstruct, rather than fabricating schema-shaped JSON.

**9. AJV deferred; rules-only validation in the prototype.** Field-level constraints are not validated in the prototype Composer. Production lives in `dex-monorepo/ui/libs/src/components/Forms/` and uses `@rjsf/core` + `@rjsf/validator-ajv8` ([`dex-monorepo/ui/package.json:40-42`](../../../dex-monorepo/ui/package.json)) with custom widgets and error formatting. Re-implementing that stack in vanilla JS for the prototype invites behavioural divergence — the prototype's AJV would lag dex-monorepo's RJSF setup. The boundary is: govaluate rules (cross-field, compound) run at the prototype's Submit; per-field constraints are stubbed with a *"validated by AJV in production"* annotation in the audit log. This is a deliberate prototype-vs-production seam, not a feature gap to close in the prototype.

**10. Orphan cleanup on bootstrap, silent.** When the workspace contains an Agreement whose `elementSnapshot.source === 'published'` but the referenced `workspace.dataElements['${id}@${version}']` is missing (a workspace cleared on a different browser, a schema-version archive that wiped the snapshot, etc.), the workspace-bootstrap GC pass drops the Agreement and cascades through `workspace.messages` (any with the dead `agreementId`) and `workspace.inboxItems` (any referencing the dropped Agreement or Messages). No banner, no read-only stub, no toast — `console.warn` only. This is the [[feedback-prototype-not-museum]] doctrine: dangling-reference UX would be reimplemented from scratch in production, where audit-trail requirements actually demand it; the prototype's job is a viscerally working demo, not a forensic record. Distinct from named retirement events (e.g., [CONTEXT.md] *Pitstop retirement*) where the reference survives by design — those are first-class lifecycle events with tombstones, not dangling state.

**Coexistence and version selection (cross-cutting on 3 and 4).** When an element id has multiple persisted versions (Sarah publishes v1.0, then v1.1), both rows live under their `${id}@${version}` keys. The catalogue surfaces v1.1 as active, v1.0 accessible via a "Previous versions" expander; Agreements snapshotted to v1.0 continue resolving. New Agreement creation always picks the latest published version by `publishedAt` timestamp ordering — semver string comparison (`v1.10` vs `v1.9`) is fragile because the version string is free-form, while `publishedAt` is unambiguous and matches the *"signed off most recently"* mental model. Operator-selectable version at Agreement creation was rejected — invites cross-version mistakes the snapshot rule was written to prevent.

## Consequences

**Architectural:**

- Workspace schema v5 → v6. Existing v5 snapshots archive-and-rebuild on first read (existing pattern from schemas v2, v3, v4, v5). [`workspace-storage.js`](../../portal-app/scripts/workspace-storage.js) gains the version-bump comment; [`workspace-bootstrap.js`](../../portal-app/scripts/workspace-bootstrap.js) gains the `dataElements: {}` initialisation and the orphan-cleanup GC pass.
- A new module `portal-app/scripts/schema-walker.js` owns the family of `regBuildSkeleton*` functions (hoisted from `register-element.js`) plus `schemaWalker_readValues`. `register-element.js` becomes a consumer rather than the owner; the registration flow's Composer Preview and Test-as-operator surfaces continue to work through the new module path.
- [`regPublish()`](../../portal-app/scripts/register-element.js:9467) extends: in addition to appending the catalogue stub, writes the full Element version record to `workspace.dataElements[${id}@${version}]` and persists the snapshot via `writeWorkspaceSnapshot`.
- Agreement-creation flow ([`wizard.js`](../../portal-app/scripts/wizard.js), [`app.js`](../../portal-app/scripts/app.js) picker leaf handlers) captures `elementSnapshot: { id, version, source }` on the agreement record alongside the existing `element: { name, version }` display tuple.
- Composer screen ([`setComposerScenario`](../../portal-app/scripts/app.js:5855) and friends) forks on `elementSnapshot.source`. A new entry point `openComposer('schema-driven', { elementVersionRef })` covers the published path; the existing scenario keys are untouched.
- Message record at [`workspace-bootstrap.js:197`](../../portal-app/scripts/workspace-bootstrap.js:197) extends with a `payload` slot; the Message detail renderer gains a "Submitted payload" section that calls the schema walker in skeleton mode.

**Behavioural / contractual:**

- The registration → agreement → message chain works end-to-end *within a single browser*. Sarah publishes "Bunker invoice v1.0" → Marcus (same browser) creates an Agreement against it → Marcus opens the Composer → form fields render from Sarah's `elementSchema` with `x-presentation` overrides honoured → Marcus submits → the payload lands on `workspace.messages[*].payload.values` keyed by the schema field paths → Marcus reopens the Message and sees the saved values rendered with the same widgets that authored them.
- Published elements survive page reloads but not cross-browser visits. This is the honest model of a prototype with no backend; cross-browser parity is a production concern, not a prototype concern.
- Agreements against the seeded fixture elements (Bill of Lading, Cargo manifest, etc.) continue to use the existing scenario-driven Composer path unchanged. Sub-decision 6's fork preserves every existing demo.
- The [[feedback-three-surfaces-check]] is now structurally enforced via the shared schema walker (sub-decision 7) rather than relying on the author remembering to verify three places. Field-type additions land in all three surfaces or in none.
- Orphan workspace state (sub-decision 10) is dropped silently on bootstrap per [[feedback-prototype-not-museum]]. Production behaviour will differ — production must preserve broken references for forensic reasons — but the prototype's role is the demo, not the audit log.

**Forward-flexible:**

- The schema walker (`schema-walker.js`) is the seam where a future production port to `@rjsf/core` slots in. The walker's interface (`(schema, values, options) → DOM` + `(rootEl, schema) → values`) maps cleanly onto RJSF's `({schema, formData}) → ReactNode` + `({schema, formData}) → formData` shape; the per-type dispatch and `x-presentation` resolution carry over.
- `workspace.dataElements['${id}@${version}']` keyed coexistence (cross-cutting paragraph above) is the model the production backend will follow — DynamoDB rows in `sgtradextech-data-element-{dev,prod}` are already keyed on `(elementId, version)` per [ADR 0026](./0026-agreement-snapshot-immutability-schema-upgrade.md). The prototype's collection shape ports directly.
- Message drafts (per [CONTEXT.md] *Message draft* — `consent_message_draft` per-user-per-Agreement, AES-256-GCM with the Agreement key, decay-with-pin lifecycle) are deliberately out of scope here. The `payload` shape from sub-decision 8 is forward-compatible: a future `workspace.messageDrafts` collection can carry the same `{ schemaRef, values }` partial-fill shape, encrypted-stub at the prototype level.
- The AJV-deferred seam (sub-decision 9) names the production handoff point. When the prototype is ported, the schema walker's `disabled: false` mode becomes RJSF's `<Form>`; the rules-only Submit gate becomes RJSF's `validate` callback chained with the govaluate rules; field-level errors come from AJV and surface in RJSF's standard error UI.
