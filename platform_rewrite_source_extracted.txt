

 Platform rewrite Initiative

Platform rewrite Initiative

This plan outlines the full-stack consolidation of the SGTraDex platform — merging admin-corev2 + pitstop-core into a single unified Go API service (expanded from be/admin-core), and admin-ui + pitstop-ui into one unified React portal (dex-monorepo/ui/apps/portal) — alongside DB consolidation, domain model evolution, and a v1→v2 strangler-fig migration so production is never broken.

1. Transaction Layer Overloading (Latency & Risk)

Coupling of Business Logic to the Critical Path: The pitstop-core handles both high-throughput message processing (MessageStore) and complex business/feature logic. Adding new features requires adding validations, state checks, or database joins directly into the transaction path, which inherently inflates end-to-end message latency.

CPU Sensitivity: High-volume tables like MessageStore and AuditTrail are append-heavy and CPU-sensitive. Expanding these tables or adding synchronous feature triggers risks DB saturation, degrading the entire transaction layer's performance.

2. Monorepo Entanglement (dex-monorepo)

Tight Coupling: Features are heavily entangled rather than modular. If a specific capability (e.g., ETR flows or a specific data element feature) is not applicable to a given DEX or participant, it is extremely difficult to extract it or cleanly toggle it off.

Blast Radius: Because features cannot be easily decoupled, bugs in unused or non-applicable features still pose a deployment and runtime risk to the entire codebase. Every deployment carries the weight of all features.

3. Domain Model Fragmentation & Duplication

Multiplicative Complexity: Implementing a single new feature often means writing logic for multiple fragmented models. For example, there are currently four separate representations for consent/sharing: Subscription (v1), SubscriptionV2, DataExchangeRelationship, and ServiceProviderRelationship.

Dual-Database Sync Overhead: Core entities like Organization, DataElement, and User exist in both the Admin DB and the Pitstop DB. This requires fragile cross-service synchronization mechanisms, leading to eventual consistency issues, data drift, and added latency when features require up-to-date reads.

4. Cross-Boundary Latency

Account & Infrastructure Splits: The platform is split between admin-corev2 (Node.js/Lambda in the Central account) and pitstop-core (Node.js/ECS in the Pitstop account). Features that require data from both sides must make cross-account HTTPS calls, introducing network latency and complex IAM/networking overhead instead of fast, in-process database lookups.

5. Fragmented Client Experiences

Separate UIs: Having separate admin-ui and pitstop-ui apps means any new feature touching both participant and admin personas requires dual implementation, separate CI/CD pipelines, and duplicate component maintenance, severely slowing down feature velocity.

6. Limitation on cross dex sharing

Separate pitstop/admin database pairs: Having separate admin-ui and pitstop-ui databases mean that given current architecture, it is unsuitable for cross dex sharing

Phase 2 — Database Consolidation (6–10 weeks)

Goal: Consolidate the Admin DB and Pitstop DB into a single platform database using AWS DMS, while permanently preserving the high-volume pitstop tables (MessageStore, AuditTrail) in their original per-pitstop databases. A migration option must be selected and approved before DMS work begins (see §2b decision gate).

2a. Principles that apply to all options

MessageStore is never migrated. It holds the full transaction history per pitstop, is append-heavy, and any ALTER TABLE or bulk movement risks CPU saturation and downtime. The v2 API reads it from each pitstop's existing DB connection as-is. No column additions, no data movement.

Pitstop AuditTrail is never migrated. Same reasoning: large, append-heavy, CPU-sensitive. v2 API reads via existing pitstop DB connection.

No ALTER TABLE on either frozen table — not even adding a column. New fields needed by v2 go into new tables in platform DB.

Admin-side AuditTrail is low-volume (org approvals, user changes) and is safe to migrate under all options.

AWS DMS (full-load + CDC) is the tooling for all data movement — enables zero-downtime cutover via ongoing replication before write switch.

Subscription v1 + SubscriptionV2: resolve Phase 3 consent_agreement design before finalising DMS tasks — avoids migrating a table that will be superseded immediately.

Go GORM structs in be/admin-core/internal/model/ are the canonical model definitions for all migrated tables.

2d. DMS execution steps (common to all options)

Step 1 — Schema pre-creation
 Create target tables via SQL DDL scripts (be/admin-core/migrations/).
 Add dex_id columns and indices on empty tables — zero risk.

Step 2 — DMS full-load (per table group, off-peak window)
 Source: Admin RDS (and Pitstop RDS for Options B/C).
 Target: platform DB.
 Validate row counts + spot-check data after each task.

Step 3 — DMS CDC (Change Data Capture)
 Enable logical replication slot on source RDS(es).
 DMS CDC keeps platform DB in sync while v1 services remain live.

Step 4 — Read-switch (per table group)
 Flip v2 API reads to platform DB one group at a time.
 v1 services still read from their original DBs.
 Shadow-read validation for ≥2 weeks before proceeding.

Step 5 — Write cutover (per table group)
 New writes go to platform DB only via v2 service layer.
 v1 shim calls v2 services — no direct legacy DB writes.

Step 6 — Source decommission
 Stop CDC, decommission Admin RDS (Options A/B/C).
 For Options B/C: decommission Pitstop RDS connections for migrated tables
 (MessageStore + AuditTrail connections remain).

Phase 3 — Domain Model Evolution (4–6 weeks, parallel with Phase 2)

Goal: Extend the DEX/Pitstop hierarchy for multi-dex orgs and multi-pitstop subscriptions without breaking existing records.

3a. Org ↔ DEX multi-membership

Current: one org belongs to one dex implicitly via EnterpriseSystem.dexId.

Target: explicit org_dex_membership join table.

CREATE TABLE org_dex_membership (
 id UUID PRIMARY KEY,
 org_id UUID REFERENCES shared_organization(org_id),
 dex_id VARCHAR NOT NULL,
 status VARCHAR NOT NULL DEFAULT 'ACTIVE',
 created TIMESTAMP,
 modified TIMESTAMP
);

3b. Multi-pitstop per org

Current: EnterpriseSystem = one pitstop endpoint per org.

Target: rename EnterpriseSystem → org_endpoint; add dex_id FK. An org can have multiple endpoints across DEXes.

3c. Consent flow generalisation

Current issues:

DataExchangeRelationship (DER) and ServiceProviderRelationship (SPR) are near-identical tables joined by DeSpRelationshipBinding — unnecessary duplication.

Subscription v1 and SubscriptionV2 represent the same concept with different schemas.

Target: one consent_agreement table replaces DER + SPR + SubscriptionV2:

CREATE TABLE consent_agreement (
 id UUID PRIMARY KEY,
 dex_id VARCHAR NOT NULL,
 agreement_type VARCHAR NOT NULL, -- 'DIRECT', 'SERVICE_PROVIDER', 'PRINCIPAL'
 initiator_org_id UUID NOT NULL,
 counterparty_org_id UUID NOT NULL,
 data_element_id VARCHAR,
 use_case_id VARCHAR,
 status VARCHAR NOT NULL DEFAULT 'PENDING',
 flow VARCHAR, -- 'SEND', 'RECEIVE', or NULL for bilateral
 contributor_org_id UUID, -- tripartite SP flows
 pitstop_id VARCHAR,
 created TIMESTAMP,
 modified TIMESTAMP
);

DataExchangeRelationshipInvites + ServiceProviderRelationshipInvites → one consent_invite table (same agreement_type discriminator).

3d. Backward compatibility

v1 API responses for /subscriptions, /data-exchange-relationships, /service-provider-relationships continue to return the same shape by reading from consent_agreement via the v1 shim's query translators in Go.

Old tables are kept read-only until v1 is retired.

Phase 4 — v2 API Layer in Go (6–8 weeks, starts after Phase 1 scaffold)

Goal: Build all v2 domains inside be/admin-core, retiring the Node.js admin-corev2 service. v1 becomes a shim in the same Go binary calling v2 service functions. The 51 route files in admin-corev2 are the full migration surface.

4a. Migration strategy options

Three approaches to ordering the work — choose one or blend them:

Option A — Domain batch migration (recommended)

Group routes by business domain and migrate one domain at a time. Each batch produces a shippable increment: v2 routes live, v1 shim live, old Node.js routes still active during transition.

Batch 1 — Auth & Identity (already partially done in be/admin-core)
 Routes: login, logout, renew-token, validate-token, forgot-password, reset-password,
 user/*, role/*, permissions/*, tfa/*
 Risk: Low — auth is already ported; user/role just CRUD.

Batch 2 — Organisation & Enrolment
 Routes: org/*, enrolments/*, admin-requests/*, orgPersona/*, my-info/*
 Risk: Medium — org status flows and maker-checker logic; email notifications.

Batch 3 — Data Elements & Schema Versioning
 Routes: data-elements/*, data-element-versions/*, data-element-version-comments/*,
 data-element-tracking/*, use-cases/*, usecasepersonas/*, personas/*,
 source-type-config/*, source-system-configuration/*
 Risk: Low-Medium — mostly CRUD + DSV feature flag.

Batch 4 — Consent & Relationships
 Routes: subscriptions/*, refreshSubscription/*, data-exchange-relation/*,
 service-provider-relation/*, verificationFlow/*, orgUseCase/*,
 principal/*, thirdpartylookup/*
 Risk: High — these map to the new consent_agreement model (Phase 3 must be done first).

Batch 5 — Pitstop / System Config
 Routes: sys/*, system/*, orgsys/*, config/*, licenseKey/*, status/*,
 network/*, feature/*, settings/*, org-de-auto-approval/*,
 notification-groups/*, groups/*
 Risk: Medium — pitstop registration and config flows.

Batch 6 — ETR & TradeTrust
 Routes: saved-address/*, tradetrustKey/*, termClause/*, termTemplate/*,
 smart-contract/*, eclip-template/*
 Risk: Medium — TradeTrust-specific flows; low traffic.

Batch 7 — Notifications, Audit, Metering, Misc
 Routes: notifications/*, notification-groups/*, audit/*, meter/*,
 version/*, whats-new/*, tandc/*, master-data/*, email/*
 Risk: Low — mostly read/append; version and whats-new to be retired.

Option B — Traffic-volume-first migration

Migrate the highest-traffic routes first to reduce Node.js load quickly, regardless of domain grouping. Use APM/CloudWatch data to rank. Useful if the team wants early wins on infra cost.

Identify top 10 endpoints by request volume from CloudWatch.

Port those first, shim the rest.

Risk: creates scattered shim coverage early; harder to maintain.

Option C — Caller-type split (internal vs external)

Split routes by who calls them:

Machine-to-machine (validateLicenseKey middleware) → port first (pitstop-core internal calls). These have machine clients that are easier to switch.

Portal-facing (auth middleware) → port second, coordinated with Phase 5 frontend migration.

Admin-only (authSuperSGTraDexAdmin) → port last; lowest volume, highest risk tolerance.

Blends well with Option A as a secondary ordering criterion within each batch.

4b. Within each batch: how to migrate without breaking existing functionality

Step 1 — Write v2 service layer (internal/service/<domain>.go)
 Implement business logic in Go against platform DB.
 Unit test with mocks against store interfaces.

Step 2 — Register v2 routes (/api/v2/<domain>/...)
 Add to router.go. Deploy to QA. Run Playwright E2E against v2.

Step 3 — Register v1 shim (/api/v1/<path>)
 Shim calls v2 service function; transforms response to v1 shape.
 Deploy to QA. Run existing admin-corev2 E2E test suite against shim.

Step 4 — Dual-run in production
 Node.js admin-corev2 still live. be/admin-core shim live.
 Traffic split: route some callers (e.g. new portal) to v2 via feature flag;
 legacy callers stay on Node.js v1 path.

Step 5 — Cut over
 Switch all callers to be/admin-core. Remove Node.js route for this domain.
 Monitor for 1 week. If stable, delete Node.js code for domain.

4f. API contract governance

All v2 routes spec'd in dex-api-specs (OpenAPI YAML) before implementation begins for that batch.

v1 shim routes are registered only when the v2 service function exists — no shim-first skeletons.

v1 published EOL date: 12 months post v2 GA; communicate before Batch 1 is deployed.

Breaking v2 changes require 30-day advance notice to external integrators.

Go file layout per batch: internal/service/<domain>.go, internal/store/<domain>_store.go, internal/dto/<domain>.go, internal/model/<domain>.go, internal/dto/shim/<domain>_v1.go.
